## 一、产品概述

### 1.1 产品定位

个人知识库系统是一个基于**知识图谱（Graph）驱动的智能知识管理平台**，核心能力是将碎片化的原始素材自动编译为结构化的知识网络，并提供基于图谱推理的智能问答。

**一句话描述**：你扔进去碎片素材，系统帮你构建可推理的知识网络，你要什么它都能从图谱中推理找到。

### 1.2 核心诉求

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 推理能力 | P0 | 关联分析、逻辑脉络推理、多跳因果推导 |
| 碎片化知识整合 | P0 | 跨文档、跨领域、跨格式的知识自动关联 |
| 成本控制 | P0 | 月成本 < $10，优先使用本地 LLM |
| 知识图谱可视化 | P1 | 完整的知识脉络可视化，支持交互式探索 |
| 可视化检索 | P1 | 自然语言查询 + 图谱导航检索 |

### 1.3 核心设计理念

```
传统知识管理：知识 → 存储 → 搜索关键词 → 找到文档 → 人工阅读理解
本系统做法：  知识 → Graph 编译（自动提取实体/关系/隐式关联）→ 存入 Neo4j
              查询 → Graph 推理（自动检索路径/隐式关系）→ 直接给出推理结论

核心差异：知识不是"存完再搜"，而是"存的时候就建立推理网络"
```

### 2.2 核心架构决策

| 决策点 | 结论 | 原因 |
|--------|------|------|
| 数据存储 | **Neo4j 唯一数据库**（Single Source of Truth） | 消除双存储同步复杂度，保障数据一致性 |
| Markdown 文件 | **Graph 自动渲染的只读视图** | 保留 Obsidian 兼容性，但不作为权威源 |
| 人类编辑 | **不存在** | 所有知识通过 Ingest 管道输入，避免人为错误 |
| 信息同步 | **单向：Neo4j → Markdown** | 不存在反向同步，无一致性问题 |
| 查询路由 | **不需要路由** | 所有查询统一走 Graph，由 Graph 内部决定检索深度 |

### 2.3 架构演进历程

```
v1: LLM Wiki 为主 + Graph 被动派生
  → 问题：Graph 只是 Wiki 的冗余副本，没有增加推理深度

v2: Wiki + Graph 双存储协作（Graph 主动推理）
  → 问题：双存储同步复杂，存在一致性风险，回写循环需要五重安全保障

v3: Graph-First（V1.0 最终版）
  → Neo4j 存储一切，Markdown 只读渲染
  → 没有同步问题，没有回写循环，没有路由层
  → 简单、一致、推理深度有保障
```

## 四、数据模型设计

### 4.1 Neo4j 节点类型

#### Entity（实体节点）

技术、工具、组织、人物等具体实体。

```
(:Entity {
  id: "RAG",                          // 唯一标识（URL-safe slug）
  name: "RAG",                        // 显示名称
  type: "technology",                 // technology/person/organization/concept
  content: "## Overview\nRAG ...",    // 完整的 Markdown 页面内容 ★
  summary: "检索增强生成技术",          // 一句话摘要
  tags: ["检索增强", "LLM"],           // 标签列表
  source: "raw/sources/2024-01-15-rag.md",  // 原始素材路径
  page_rank: 0.12,                    // 中心性指标
  cluster_id: 3,                      // 所属知识群落
  created_at: datetime,
  updated_at: datetime
})
```

#### Concept（概念节点）

理论、方法论、框架等抽象概念。

```
(:Concept {
  id: "知识编译范式",
  name: "知识编译范式",
  type: "methodology",
  content: "完整 Markdown 内容...",
  summary: "一种将原始信息转化为结构化知识的方法论",
  ...
})
```

#### Source（来源节点）

原始素材的元数据记录。

```
(:Source {
  id: "2024-01-15-rag-paper",
  name: "RAG Paper",
  raw_path: "raw/sources/2024-01-15-rag-paper.md",
  raw_content: "原始素材完整内容...",
  format: "article",
  word_count: 3200,
  ...
})
```

#### Cluster（知识群落节点）

Leiden 聚类算法自动发现的知识群落。

```
(:Cluster {
  id: 3,
  label: "图谱增强技术",
  summary: "该群落包含基于知识图谱的检索增强技术...",
  node_count: 8
})
```

### 4.2 Neo4j 边类型

| 边标签 | 含义 | 关键属性 | 示例 |
|--------|------|----------|------|
| `EXPLICIT` | 显式关系（素材分析直接提取） | `type`(uses/related_to/contains/...), `context`, `created_at` | `(RAG)-[:EXPLICIT {type:"uses"}]->(向量数据库)` |
| `IMPLICIT` | 隐式关系（LLM 推理发现）★ | `type`(depends_on/trade_off/...), `confidence`(0-1), `evidence`, `discovered_at` | `(RAG)-[:IMPLICIT {type:"depends_on", confidence:0.85}]->(语义理解)` |
| `BELONGS_TO` | 群落归属（节点 → Cluster） | `assigned_at` | `(RAG)-[:BELONGS_TO]->(Cluster{id:3})` |
| `DERIVED_FROM` | 知识来源（节点 → Source） | `relevance`(0-1) | `(Entity)-[:DERIVED_FROM]->(Source)` |

### 4.3 完整内容 vs 传统三元组

传统 GraphRAG 只存三元组（头实体-关系-尾实体），丢失大量上下文。本方案在节点中存储完整内容：

```
传统 GraphRAG：
  节点：(RAG, {name: "RAG"})
  边：(RAG)-[:USES]->(向量数据库)
  → 丢失了"为什么用"、"怎么用"的完整解释

V1.0 本方案：
  节点：(RAG, {content: "## Overview\nRAG (Retrieval Augmented Generation) 是一种...(2000字完整内容)"})
  显式边：(RAG)-[:EXPLICIT {type: "uses"}]->(向量数据库)
  隐式边：(RAG)-[:IMPLICIT {type: "depends_on", confidence: 0.85}]->(语义理解)
  → 推理时可读取完整上下文，不丢失信息
```

## 五、输入管道（Ingest Pipeline）

### 5.1 输入流全景

输入管道由 `IngestPipeline` 类（[pipeline.py](file:///Users/sam/Desktop/个人知识库/kb/app/ingest/pipeline.py)）编排，是一个**异步四步处理流程**，每步自动计时并更新任务状态。

```
用户输入素材
     │
     ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 预处理（Preprocessor，无 LLM）                       │
│  格式统一 → Markdown → 存入 raw/sources/（不可变）            │
│  提取元数据（标题、来源、日期、标签、词数）                    │
│  LLM 调用：0 次                                               │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: 分析分类（Analyzer，LLM 调用 1）                     │
│  输入：raw 素材 + Neo4j 现有节点摘要                          │
│  产出：AnalysisReport（类型 + 实体 + 关系 + 矛盾 + 缺口）     │
│  LLM 调用：1 次（qwen-turbo）                                 │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: Graph 处理（GraphProcessor，LLM 调用 1-2）★核心      │
│  3a. 快速路径：直接创建节点和显式边（无 LLM）                  │
│  3b. 后台 LLM：页面内容编写（qwen3.5-plus）                    │
│  3c. 后台 LLM：隐式关系推理（qwen-flash）+ 写入 IMPLICIT 边    │
│  3d. 图结构更新（Leiden 聚类 / PageRank）                     │
│  LLM 调用：0-2 次（快速路径 0，后台 2）                       │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 4: Markdown 渲染（MarkdownRenderer，无 LLM）            │
│  Neo4j 节点 content → .md 文件                                │
│  关系 → [[wikilinks]]                                         │
│  更新 index.md / log.md / overview.md                         │
│  LLM 调用：0 次                                               │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 Step 1：预处理（Preprocessor）

**文件**：[preprocess.py](file:///Users/sam/Desktop/个人知识库/kb/app/ingest/preprocess.py)

**支持的 6 种输入格式：**

| 格式 | 处理方式 | 产出 |
|------|----------|------|
| `text` | 直接使用 | Markdown |
| `markdown` | 直接使用 | Markdown |
| `url` | HTTP 获取网页内容 → HTML→Markdown 清洗（去广告/导航） | Markdown |
| `file` | 根据 MIME 类型分发处理（PDF→pymupdf, Word→python-docx, 图片→OCR） | Markdown |
| `image` | OCR 文字提取 | Markdown |
| `audio` | 语音转文字 | Markdown |

**核心特性：**
- Raw 素材存储至 `raw/sources/<YYYY-MM-DD>-<标题>.md`，**不可变原则**（存入后永不修改或删除）
- 自动提取元数据（标题、来源、日期、标签、格式、词数）
- 格式统一为 Markdown

### 5.3 Step 2：分析分类（Analyzer）

**文件**：[analyze.py](file:///Users/sam/Desktop/个人知识库/kb/app/ingest/analyze.py)

**素材五类分型：**

| 类型 | 识别信号 | 典型内容 | 编译策略 |
|------|---------|---------|---------|
| **事实型** (`factual`) | 数据、定义、规格、日期 | "RAG 全称是 Retrieval Augmented Generation" | 创建/更新 Entity 节点 |
| **概念型** (`conceptual`) | 理论、方法论、框架、范式 | "知识编译范式是一种将原始信息转化为结构化知识的方法" | 创建 Concept 节点 + 关联实体 |
| **经验型** (`experiential`) | "我发现"、"实践证明"、"踩坑" | "Ollama 本地跑编译，质量差但成本为零" | 创建经验节点 + 关联概念 |
| **对比型** (`comparative`) | "vs"、"比较"、"优缺点" | "LLM Wiki vs GraphRAG：前者成本低但多跳弱" | 创建对比节点 + 关联双方 |
| **关系型** (`relational`) | "导致"、"因为"、"推导" | "RAG 的多跳推理瓶颈导致了 GraphRAG 的出现" | 更新双方节点 + 创建语义边 |

**分析内容：**
1. **实体提取** — 从素材中识别关键实体
2. **存在性判断** — 查询 Neo4j 确定实体是否已存在（`MATCH (n) WHERE n.id = $name`）
3. **关系推理** — 推断新素材与已有知识的关系
4. **矛盾检测** — 三种类型：
   - `factual_conflict`：事实冲突（如 "Meta" vs "Facebook AI"）
   - `opinion_conflict`：观点冲突（两种观点共存）
   - `temporal_update`：时序更新（新信息替换旧信息）
5. **知识缺口识别** — 标注素材中缺失的关联知识

**LLM 配置**：使用 `qwen-turbo`（`dashscope_model_analyze`）

### 5.4 Step 3：Graph 处理（GraphProcessor）★

**文件**：[graph_process.py](file:///Users/sam/Desktop/个人知识库/kb/app/ingest/graph_process.py)（483行，最复杂的模块）

#### 3a. 快速路径（无 LLM，即时完成）

直接根据分析报告创建 Neo4j 节点和显式边，确保 API 快速响应：
- 创建/更新 Entity/Concept 节点（`MERGE` 操作）
- 创建 Source 节点
- 创建显式边（`EXPLICIT`）

#### 3b. 后台 LLM 内容增强

在快速路径完成节点创建后，异步调用 LLM 为每个节点编写完整 Markdown 内容并更新 Neo4j。

**LLM 配置**：使用 `qwen3.5-plus`（`dashscope_model_compile`）

#### 3c. 隐式关系推理（核心增量价值）

**LLM 配置**：使用 `qwen-flash`（`dashscope_model_reasoning`）

**上下文组装**：
```
输入给 LLM 的上下文：
  ├── 新编译节点的完整内容
  ├── 新编译节点的邻居节点摘要（1-2 跳范围）
  └── 现有 Graph 中与该节点相关的子图结构
```

**五种隐式关系类型：**

| 类型 | 语义 | 示例 |
|------|------|------|
| `depends_on` | A 的核心能力依赖 B | RAG depends_on 语义理解 |
| `trade_off` | A 和 B 存在取舍/矛盾 | 向量数据库 trade_off Neo4j |
| `bridges` | A 连接不同知识领域 | 成本控制 bridges 技术选型和项目管理 |
| `evolves_to` | A 是 B 的前身/替代/演进 | RAG evolves_to GraphRAG |
| `solves` | A 解决了 B 的某个问题 | 知识图谱 solves RAG 的多跳推理瓶颈 |

隐式关系存储为 `IMPLICIT` 边，携带 `confidence`（0.0-1.0）和 `evidence`（推理依据）。

#### 3d. 图结构更新

每次 Ingest 后，增量计算以下指标：
- **Leiden 聚类**：使用 Neo4j GDS 库，更新节点 `cluster_id`
- **PageRank**：更新节点 `page_rank`
- **桥接系数**：识别跨界枢纽节点

### 5.5 Step 4：Markdown 渲染（MarkdownRenderer）

**文件**：[render.py](file:///Users/sam/Desktop/个人知识库/kb/app/ingest/render.py)

**渲染逻辑**：
1. 从 Neo4j 读取受影响节点的 `content` 字段
2. 查询节点的所有显式关系 → 渲染为 `[[wikilinks]]`
3. 查询节点的隐式关系 → 标注置信度
4. 写入 `wiki/` 目录下的 `.md` 文件

**特殊文件渲染**：
- `wiki/index.md`：查询所有节点 → 目录列表
- `wiki/log.md`：近 30 天操作日志
- `wiki/overview.md`：全局摘要 + 群落分布

**渲染时机**：每次 Ingest 完成后自动渲染受影响文件

## 六、查询管道（Query Pipeline）

### 6.1 查询流程全景

查询管道由 `QueryPipeline` 类（[pipeline.py](file:///Users/sam/Desktop/个人知识库/kb/app/query/pipeline.py)）编排，三步处理：

```
用户查询
     │
     ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 查询理解（QueryUnderstanding）                      │
│  识别实体 + 关系类型 + 推理深度（0-3）                       │
│  延迟：< 50ms（规则匹配）或 ~200ms（LLM）                    │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: Graph 检索（Retriever）                              │
│  根据查询深度执行对应 Cypher 查询：                           │
│  depth=0: MATCH (n) RETURN n.content（< 10ms）              │
│  depth=1: MATCH (n)-[r]->(m)（< 50ms）                      │
│  depth=2: MATCH path = (a)-[*1..3]->(b) + 隐式关系（<200ms）│
│  depth=3: 整群落扫描 + 隐式关系（< 500ms）                   │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: LLM 生成回答（Generator）                            │
│  输入：用户查询 + Graph 检索结果                             │
│  ├── 节点完整内容（精确上下文）                              │
│  ├── 显式关系路径（关联线索）                                │
│  └── 隐式关系（推理线索，含置信度）                          │
│  回复含：answer + sources + confidence                       │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 四种查询深度

| 深度 | 类型 | 适用场景 | Cypher 策略 | 延迟 |
|------|------|---------|------------|------|
| 0 | 事实查询 (`factual`) | "什么是 RAG？" | 直接返回匹配节点的完整 content | < 10ms |
| 1 | 关联查询 (`relational`) | "RAG 用了哪些技术？" | 一跳邻域遍历 | < 50ms |
| 2 | 推理查询 (`reasoning`) | "RAG 和 GraphRAG 的关系？" | 多跳路径遍历 + 隐式关系边 | < 200ms |
| 3 | 全局查询 (`global`) | "知识库中检索技术全景？" | 扫描整个知识群落的所有节点和隐式关系 | < 500ms |

### 6.3 为什么不需要 Tier 路由

传统双存储架构需要 Tier 路由的原因是 Wiki 和 Graph 是两个独立存储。本方案只有 Neo4j 一个数据库：

```
本方案：所有 Tier → Neo4j（统一路径）
  → Graph 内部根据查询复杂度自动选择检索深度
  → 简单查询 Neo4j 直接返回（< 10ms，比读 Wiki 文件还快）
  → 复杂查询 Neo4j 路径遍历 + 隐式关系（< 500ms）
  → 不需要外部路由层
```

### 6.4 查询回答结构

**请求**：`POST /api/query` → `{"question": "RAG 和 GraphRAG 有什么关系？"}`

**响应**（`QueryResult`）：
```json
{
  "answer": "RAG 和 GraphRAG 都是检索增强生成技术。RAG 通过向量检索获取相关文档...",
  "sources": [
    {"node_id": "RAG", "node_name": "RAG", "relevance": 0.95},
    {"node_id": "GraphRAG", "node_name": "GraphRAG", "relevance": 0.90}
  ],
  "implicit_relations_used": [
    {"source": "RAG", "target": "GraphRAG", "type": "evolves_to", "confidence": 0.80}
  ],
  "confidence": 0.85,
  "query_type": "reasoning",
  "depth": 2
}
```

飞书交互采用**三级渐进式反馈**，避免用户等待焦虑：

```
Level 1（即时确认，< 1 秒）
  ─────────────────────────
  收到消息后立即回复确认卡片
  显示："已收到，正在处理中..."
  让用户知道系统已接收

Level 2（分类反馈，2-5 秒）
  ─────────────────────────

Level 3（完成通知，10-20 秒）
  ─────────────────────────
  全部处理完成后更新卡片
  显示：创建的节点、发现的隐式关系、耗时统计
  附带可点击的节点链接
```

### 10.2 分环节模型分配

为实现性能与成本的最优平衡，系统对输入管道的不同环节分配**不同的模型**：

| 环节 | 配置项 | 默认模型 | 选型理由 |
|------|--------|---------|---------|
| Step 2 分析分类 | `DASHSCOPE_MODEL_ANALYZE` | `qwen-turbo` | 分类任务简单，turbo 足够（最快、最便宜） |
| Step 3b 页面编写 | `DASHSCOPE_MODEL_COMPILE` | `qwen3.5-plus` | 内容质量关键环节，需强模型 |
| Step 3c 隐式推理 | `DASHSCOPE_MODEL_REASONING` | `qwen-flash` | 结构化推理，flash 够用（平衡速度与成本） |
| Query 回答生成 | `DASHSCOPE_MODEL_QUERY` | `qwen3.5-plus` | 前端回答质量直接面向用户 |

### 10.3 模型能力对比

| 模型 | 参数量 | 速度 | 推理质量 | 成本（每1K token） | 适用场景 |
|------|--------|------|----------|-------------------|---------|
| Ollama Qwen2.5-7B | 7B | 慢（本地） | 中等 | $0 | 离线使用/开发测试 |
| qwen-turbo | — | 快 | 基本够用 | 最低 | 简单分类任务 |
| qwen-flash | — | 极快 | 中等 | 低 | 隐式关系推理 |
| qwen3.5-plus | — | 快 | 强 | 中 | 内容编译/回答生成 |

### 10.4 连通性检查

`/health` 端点自动检查 LLM 配置状态：如果 `DASHSCOPE_API_KEY` 已配置，则报告 `llm: ok (dashscope/qwen3.5-plus)`；未配置则报告 `llm: not_configured`。

## 十三、质量保障（Lint）

### 13.1 检查机制

**实现文件**：[lint/checker.py](file:///Users/sam/Desktop/个人知识库/kb/app/lint/checker.py)（160行）

| 检查项 | 类型 | 检测方法 | 处理方式 |
|--------|------|----------|----------|
| **孤儿节点** | 确定性 | `MATCH (n) WHERE NOT (n)--() RETURN n` | 报告（列出所有无连接节点） |
| **重复节点** | 确定性 | 名称/标签/摘要相似度匹配 | 报告（建议合并） |
| **低置信度边** | 确定性 | `confidence < 0.3` 的 `IMPLICIT` 边 | 标记待审 |
| **断裂关系** | 确定性 | `MATCH ()-[r]->(m) WHERE m IS NULL RETURN r` | 删除无效边 |
| **事实矛盾** | 启发式 | 跨节点内容对比 | 报告 |
| **过时内容** | 启发式 | 时间戳 + 内容时效性判断 | 报告 |
| **隐式实体** | 启发式 | 高频概念统计，无对应节点 | 建议创建 |

### 13.2 触发机制

```
手动触发：GET /api/graph/lint
定期触发：每日自动执行（cron）
阈值触发：知识节点数增长 > 10% 时自动触发
```

### 13.3 Lint 报告结构

```json
{
  "orphan_nodes": [
    {"id": "orphan-1", "name": "孤立节点名", "label": "Entity"}
  ],
  "duplicate_groups": [
    {"entities": ["RAG", "Retrieval-Augmented-Generation"], "similarity": 0.92}
  ],
  "low_confidence_edges": [
    {"source": "A", "target": "B", "type": "depends_on", "confidence": 0.25}
  ],
  "broken_relations": [
    {"from": "DeletedNode", "to": null, "type": "EXPLICIT"}
  ],
  "fixed_count": 3
}
```

### 13.4 修复能力

`fixed_count` 字段记录自动修复的数量。当前支持：
- 自动删除断裂关系（`MATCH ()-[r]->() WHERE endNode(r) IS NULL DELETE r`）

## 十五、架构决策记录（ADR）

### ADR-001：Graph-First 统一存储

- **日期**：2026-06
- **状态**：已采纳
- **决策**：Neo4j 作为唯一数据存储，Markdown 为只读渲染产物
- **替代方案**：LLM Wiki 为主 + Graph 增强（v1）/ Wiki + Graph 双存储（v2）
- **理由**：
  - 消除双存储同步复杂度和一致性风险
  - 消除"回写循环"的五重安全保障需求
  - 消除查询路由层的必要性
  - 简单、一致、推理深度有保障

### ADR-002：Neo4j 节点存储完整内容

- **日期**：2026-06
- **状态**：已采纳
- **决策**：Neo4j 节点 `content` 字段存储完整 Markdown 页面内容（非仅三元组）
- **替代方案**：传统 GraphRAG 三元组存储
- **理由**：
  - 保留完整上下文，推理时不丢失信息
  - 渲染 Markdown 时无需额外查询
  - 虽然增加存储开销，但个人知识库规模有限（千级节点），可忽略

### ADR-003：Markdown 只读渲染

- **日期**：2026-06
- **状态**：已采纳
- **决策**：Markdown 文件不接受人类编辑，仅由 Neo4j 渲染器单向生成
- **理由**：
  - 保证 Neo4j 作为 SSOT 的权威性
  - 避免双向同步的复杂性
  - Obsidian 只读浏览仍能提供良好的阅读体验

### ADR-004：分环节模型分配

- **日期**：2026-06
- **状态**：已采纳
- **决策**：输入管道的不同环节使用不同的 LLM 模型
- **配置**：分析→qwen-turbo / 编译→qwen3.5-plus / 推理→qwen-flash
- **理由**：
  - 分类和推理任务不需要最强模型
  - 编译质量直接决定知识库内容质量，需要强模型
  - 在性能与成本之间取得最优平衡

### ADR-005：双 LLM 提供商

- **日期**：2026-06
- **状态**：已采纳
- **决策**：同时支持 Ollama 本地和 DashScope API
- **理由**：
  - Ollama 零成本，适合离线使用和开发测试
  - DashScope API 质量高、速度快，适合生产环境
  - 通过 `LLM_PROVIDER` 一键切换

### ADR-006：后台异步内容增强

- **日期**：2026-06
- **状态**：已采纳
- **决策**：Ingest 采用"快速路径 + 后台增强"模式
- **理由**：
  - 快速路径（1-3 秒）确保 API 及时响应
  - 后台 LLM 内容编写和隐式推理不阻塞用户
  - 飞书反馈用三级渐进式卡片告知处理进度

## 十六、已知局限与未来方向

### 16.1 V1.0 已知局限

| 局限 | 影响 | 优先级 |
|------|------|--------|
| **单机部署** | Neo4j + FastAPI 均单机运行，无高可用 | P2 |
| **无向量搜索** | 仅支持 Cypher 关键词模糊匹配，不支持语义相似度搜索 | P1 |
| **无权限管理** | 所有 API 调用共享同一个 Bearer Token | P2 |
| **无 Web UI** | 图谱可视化仅能通过 Obsidian 或 Neo4j Browser | P1 |
| **内存任务存储** | 异步任务状态存储在内存 dict 中，重启丢失 | P2 |
| **隐式关系准确性** | 隐式关系依赖 LLM 推理，存在误判可能 | P1 |
| **无增量 Markdown 渲染** | 每次 Ingest 后全量重渲染受影响文件，而非 diff 级增量 | P3 |
| **无数据备份机制** | Neo4j 数据未配置自动备份 | P2 |

### 16.2 V2.0 规划方向

| 方向 | 说明 |
|------|------|
| **向量搜索集成** | 在 Neo4j 中集成向量索引（Embedding），支持语义相似度搜索 |
| **Web UI** | 基于 vis-network 构建交互式知识图谱探索界面 |
| **权限体系** | 多用户 API Key 管理，区分读写权限 |
| **高可用** | Neo4j 集群 / FastAPI 多副本部署 |
| **知识自动过期** | 基于时效性自动标记过时知识 |
| **多模态增强** | 图片描述生成、音频转录质量提升 |
| **知识导出** | 支持导出为标准格式（JSON-LD、RDF） |

## 十八、变更管理（全自动化 CI/CD 体系）

### 18.1 标准化开发-验证-发布全链路流程

系统实现了 5 层自动化验证-发布链路，开发者只需 `git commit` + `git push`，其余全部自动完成：

```
开发者修改代码
    │
    ▼
[Layer 1] git commit
    │  └─ pre-commit hook 自动触发
    │      ├─ Python 编译检查（仅暂存的 .py 文件）
    │      └─ pytest（仅 kb/ 有变更时）
    │
    ▼
[Layer 2] git push
    │  └─ pre-push hook 自动触发
    │      └─ pre-deploy-check.sh 完整 6 步验证
    │          ├─ Python 编译检查（全量）
    │          ├─ 应用导入检查
    │          ├─ pytest 全量测试
    │          ├─ 前端 TypeScript 类型检查
    │          ├─ 前端构建验证
    │          └─ Git 工作区状态检查
    │
    ▼
[Layer 3] GitHub Actions CI（云端并行）
    │  ├─ backend-test: Python 编译 + pytest
    │  └─ frontend-build: TypeScript 检查 + Vite 构建
    │
    ▼
[Layer 4] GitHub Actions CD（CI 通过后自动触发）
    │  ├─ 上传运维脚本（backup.sh 等）
    │  ├─ ECS 备份（backup.sh）
    │  ├─ rsync 上传后端代码
    │  ├─ pip install 依赖
    │  ├─ systemctl restart
    │  └─ /health 健康检查（5 次重试）
    │
    ▼
[Layer 5] 生产环境
       ├─ 服务自动恢复
       └─ GitHub Step Summary 输出部署报告
```

### 18.2 核心能力组件抽象

| 能力组件 | 实现文件 | 职责 |
|----------|----------|------|
| **Git Hooks 自动化** | `hooks/pre-commit`, `hooks/pre-push` | 本地 commit/push 门禁 |
| **Hook 安装器** | `setup-hooks.sh` | 符号链接 `hooks/` → `.git/hooks/` |
| **本地验证套件** | `pre-deploy-check.sh` | 6 步全量验证（编译/导入/pytest/TS/build/git） |
| **CI 流水线** | `.github/workflows/ci.yml` | 云端并行测试（backend + frontend） |
| **CD 流水线** | `.github/workflows/ci.yml` deploy job | 自动部署到 ECS（备份→上传→重启→验证） |
| **一键部署** | `deploy.sh` | 手动部署备选方案（支持 --dry-run / --skip-backup） |
| **一键回滚** | `rollback.sh` | 代码回滚 / 数据回滚（--code-only / --with-data） |
| **变更记录** | `CHANGELOG.md` | Keep a Changelog 规范记录 |
| **部署检查清单** | `DEPLOY_CHECKLIST.md` | 9 项人工确认 + Neo4j 影响评估 |
| **数据备份** | `backup.sh` | Neo4j dump + raw 素材 + .env |

### 18.3 动作映射表

| 流程节点 | 触发方式 | 执行内容 | 失败处理 |
|----------|----------|----------|----------|
| **git commit** | 开发者执行 | pre-commit: 编译检查 + pytest | 阻止 commit，输出错误 |
| **git push** | 开发者执行 | pre-push: pre-deploy-check.sh 6 步验证 | 阻止 push，输出错误 |
| **CI backend-test** | push 自动触发 | Python 编译 + pytest（云端） | CI 标红，CD 不触发 |
| **CI frontend-build** | push 自动触发 | TS 检查 + Vite 构建（云端） | CI 标红，CD 不触发 |
| **CD deploy** | CI 通过自动触发 | 备份→上传→安装→重启→健康检查 | 部署失败，服务可能受损 |
| **健康检查失败** | CD 自动检测 | 输出 journalctl 日志 | GitHub Actions 标红，需手动介入 |
| **手动回滚** | 开发者执行 | `./rollback.sh --code-only` 或 `--with-data` | 回滚到上一版本 |
| **手动部署** | 开发者执行 | `./deploy.sh`（CI/CD 备选方案） | 部署失败时手动排查 |

### 18.4 问题复盘与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **CI pytest 模块导入失败** | `cd kb && pytest` 时 Python 找不到 `app` 模块 | 设置 `working-directory: kb` + `PYTHONPATH: ${{ github.workspace }}/kb` |
| **ECS backup.sh 不存在** | CD 先执行备份，但脚本尚未上传 | 增加 "Upload scripts first" 步骤，先传脚本再执行备份 |
| **ECS kb-web 目录不存在** | rsync 目标父目录不存在 | ECS 仅运行后端 API，移除前端上传步骤 |
| **ECS 部署路径错误** | CI 上传到 `/opt/knowledge-base/kb/` 而非 `/opt/knowledge-base/` | 修正 rsync 目标路径为 `/opt/knowledge-base/`（与 deploy.sh 一致） |
| **pip install 未使用 venv** | CI 直接 pip install，未激活虚拟环境 | 修改为 `source venv/bin/activate && pip install` |

### 18.6 可复用性设计

| 适配维度 | 配置方式 |
|----------|----------|
| **不同语言栈** | 修改 pre-commit 中的编译检查命令 + CI 的 setup 步骤（如 `actions/setup-go`） |

### 18.7 变更记录规范

CHANGELOG.md 采用 [Keep a Changelog](https://keepachangelog.com/zh/1.1.0/) 规范：

| 标签 | 含义 |
|------|------|
| `CRITICAL` | 数据完整性修复（Neo4j schema/约束变更） |
| `BREAKING` | 不兼容的 API 或数据结构变更 |
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `perf` | 性能优化 |
| `refactor` | 架构调整（不改变外部行为） |
| `test` | 测试相关变更 |
| `deploy` | 部署/运维相关 |

每条记录包含：变更描述、commit 链接、影响范围、验证结果、部署状态（pending / deployed / rolled-back）。
