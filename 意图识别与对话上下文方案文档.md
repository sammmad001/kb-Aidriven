## 1. 设计背景

飞书 Bot 接收到的用户消息需要自动分类为三种意图：

| 意图 | 说明 | 处理流向 |
|------|------|---------|
| **query** | 用户想查询知识库 | → 查询管道（V1.2 五步架构） |
| **input** | 用户想存入新知识 | → Ingest 管道 |
| **research** | 用户想进行深度研究 | → MiroMind 研究管道 |

### 设计原则

**安全偏向 INPUT**：当判断有歧义时，默认为 input。

- 误判 query → input：用户多发一条 `/q` 即可纠正，损失小。
- 误判 input → query：导致**静默数据丢失**，用户以为已存入但实际被丢弃，损失大。

### 检测优先级

**RESEARCH 先于 QUERY 检测**：因为 "帮我研究 X？" 这类句子以问号结尾，但同时包含研究关键词，应路由到 MiroMind 而非普通查询。

## 2. 三层意图识别架构

### 2.1 架构总览

```
用户消息
    │
    ▼
┌──────────────────────────────────┐
│ Layer 1: 显式命令前缀（0ms）       │  /q, /search, /research → parse_command 处理
│ 到此层已返回，不再进入 detect()    │
└──────────────────────────────────┘
    │ （无命令前缀的自然语言）
    ▼
┌──────────────────────────────────┐
│ Layer 2: 规则引擎（0ms，同步）     │  正则匹配，覆盖 80%+ 场景
│  2a: RESEARCH 信号                │  → "research"
│  2b: QUERY 信号                   │  → "query"
│  2b: INPUT 信号                   │  → "input"
│  2c: Follow-up 指代词（需上下文）   │  → "query"
└──────────────────────────────────┘
    │ （规则无法判断的模糊文本）
    ▼
┌──────────────────────────────────┐
│ Layer 3: LLM 快速分类（~300ms）   │  deepseek-v4-flash
│  超时=2s，任何错误 → INPUT        │  仅处理 Layer 2 无法覆盖的边缘 case
└──────────────────────────────────┘
```

### 2.2 Layer 1：显式命令前缀

用户通过显式前缀直接指定意图，由 `parse_command()` 处理，不进入 `detect()` 流程：

| 命令 | 意图 | 说明 |
|------|------|------|
| `/q`、`/search` | query | 直接查询知识库 |
| `/research` | research | 触发 MiroMind 深度研究 |

### 2.3 Layer 2：规则引擎（Rule-Based Detection）

#### 2.3.1 RESEARCH 信号（优先检测）

```python
_RESEARCH_PATTERNS = [
    re.compile(r"(深度研究|深入研究|帮我研究|研究一下|深度分析|详细分析|全面分析)"),
    re.compile(r"(用miromind|调用miromind|用MiroMind|调用MiroMind)", re.IGNORECASE),
    re.compile(r"(帮我.{0,4}研究|请.{0,4}研究|帮我.{0,4}分析)"),
]
```

匹配示例：
- "帮我深度研究一下 RAG 架构" → **research**
- "用 MiroMind 分析 Transformer 原理" → **research**
- "全面分析向量数据库的选型方案？" → **research**（虽然以 ？结尾，但 RESEARCH 优先）

#### 2.3.2 QUERY 信号

```python
_QUERY_PATTERNS = [
    re.compile(r"[？?]\s*$"),                        # 以 ? 或 ？ 结尾
    re.compile(r"^(什么是|怎么|如何|为什么|为何|哪里|哪个|哪些|多少|是不是|有没有|能不能|可不可以|会不会)"),
    re.compile(r"^(what|how|why|where|which|who|when|is|are|can|do|does)\b", re.IGNORECASE),
    re.compile(r"(区别|差异|对比|比较|关系|联系|原因|原理|定义|介绍|解释)"),
]
```

匹配示例：
- "什么是知识图谱？" → **query**
- "RAG 和 Fine-tuning 的区别" → **query**
- "How does attention work?" → **query**

#### 2.3.3 INPUT 信号

```python
_INPUT_PATTERNS = [
    re.compile(r"https?://\S+"),                     # 包含 URL
]
_INPUT_MIN_LENGTH = 100  # 超过 100 字符判定为 input
```

判定规则：
1. 文本长度 > 100 字符 → **input**
2. 文本包含换行符（多段落） → **input**
3. 文本包含 URL → **input**

#### 2.3.4 Follow-up 指代词（需活跃上下文）

```python
_FOLLOWUP_PATTERNS = [
    re.compile(r"(它的|它的|这个|那个|该|此|前面提到的|刚才说的)"),
    re.compile(r"(呢|嘛|吗|吧)\s*[？?]?\s*$"),
]
```

**条件**：仅当 `has_recent_query = True`（用户有活跃的对话上下文）时才检查此规则。

匹配示例：
- 用户刚问了 "什么是 RAG？"，随后说 "它的优点呢" → **query**（follow-up）
- 用户无上下文，说 "它的优点呢" → 进入 Layer 3 LLM 判断

### 2.4 Layer 3：LLM 快速分类

当 Layer 2 无法判断时，调用 LLM 进行兜底分类：

```python
class IntentDetector:
    def __init__(
        self,
        llm: LLMClient | None = None,
        llm_model: str = "deepseek-v4-flash",     # 快速模型
        llm_timeout: float = 2.0,                   # 2秒超时
    ) -> None:
```

#### LLM Prompt

```
System:
你是一个意图分类器。判断用户输入是要查询知识库、存入知识库、还是深度研究。
查询：提问、寻求信息、寻找关系。
存入：分享知识、笔记、陈述事实、记录信息。
研究：要求深度研究、深入分析、全面调研某个话题。
只输出一个词：QUERY、INPUT 或 RESEARCH

User:
用户输入：{text[:200]}

请判断意图，只输出 QUERY、INPUT 或 RESEARCH：
```

#### 容错策略

| 异常场景 | 返回值 | 原因 |
|---------|--------|------|
| LLM 输出包含 "RESEARCH" | `research` | — |
| LLM 输出包含 "QUERY" | `query` | — |
| LLM 输出包含 "INPUT" | `input` | — |
| LLM 输出无法解析 | `input` | 安全默认 |
| `asyncio.TimeoutError`（>2s） | `input` | 超时不阻塞用户 |
| 任何其他异常 | `input` | 安全默认 |

## 3. IntentDetector 类设计

### 3.1 类结构

```python
IntentType = Literal["query", "input", "research"]

class IntentDetector:
    def __init__(self, llm, llm_model, llm_timeout) -> None
    async def detect(self, text: str, has_recent_query: bool = False) -> IntentType
    async def _llm_detect(self, text: str) -> IntentType
```

### 3.2 detect() 方法流程

```
1. 空文本 → return "input"

2. RESEARCH 规则匹配 → return "research"

3. QUERY 规则匹配 → return "query"

4. INPUT 规则匹配（长度/段落/URL）→ return "input"

5. has_recent_query 且 FOLLOWUP 规则匹配 → return "query"

6. LLM 可用？→ 调用 _llm_detect()
   否则 → return "input"（安全默认）
```

### 3.3 初始化

在 `handlers.py` 中由 `init_intent_context()` 注入：

```python
_intent_detector: IntentDetector | None = None
_context: ConversationContext | None = None

def init_intent_context(intent_detector, context):
    global _intent_detector, _context
    _intent_detector = intent_detector
    _context = context
```

## 4. 对话上下文管理（ConversationContext）

### 4.1 设计目的

让用户可以进行连续的追问对话，而不需要每次都输入完整的实体名称。

**示例**：
```
用户：什么是 RAG？
Bot：  RAG（Retrieval-Augmented Generation）是一种结合检索和生成的技术...

用户：它的优点呢？
       ↓ 富化后
       "RAG的优点呢？"
Bot：  RAG 的主要优点包括：1. 减少幻觉...
```

### 4.2 数据结构

```python
@dataclass
class ConversationTurn:
    """单轮对话记录"""
    question: str                           # 用户原始问题
    answer: str                             # Bot 回答
    entities: list[str] = []                # 从回答中提取的实体（≤10）
    timestamp: float = time.time()          # 创建时间戳（TTL 判断用）
```

```python
class ConversationContext:
    def __init__(self, max_turns=5, ttl_seconds=600):
        self._max_turns = max_turns
        self._ttl = ttl_seconds
        # user_id → OrderedDict[turn_key, ConversationTurn]
        self._contexts: dict[str, OrderedDict] = {}
        # 单调递增计数器，确保 key 唯一
        self._counter = itertools.count()
```

### 4.3 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_turns` | 5 | 每用户最多保留 5 轮对话 |
| `ttl_seconds` | 600s (10分钟) | 对话上下文过期时间 |

### 4.4 核心方法

#### add_turn() — 记录一轮 Q&A

```python
def add_turn(self, user_id, question, answer, entities=None):
```

- 如果 `entities` 未传入，自动从回答中提取 `**bold**` 标记的实体
- 超过 `max_turns` 时，自动驱逐最旧的一轮（FIFO）
- 使用 `itertools.count()` 保证 key 唯一，即使同一毫秒添加多条

#### has_active_context() — 判断是否有活跃上下文

```python
def has_active_context(self, user_id) -> bool:
```

供 `IntentDetector.detect()` 的 `has_recent_query` 参数使用。

#### enrich_followup() — Follow-up 富化

```python
def enrich_followup(self, user_id, text) -> str:
```

1. 检查文本是否包含指代词（`_REFERENCE_PATTERNS`）
2. 获取最近一轮对话的实体列表
3. 将指代词替换为 `primary_entity的`
4. 例：`"它的优点"` → `"RAG的优点"`

**指代词模式**：

```python
_REFERENCE_PATTERNS = [
    re.compile(r"(它的|这个|那个|该|此|前面提到的|刚才说的)"),
    re.compile(r"(它|这|那)(有|是|能|会|可以)"),
]
```

#### get_context_history() — 获取 LLM 上下文历史

```python
def get_context_history(self, user_id) -> list[dict]:
```

返回格式（OpenAI Chat 格式）：

```python
[
    {"role": "user", "content": "什么是RAG？"},
    {"role": "assistant", "content": "RAG是一种..."},
    {"role": "user", "content": "它的优点呢？"},
    {"role": "assistant", "content": "RAG的优点包括..."},
]
```

此列表传入 `QueryRequest.context_history`，供查询管道的 LLM 使用。

### 4.5 实体提取

```python
_ENTITY_PATTERN = re.compile(r"\*\*(.+?)\*\*")
```

从 Bot 回答中的 `**bold**` Markdown 标记提取实体名称。

规则：
- 去重保序
- 长度 < 50 字符（过滤误匹配）
- 最多保留 10 个实体

### 4.6 TTL 过期清理

`_evict_expired()` 在每次 `get_history()` / `has_active_context()` 调用时自动触发：
- 从 OrderedDict 头部开始检查（FIFO 有序）
- `timestamp < now - ttl` 的 turn 被驱逐
- 当用户的上下文 dict 为空时，从 `_contexts` 中移除该用户

### 4.7 清除上下文

用户可通过 `/end` 命令主动清除对话上下文：

```python
def clear(self, user_id) -> None:
    self._contexts.pop(user_id, None)
```

## 5. 集成流程

### 5.1 完整消息处理流程

```
飞书消息到达
    │
    ▼
handlers.py: _handle_text_message()
    │
    ├─ 1. parse_command() → 显式命令？ → 直接路由
    │
    ├─ 2. 获取 has_ctx:
    │     has_ctx = _context.has_active_context(user_id)
    │
    ├─ 3. 意图检测:
    │     intent = await _intent_detector.detect(text, has_recent_query=has_ctx)
    │
    ├─ 4. 路由处理:
    │     intent == "query"   → _handle_query()
    │     intent == "input"   → _handle_ingest()
    │     intent == "research" → _handle_research()
    │
    └─ 5. query 处理中:
          ├─ enrich:  enriched = _context.enrich_followup(user_id, question)
          ├─ history: context_history = _context.get_context_history(user_id)
          ├─ 执行查询
          └─ 记录:    _context.add_turn(user_id, question, answer)
```

### 5.2 handlers.py 关键集成点

```python
# 意图检测
user_id = Neo4jDatabase.get_current_user_id_or_default()
has_ctx = _context.has_active_context(user_id) if _context else False
if _intent_detector:
    intent = await _intent_detector.detect(text, has_recent_query=has_ctx)
else:
    intent = "input"  # 检测器不可用时安全默认

# 查询富化
context_history = []
if _context:
    enriched_question = _context.enrich_followup(user_id, question)
    context_history = _context.get_context_history(user_id)
```

## 7. 性能分析

| 检测层 | 响应时间 | 覆盖率 | 说明 |
|--------|---------|--------|------|
| Layer 1（显式命令） | 0ms | ~20% | 用户主动使用 `/q` 等 |
| Layer 2（规则引擎） | 0ms | ~70% | 正则匹配，覆盖常见场景 |
| Layer 3（LLM 分类） | ~300ms | ~10% | 仅处理模糊文本，2s 超时 |

**总延迟预期**：
- 90% 的消息：0ms（Layer 1+2 即可判断）
- 10% 的消息：~300ms（需要 LLM 兜底）
- 极端情况：2s（LLM 超时，降级为 input）
