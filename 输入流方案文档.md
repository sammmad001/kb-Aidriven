## 一、输入流概述

### 1.1 设计原则

```
原则 1：Graph 是唯一的写入目标
  所有知识最终写入 Neo4j，不存在独立的 Wiki 存储
  Markdown 文件是 Neo4j 的只读渲染产物

原则 2：一次处理，全部到位
  一次 Ingest 同时完成：节点创建 + 显式关系 + 隐式关系推理 + 图结构更新
  不需要"先编译 Wiki → 再增强 Graph → 再回写 Wiki"的串行过程

原则 3：隐式关系是核心增量价值
  Graph 增强层通过 LLM 推理发现 Wiki 文本中不被显式记录的隐式关联
  这是 Graph 超越纯文本存储的关键能力
```

## 二、Step 1：预处理层

### 2.1 支持的输入格式

| 输入类型 | 枚举值 | 预处理方式 | 产出格式 |
|---------|---------|----------|---------|
| 文档 / PDF | FILE | pymupdf 提取文本，保留标题层级 | Markdown |
| 网页 / 文章 | URL | HTML→Markdown 清洗（去广告/导航） | Markdown |
| 笔记 / 想法 | TEXT | 无需预处理 | Markdown |
| Markdown | MARKDOWN | 元数据提取 + 标准化 | Markdown |
| 代码片段 | FILE | 包裹在代码块中，标注语言类型 | Markdown |
| 对话记录 | TEXT | 结构化提取（区分角色） | Markdown |
| 图片（含 OCR） | IMAGE | PaddleOCR 提取文本（置信度 < 0.6 时 qwen-vl-max fallback） | Markdown |
| 语音消息 | AUDIO | DashScope Paraformer ASR 语音转写 | Markdown |
| 社交媒体（小红书/微博） | URL | SocialFetcher 抓取（Playwright）+ OCR 图片提取 | Markdown |
| MiroMind 研究报告 | ADAPTER | MiroMindAdapter extract/validate/transform | Markdown |

> **V2.0 新增通道**：IMAGE、AUDIO、社交媒体 URL、MiroMind ADAPTER

### 2.2 Raw 素材存储（多用户隔离）

```
命名规范：raw/sources/{user_id}/<YYYY-MM-DD>-<简短标题>.md

V2.0 变更：按 user_id 分目录存储，实现用户级文件系统隔离
  Preprocessor._user_raw_dir(user_id) → raw/sources/{user_id}/

示例：
  raw/sources/user_a01b/2024-01-15-rag-retrieval-augmented-generation.md
  raw/sources/user_a01b/2024-01-20-llm-wiki-karpathy-method.md
  raw/sources/user_c33f/2024-02-03-personal-knowledge-graph-notes.md

不可变原则：
  → Raw 文件一旦存入，永不修改或删除
  → 所有后续处理只读 raw 文件
  → 修正需求通过新的 Ingest 覆盖（Neo4j 节点更新）

文件系统权限：
  → 每个用户只能访问自己 user_id 目录下的文件
  → API 层通过 JWT 中间件注入 current_user.id
  → Neo4j 写入时同步设置 n.user_id = $user_id（复合唯一约束）
```

```
触发：输入类型为 IMAGE 或社交媒体抓取中的图片
```

### 2.5 IngestRetryScheduler（V2.0 新增）

```
功能：自动重试失败的 Ingest 任务

参数：
  轮询间隔：5 分钟
  最大重试次数：max_retries=3
  重试策略：指数退避（5min → 10min → 20min）

工作流：
  ① IngestTracker 标记失败任务为 retry_pending
  ② Scheduler 每 5 分钟扫描 retry_pending 任务
  ③ 重新提交到 ingest pipeline
  ④ 成功 → 标记 completed；失败 → retry_count++
  ⑤ retry_count >= max_retries → 标记 failed（不再重试）

影响文件：maintenance/scheduler.py
```

## 三、Step 2：分析分类层

### 3.1 素材五类分型

素材经过 LLM 分析后，归入以下五种类型之一，每种类型对应不同的编译策略：

| 类型 | 识别信号 | 典型内容 | 编译策略 |
|------|---------|---------|---------|
| **事实型** | 数据、定义、规格、日期、数字 | "RAG 全称是 Retrieval Augmented Generation" | 创建/更新实体节点 |
| **概念型** | 理论、方法论、框架、范式 | "知识编译范式是一种将原始信息转化为结构化知识的方法" | 创建概念节点 + 关联实体 |
| **经验型** | "我发现"、"实践证明"、"踩坑" | "Ollama 本地跑 LLM Wiki，编译质量差但成本为零" | 创建经验节点 + 关联概念 |
| **对比型** | "vs"、"比较"、"优缺点"、"区别" | "LLM Wiki vs GraphRAG：前者成本低但多跳弱" | 创建对比节点 + 关联双方 |
| **关系型** | "导致"、"因为"、"推导"、"因果" | "RAG 的多跳推理瓶颈导致了 GraphRAG 的出现" | 更新双方节点 + 创建语义边 |

### 3.4 矛盾检测

```
检测到实体已存在时，对比以下维度：

事实冲突：
  Neo4j 中："RAG 由 Meta 在 2020 年提出"
  新素材中："RAG 由 Facebook AI Research 在 2020 年提出"
  → 标注为 "factual_conflict"（可能等价，需确认）
```

## 四、Step 3：Graph 处理层（核心）

### 4.1 编译决策

根据分析报告，确定具体的 Cypher 操作：

| 场景 | 素材类型 | 实体状态 | Neo4j 操作 |
|------|---------|---------|-----------|
| 场景 1 | 事实型 | 新实体 | CREATE (:Entity) |
| 场景 2 | 事实型 | 已有（无冲突） | MATCH + SET 更新 content |
| 场景 3 | 事实型 | 已有（有冲突） | SET + 添加冲突标注 |
| 场景 4 | 概念型 | 新概念 + 已有实体 | CREATE (:Concept) + CREATE 显式边 |
| 场景 5 | 对比型 | 双方已存在 | CREATE (:Comparison) + 关联双方 |
| 场景 6 | 关系型 | 双方已存在 | CREATE 显式边 + 更新双方 content |
| 场景 7 | 综合型 | 多个实体 | CREATE (:Synthesis) + 多条显式边 |

### 4.2 页面内容编写（LLM 调用 2）

LLM 根据分析报告和 raw 素材，编写每个节点的完整 Markdown 内容：

```
你是一个知识库编译器。请根据以下信息编写知识页面内容：

【原始素材】
{raw_content}

【分析报告】
{analysis_report}

【相关已有节点内容摘要】
{related_nodes_summary}

请编写完整的 Markdown 内容，包含：
1. 一句话定义（> 引用格式）
2. 概述（2-5 段，基于素材内容展开）
3. 关键事实（要点列表）
4. 与其他知识的关系（标注 [[wikilinks]])

注意：
- 内容必须完全基于素材，不编造信息
- wikilinks 使用 [[实体名]] 格式
- 保留素材中的关键数据和引用
```

### 4.3 写入 Neo4j

```python
# 示例：创建新实体节点
def create_entity_node(neo4j_driver, entity_data, content):
    query = """
    MERGE (n:Entity {id: $id})
    SET n.name = $name,
        n.type = $type,
        n.content = $content,
        n.summary = $summary,
        n.tags = $tags,
        n.source = $source,
        n.updated_at = datetime()
    SET n.created_at = CASE WHEN n.created_at IS NULL 
                        THEN datetime() ELSE n.created_at END
    RETURN n
    """
    with neo4j_driver.session() as session:
        session.run(query, **entity_data, content=content)

# 示例：创建显式关系
def create_explicit_edge(neo4j_driver, from_id, to_id, edge_type, context):
    query = """
    MATCH (a), (b) WHERE a.id = $from_id AND b.id = $to_id
    MERGE (a)-[r:EXPLICIT {type: $edge_type}]->(b)
    SET r.context = $context,
        r.created_at = datetime()
    RETURN r
    """
    with neo4j_driver.session() as session:
        session.run(query, from_id=from_id, to_id=to_id, 
                   edge_type=edge_type, context=context)
```

### 4.4 隐式关系推理（LLM 调用 3）★ 核心

这是 Graph 提供推理深度的关键步骤。

#### 上下文组装

```
输入给 LLM 的上下文：
  ├── 新编译节点的完整内容
  ├── 新编译节点的邻居节点摘要（1-2 跳范围内的 Neo4j 节点）
  └── 现有 Graph 中与该节点相关的子图结构
```

#### LLM 推理 Prompt

```
你正在分析一个个人知识库的知识图谱。以下是新编译的知识节点及其上下文：

【新节点】{node_name}: {node_content}

【关联节点摘要】
- {node_A}: {summary}
- {node_B}: {summary}

【现有图谱关系】
({entity_X}) -[:EXPLICIT {type: "uses"}]-> ({entity_Y})
({entity_P}) -[:IMPLICIT {type: "depends_on", confidence: 0.8}]-> ({entity_Q})

请发现以下类型的隐式关系：

1. 依赖关系（depends_on）：A 的核心能力依赖 B（但 A 没有直接链接到 B）
2. 对立关系（trade_off）：A 和 B 存在取舍或矛盾
3. 桥接关系（bridges）：A 连接不同知识领域
4. 演化关系（evolves_to）：A 是 B 的前身/替代/演进版本
5. 解决关系（solves）：A 解决了 B 的某个问题

对每个发现的关系，输出 JSON：
{"source": "实体名", "target": "实体名", "type": "关系类型", 
 "confidence": 0.0-1.0, "evidence": "推理依据（一句话）"}

注意：
- 只发现【隐式】关系——如果两个实体之间已有显式关系，不要重复
- 置信度要保守：只有强有力的推理依据才给 > 0.8
```

#### 隐式关系类型定义

| 类型 | 语义 | 示例 |
|------|------|------|
| depends_on | A 的核心能力依赖 B | RAG depends_on 语义理解 |
| trade_off | A 和 B 存在取舍 | 向量数据库 trade_off Neo4j |
| bridges | A 连接不同知识群落 | 成本控制 bridges 技术选型和项目管理 |
| evolves_to | A 是 B 的前身/演进 | RAG evolves_to GraphRAG |
| solves | A 解决了 B 的问题 | 知识图谱 solves RAG 的多跳推理瓶颈 |

```python
    CREATE (a)-[r:IMPLICIT {
        type: $edge_type,
        confidence: $confidence,
        evidence: $evidence,
        discovered_at: datetime()
    }]->(b)
    RETURN r
    """
    with neo4j_driver.session() as session:
        session.run(query, from_id=from_id, to_id=to_id,
                   edge_type=edge_type, confidence=confidence, evidence=evidence)
```

### 4.6 图结构更新

```
每次 Ingest 后，增量重新计算以下图结构指标：

① Leiden 聚类
  ──────────────
  发现知识群落（如"检索技术群落"、"图谱技术群落"、"知识编译群落"）
  Cypher：调用 Neo4j GDS 库的 leiden 算法
  更新：节点的 cluster_id 字段

② PageRank
  ──────────
  识别核心实体（被最多关系指向的节点）
  Cypher：CALL gds.pageRank.stream(...)
  更新：节点的 page_rank 字段

③ 桥接系数
  ──────────
  识别连接不同知识群落的关键节点
  Cypher：CALL gds.betweenness.stream(...)
  更新：存储为 Cluster 节点的 bridge_entities 属性
```

## 五、Step 4：Markdown 渲染层

### 5.1 渲染逻辑

```python
def render_markdown(neo4j_driver, node_id):
    """将 Neo4j 节点渲染为 Markdown 文件"""
    
    # 1. 读取节点数据
    node = neo4j_query("MATCH (n) WHERE n.id = $id RETURN n", id=node_id)
    
    # 2. 读取所有关系
    explicit_rels = neo4j_query("""
        MATCH (n)-[r:EXPLICIT]->(m) WHERE n.id = $id 
        RETURN r.type, m.name, r.context ORDER BY r.type
    """, id=node_id)
    
    implicit_rels = neo4j_query("""
        MATCH (n)-[r:IMPLICIT]->(m) WHERE n.id = $id 
        RETURN r.type, m.name, r.confidence, r.evidence 
        ORDER BY r.confidence DESC
    """, id=node_id)
    
    # 3. 组装 Markdown
    md = f"# {node['name']}\n\n"
    md += f"> {node['summary']}\n\n"
    md += node['content']  # 完整内容
    
    # 显式关系 → wikilinks
    md += "\n\n## Relationships\n"
    for rel in explicit_rels:
        md += f"- **{rel['type']}** → [[{rel['m.name']}]]\n"
    
    # 隐式关系（标注置信度）
    for rel in implicit_rels:
        md += f"- *{rel['type']}* → [[{rel['m.name']}]] ({rel['confidence']:.0%})\n"
    
    # 4. 写入文件
    node_type = node.get('label', 'entities').lower()
    file_path = f"wiki/{node_type}/{node['name']}.md"
    write_file(file_path, md)
```

```

  → Cypher: MATCH (n) RETURN n.name, n.summary, labels(n) ORDER BY n.updated_at DESC

overview.md：
```

### 8.2 检查项

| 检查项 | 类型 | 检测方法 | 处理方式 |
|-------|------|---------|---------|

| 断裂关系 | 确定性 | MATCH ()-[r]->(m) WHERE m IS NULL RETURN r | 删除无效边 |
| 重复节点 | 确定性 | 相似度匹配（name/tags/summary） | 合并 |

| 过时内容 | 启发式 | 时间戳 + 内容时效性判断（LLM） | 报告 |

| 低置信度关系 | 确定性 | confidence < 0.3 的隐式边 | 标记待审 |
