# 前端输入入口方案设计：实施计划

## Context

用户正在设计个人知识库系统（Graph-First 统一存储架构，Neo4j 唯一数据库）。后端 Ingest 管道（四步处理流程）已设计完毕，现需设计**前端输入入口方案**——解决"知识如何从前端送达 ingest() API"的问题。

用户是个人开发者，日常需要将碎片知识快速录入系统，场景覆盖移动端和桌面端。

## 目标

生成一份详细的前端输入入口设计文档 `前端输入入口方案设计文档.md`，涵盖：输入渠道分析、输入方式设计、用户体验方案、技术集成方案、优先级建议。

## 方案核心设计

### 输入渠道选择（按优先级）

| 优先级 | 渠道 | 理由 |
|-------|------|------|
| **P0** | Telegram Bot | API 完全开放、零成本、支持所有消息类型、ECS 海外直连无障碍 |
| **P0** | Web 表单 | 最直接的输入方式、FastAPI 内嵌即可、零第三方依赖 |
| **P0** | CLI (curl) | 零开发量、开发者习惯 |
| **P1** | 飞书自建机器人 | 国内移动端核心渠道、免费创建应用、可接收消息事件 |
| **P1** | 浏览器扩展 | 一键抓取网页内容、选中文本直接发送、输入便捷性最高 |
| **P1** | n8n 消息路由 | 当渠道 >= 3 时引入、统一管理路由和认证 |
| **P2** | Obsidian 插件 | 存量笔记导入场景 |
| **P2** | iOS 快捷指令 | 移动端分享菜单直接发送 |
| **不推荐** | 微信生态 | 个人无法使用消息接口，小程序需备案，投入产出比最低 |

### 统一架构

```
输入渠道层（各渠道只采集素材）→ 消息路由层（n8n，P1引入）→ 统一 Ingest API（FastAPI:8080）
```

- 所有渠道构造同一个 JSON 发给 `POST /api/ingest`
- 新增渠道只需写"适配器"，不修改后端
- 认证：静态 API Token（单用户）
- 前端不做 LLM 调用，只做采集

### 后端 API 扩展需求

1. 异步任务模式（POST → 202 + task_id）
2. 文件上传端点（`POST /api/ingest/file`，multipart/form-data）
3. 任务状态查询（`GET /api/tasks/{task_id}`）
4. URL 内容获取（`POST /api/ingest/url`，后端抓取网页）
5. 认证中间件（Bearer Token）

### 分阶段时间线

- **P0（1.5-2.5天）**：CLI + Web 表单 + Telegram Bot
- **P1（5-7.5天）**：飞书机器人 + 浏览器扩展 + n8n 路由
- **P2（4-5.5天）**：Obsidian 插件 + iOS 快捷指令 + 批量导入
