# 存储架构深度设计：Wiki-Graph 双写入协作架构

> **[已废弃]** 本文档描述的是 Wiki-Graph 双写入架构（v2），已被 V1.0 Graph-First 统一存储架构替代。
> 当前实现请参考 `系统架构设计方案.md` 和 `个人知识库系统-V1.0-产品存档文档.md`。

## Context

用户对上一版存储方案提出两个核心质疑：

1. **为什么不是统一存储？** 为什么不让 Wiki 和 Graph 串行处理信息，最终知识直接落在 Graph 里？Wiki 作为中间处理层，Graph 作为最终存储。
2. **推理深度如何保障？** 知识同步如果没有 Graph 的介入处理（Graph 只是被动镜像 Wiki），能保障推理的深度吗？被动派生的 Graph 不就是 Wiki 的冗余副本吗？

**结论：用户的质疑完全成立。** 上一版设计中 Graph 仅作为被动派生视图，实际上是 Wiki 的"低信息量镜像"——它只复制了 wikilinks 的显式关联，而没有增加任何 Wiki 本身不具备的语义深度。这样的 Graph 在推理链路中提供的价值极为有限。

本方案基于此质疑重新设计存储架构。

## 一、核心问题：被动派生的 Graph 到底有没有用？

### 上一版设计的信息流

```
上一版（Graph 被动派生）：

Raw 素材
   │
   ▼
Wiki 编译引擎 ──→ Wiki 页面（含 [[wikilinks]]）
                        │
                        │ 被动提取
                        ▼
                  Graph 节点 + 边
                  （只是 wikilinks 的结构化副本）
```

### 问题剖析

| 维度 | Wiki 本身能做什么 | 被动 Graph 额外能做什么 | 增量价值 |
|------|-----------------|----------------------|---------|
| 简单事实查询 | ✅ 直接读页面 | ✅ 查节点属性 | **零**——节点属性就是页面内容 |
| 单跳关联 | ✅ wikilinks 跳转 | ✅ 遍历一条边 | **零**——边就是 wikilinks |
| 多跳推理 | ⚠️ 需 LLM 沿 wikilinks 逐步跳转 | ⚠️ 沿边路径遍历 | **极低**——路径就是 wikilinks 链 |
| 全局综合 | ⚠️ 需 LLM 读大量页面 | ⚠️ 社区检测+摘要 | **低**——聚类基于同样的 wikilinks |

**核心问题**：如果 Graph 的节点 = Wiki 页面，Graph 的边 = wikilinks，那 Graph 就是 Wiki 的**另一种表示形式**，不增加任何新信息。所谓"推理深度"的提升只是数据结构层面的（图遍历比文本解析快），而不是**语义层面**的（Graph 不比 Wiki 知道更多东西）。

### 真正需要 Graph 的原因

Graph 要提供超过 Wiki 的推理深度，必须在**知识编译过程中**主动介入，产出 Wiki 本身不具备的语义关系：

```
Wiki 能提供的：显式关联（[[wikilinks]]，人类或 LLM 直接写入的链接）
Wiki 不能提供的：隐式关系（需要推理才能发现的关联）

例如：
- Wiki 页面 "RAG" 有 [[向量数据库]] 的链接           → 显式关联
- Wiki 页面 "RAG" 和 "知识图谱" 都依赖 "语义理解"    → 隐式关系（需要推理发现）
- Wiki 没有一个叫 "语义理解" 的页面，但 20 个页面都涉及这个概念 → 隐式实体（需要提取）
- 页面 A 的方法论可以解决页面 B 中提到的问题           → 跨域应用关系（需要推理）
```

**结论：Graph 的价值不在于"存储 Wiki 已有的信息"，而在于"在编译过程中发现 Wiki 没有显式记录的隐式关系"。**

## 三、Graph 增强的具体内容：Wiki 不具备的语义深度

### 3.1 显式关系 vs 隐式关系

```
Wiki 页面中的显式关系（人类/LLM 直接写入的 wikilinks）：

  [[RAG]] ──uses──→ [[向量数据库]]
  [[RAG]] ──related──→ [[知识图谱]]
  [[LLM Wiki]] ──alternative──→ [[GraphRAG]]


Graph 增强后发现的隐式关系（需要推理才能发现）：

  [[RAG]] ──depends_on──→ [[语义理解]]          ← RAG 的核心能力依赖语义理解
  [[LLM Wiki]] ──superior_in──→ [[个人知识管理]]  ← 在个人场景优于 GraphRAG
  [[知识图谱]] ──solves──→ [[RAG 的多跳推理瓶颈]]  ← 知识图谱解决了 RAG 的多跳问题
  [[向量数据库]] ──trade_off──→ [[Neo4j]]         ← 向量数据库和图数据库的取舍
  [[Obsidian]] ──visualizes──→ [[wikilinks 图谱]]  ← Obsidian 能可视化 wikilinks 网络
```

**这些隐式关系是 Graph 增强层的核心产出，它们不存在于任何单个 Wiki 页面中，而是需要 LLM 在理解多个页面内容后推理得出。**

### 3.2 Graph 增强层产出的四类知识

| 知识类型 | 说明 | 示例 | Wiki 中是否存在 |
|---------|------|------|---------------|
| **隐式关系** | 两个实体间存在但未被 wikilinks 记录的关系 | RAG 对语义理解的依赖 | ❌ 需推理 |
| **隐式实体** | 多个页面频繁提及但无独立页面的概念 | "语义理解"出现 20 次但无页面 | ❌ 需识别 |
| **跨域桥接** | 连接不同知识群落的关键节点/关系 | "成本控制"桥接技术选型和项目管理 | ❌ 需分析 |
| **时序演化** | 知识的演进轨迹和因果时序 | 从 RAG → GraphRAG → 融合方案的技术演进 | ❌ 需推理 |

### 3.3 Graph 增强的 LLM 推理过程

```
输入：新编译的 Wiki 页面 + 现有 Wiki 结构 + 现有 Graph

Step 1：上下文组装
  ──────────────────
  读取新编译的页面内容
  + 读取该页面通过 wikilinks 关联的 1-2 跳邻居页面
  + 读取现有 Graph 中与该页面节点相关的子图结构

Step 2：隐式关系发现（LLM 推理）
  ────────────────────────────────
  Prompt 示例：

  你正在分析一个知识库系统。以下是新编译的 Wiki 页面及其关联上下文：

  【新页面】{页面内容}
  【关联页面摘要】{邻居页面的一行摘要}
  【现有图谱关系】{相关子图的节点和边}

  请发现以下类型的隐式关系：
  1. 依赖关系：A 的核心能力依赖 B（但 A 的页面没有明确链接到 B）
  2. 对立关系：A 和 B 在某个维度上存在矛盾或取舍
  3. 桥接关系：A 是连接两个不同知识领域的枢纽
  4. 演化关系：A 是 B 的前身/替代/演进版本

  对每个发现的关系，给出：
  - 关系类型
  - 置信度（0.0-1.0）
  - 推理依据（一句话解释为什么）

Step 3：隐式实体识别
  ────────────────────
  扫描新页面内容，统计跨页面高频概念
  如果一个概念在 3+ 个页面中出现但没有独立 Wiki 页面
  → 标记为隐式实体
  → 置信度 = 出现次数 / 总页面数

Step 4：图结构分析
  ──────────────────
  将新节点和边加入 Graph 后重新计算：
  - Leiden 聚类（发现知识群落）
  - PageRank（识别核心实体）
  - 桥接系数（识别跨界枢纽）
  - 结构洞（信息流通的关键节点）
```

## 四、知识同步机制：双向协作而非单向派生

### 4.1 新的同步模型

```
╔══════════════════════════════════════════════════════════════╗
║                 知识同步：双向协作模型                        ║
╠══════════════════════════════════════════════════════════════╣
║                                                              ║
║   写入方向（始终从 Raw 出发）：                               ║
║                                                              ║
║   Raw ──→ Wiki 编译 ──→ Graph 增强 ──→ 回写 Wiki            ║
║            │               │               │                 ║
║            │               │               │                 ║
║            ▼               ▼               ▼                 ║
║         Wiki 页面      Graph 节点/边    Wiki 页面更新         ║
║         (显式知识)    (显式+隐式知识)  (隐式知识显式化)        ║
║                                                              ║
║                                                              ║
║   读取方向（查询时）：                                        ║
║                                                              ║
║   Tier 0: Wiki 单独读取（简单事实）                           ║
║   Tier 1: Wiki + wikilinks 遍历（单跳关联）                  ║
║   Tier 2: Wiki + Graph 隐式关系（多跳推理）★ Graph 介入      ║
║   Tier 3: Graph 社区结构 + Wiki 全文（全局综合）              ║
║                                                              ║
╚══════════════════════════════════════════════════════════════╝
```

### 4.2 同步触发的三个阶段

```
阶段 A：Wiki 编译完成（显式知识落地）
─────────────────────────────────────
  触发条件：Ingest 命令执行完毕
  执行动作：
    1. 从编译结果提取显式实体和 wikilinks
    2. 更新 Graph 的节点和显式边
  信息流向：Wiki → Graph（单向，快速）
  延迟：< 100ms（纯解析，无 LLM 调用）


阶段 B：Graph 增强（隐式知识发现）★
──────────────────────────────────────
  触发条件：阶段 A 完成后
  执行动作：
    1. LLM 推理发现隐式关系
    2. 隐式实体识别
    3. 图结构重新计算
  信息流向：Wiki + 旧 Graph → 新 Graph
  延迟：2-5s（需要 1 次 LLM 调用）
  ★ 这是 Graph 提供推理深度的关键步骤


阶段 C：回写增强（隐式知识显式化）
────────────────────────────────────
  触发条件：阶段 B 中发现高置信度（>0.8）的隐式关系
  执行动作：
    1. 在相关 Wiki 页面添加 "Related Concepts" 区域
    2. 如发现隐式实体出现频率 > 阈值 → 创建新 Wiki 页面
    3. 在 index.md 中更新关联
  信息流向：Graph → Wiki（反向传播，有条件触发）
  延迟：< 500ms（文件写入）
  频率：不是每次 Ingest 都触发，只在发现高质量隐式关系时
```

### 4.3 回写的安全保障

回写（Graph → Wiki）是有条件的、受控的，不会导致无限循环：

```
回写条件：

1. 置信度门控
   ────────────
   只有置信度 > 0.8 的隐式关系才触发回写
   低置信度关系只存在于 Graph 中，不回写

2. 去重检测
   ────────────
   回写前检查 Wiki 页面是否已包含该关联
   如果已有（通过 wikilinks 或文字描述）→ 跳过

3. 内容隔离
   ────────────
   回写内容只添加到页面的 "Related Concepts" 区域
   不修改页面的主体内容（保持人类可审计）

4. 日志记录
   ────────────
   每次回写操作记录到 log.md
   包含：触发来源（Graph 增强）、关系类型、置信度

5. 不触发二次增强
   ────────────────
   回写后的 Wiki 更新不再触发新一轮 Graph 增强
   → 每个 Raw 素材最多触发一轮 "编译→增强→回写" 循环
```

### 5.2 处理分工：Wiki 编译 vs Graph 增强

| 维度 | Wiki 编译层 | Graph 增强层 |
|------|-----------|-------------|
| **处理对象** | 单个素材 → 结构化页面 | 页面 + 页面间关系 → 语义图谱 |
| **关注点** | 这段知识是什么？属于哪里？ | 这段知识和其他知识有什么隐式联系？ |
| **LLM 调用** | 1-2 次（分类+编译） | 1-2 次（关系推理+实体识别） |
| **产出** | 结构化 Wiki 页面 | 语义关系网络 + 结构指标 |
| **人类可见** | ✅ 直接可见可编辑 | ✅ 通过可视化界面查看 |
| **可逆性** | ✅ 可删除页面回滚 | ✅ 可重建 Graph |
| **成本** | 中（LLM 编译） | 中（LLM 推理，可用小模型） |

### 5.3 为什么仍然保留 Wiki 作为第一步？

即使用户质疑"为什么知识不直接落在 Graph 里"，我们仍然让知识先经过 Wiki 编译：

```
原因 1：Graph 的边需要 Wiki 页面作为节点锚点
───────────────────────────────────────────────
  Graph 的节点对应 Wiki 页面
  → 必须先有 Wiki 页面，才能创建 Graph 节点
  → 如果直接写 Graph，节点没有对应页面，查询时无法获取完整内容

原因 2：Graph 增强的上下文来自 Wiki
─────────────────────────────────────
  Graph 增强需要读取页面内容来推理隐式关系
  → 如果知识只存在 Graph 的三元组中，信息量不足以推理
  → 三元组 (RAG, uses, 向量数据库) 丢失了 RAG 为什么用向量数据库的推理

原因 3：人类审计需要 Wiki
─────────────────────────────
  Graph 增强的推理可能出错
  → 需要人类能检查并修正
  → Wiki 页面是人类审计和修正的界面
  → 修正 Wiki 后 Graph 会相应更新
```

## 六、推理深度对比：被动 Graph vs 主动 Graph

### 场景对比

以下用实际查询展示被动 Graph（v1）和主动 Graph（v2）的推理深度差异：

#### 查询："为什么个人场景下 LLM Wiki 比 GraphRAG 更适合？"

```
被动 Graph（v1）能提供的：
───────────────────────────
  Wiki 页面 [[LLM Wiki]] 的内容
  Wiki 页面 [[GraphRAG]] 的内容
  wikilinks 跳转 → [[知识图谱]]、[[RAG]]、[[成本控制]]
  → LLM 读取这些页面内容后自行推理
  → 推理质量完全取决于 LLM 的通用能力 + 页面内容
  → 没有任何"超出页面内容"的结构化线索


主动 Graph（v2）额外能提供的：
───────────────────────────────
  隐式关系：
    (LLM Wiki) ──[superior_in, 置信度 0.9]──→ (个人知识管理)
      推理依据：LLM Wiki 的 Markdown 格式天然适合个人笔记场景

    (GraphRAG) ──[requires, 置信度 0.85]──→ (大规模语料)
      推理依据：GraphRAG 的社区检测需要大量文档才有意义

    (LLM Wiki) ──[trade_off: 低成本/高可读, 置信度 0.9]──→ (GraphRAG)
      推理依据：LLM Wiki 零基础设施成本，GraphRAG 需要 Neo4j

  隐式实体：
    "知识规模" —— 在 15 个页面中出现但无独立页面
      → Graph 标记为隐式实体，关联到 LLM Wiki（小规模）和 GraphRAG（大规模）

  桥接分析：
    "成本控制" 是连接 "技术选型" 和 "个人场景" 群落的桥接节点
      → 提示 LLM 从成本角度分析两者差异

  → LLM 获得结构化的推理线索，而不仅仅是原始文本
  → 推理深度显著提升：从"LLM 自行阅读文本推理"到"有图谱引导的结构化推理"
```

#### 查询："综合分析所有知识库方案的技术演进趋势"

```
被动 Graph（v1）：
───────────────────
  → 读取所有相关 Wiki 页面
  → LLM 自行从文本中提取时序关系
  → 容易遗漏跨页面的技术演进线索
  → 推理质量不稳定


主动 Graph（v2）：
───────────────────
  → Graph 中已存在时序演化边：
    (RAG) ──[evolves_to]──→ (GraphRAG)
    (LLM Wiki) ──[evolves_to]──→ (LLM Wiki + Graph 融合)
    (向量数据库) ──[complemented_by]──→ (知识图谱)

  → Graph 的社区结构直接展示技术群落分布：
    群落 A: 基础检索技术 {RAG, 向量数据库, 语义搜索}
    群落 B: 图谱增强技术 {知识图谱, GraphRAG, Neo4j}
    群落 C: 知识编译技术 {LLM Wiki, Markdown, Schema}

  → 群落间的桥接关系展示技术融合趋势：
    桥接 1: "语义理解" 连接 A 和 C
    桥接 2: "实体关系" 连接 A 和 B

  → LLM 基于这些结构化线索生成演进分析
  → 推理质量稳定且深入
```

### 推理深度量化对比

| 查询类型 | Wiki 单独 | Wiki + 被动 Graph (v1) | Wiki + 主动 Graph (v2) |
|---------|----------|----------------------|----------------------|
| Tier 0 简单事实 | ✅ 充分 | ✅ 充分 | ✅ 充分 |
| Tier 1 单跳关联 | ✅ 基本够 | ✅ 略快 | ✅ 略快 |
| Tier 2 多跳推理 | ⚠️ 依赖 LLM 能力 | ⚠️ 和 Wiki 单独无差异 | ✅ 有隐式关系辅助 |
| Tier 3 全局综合 | ❌ 信息量不足 | ⚠️ 聚类基于 wikilinks | ✅ 有结构化群落+桥接 |

## 七、统一访问接口（Knowledge API）

### 7.1 接口设计

API 层屏蔽 Wiki 和 Graph 的存储细节，上层只需调用统一接口：

```
┌────────────────────────────────────────────────────────────────┐
│                     Knowledge API 接口层                        │
│                                                                │
│  写入接口：                                                    │
│  ─────────                                                     │
│  ingest(source, options?)                                      │
│    → 触发完整流水线：预处理 → Wiki 编译 → Graph 增强 → 回写  │
│    → 返回：{wiki_pages: [...], graph_updates: {...}}           │
│                                                                │
│  查询接口：                                                    │
│  ─────────                                                     │
│  query(question, options?)                                     │
│    → 自动路由到 Wiki / Graph / 混合                           │
│    → 返回：{answer: "...", sources: [...], confidence: 0.x}   │
│                                                                │
│  search(keywords, options?)                                    │
│    → 全文搜索 + 图谱搜索                                      │
│    → 返回：{pages: [...], graph_paths: [...]}                  │
│                                                                │
│  图谱接口：                                                    │
│  ─────────                                                     │
│  graph(options?)          → 完整图谱数据（vis-network 格式）  │
│  graph_neighbors(entity, depth?)  → N 跳邻域                  │
│  graph_path(from, to)     → 两实体间最短路径                   │

│  graph_bridge_entities()  → 桥接实体列表                      │
│                                                                │
│  维护接口：                                                    │
│  ─────────                                                     │
│  lint()                   → 质量检查                           │
│  rebuild_graph()          → 从 Wiki 重建 Graph                │
│  graph_stats()            → 图谱统计信息                      │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.2 查询路由的存储感知

```
query() 内部路由逻辑：

Tier 0: Wiki 直接读取
  → 只调用 Wiki 适配器
  → 读取单个页面，返回内容
  → 延迟：< 50ms

Tier 1: Wiki wikilinks 遍历
  → 调用 Wiki 适配器
  → 读取起始页面 + 沿 wikilinks 跳转 2-3 页
  → 延迟：< 200ms

Tier 2: Wiki + Graph 隐式关系（★ 核心差异）
  → Wiki 适配器：读取相关页面内容
  → Graph 适配器：查询隐式关系边 + 路径
  → 合并：Wiki 内容（精确上下文）+ Graph 关系（推理线索）
  → LLM 基于合并结果生成回答
  → 延迟：500ms - 2s

Tier 3: Graph 社区结构 + Wiki 全文
  → Graph 适配器：获取社区聚类 + 桥接节点 + 演化关系
  → Wiki 适配器：扫描相关群落的所有页面
  → LLM 基于全局结构化线索生成综合分析
  → 延迟：2-10s
```

## 九、回答用户的两个核心问题

### 问题 1：为什么不是统一存储，让知识最终落在 Graph 里？

**回答**：不是"Wiki 或 Graph 二选一"的问题，而是**两者各有不可替代的角色**：

```
Wiki 的不可替代性：
  → 人类可读可编辑（审计 Graph 推理结果的界面）
  → Git 版本控制（知识的历史追溯）
  → Obsidian 兼容（零成本的可视化浏览）
  → Graph 增强的上下文来源（隐式关系推理需要完整文本）

Graph 的不可替代性：
  → 隐式关系存储（Wiki 的 wikilinks 无法表达"依赖"、"桥接"等语义）
  → 结构化推理线索（Tier 2-3 查询的深度保障）
  → 全局模式发现（社区聚类、演化趋势、桥接分析）

正确的设计：
  不是"统一到一个存储"，而是"让两个存储各自发挥优势，通过编译管道协作"。
  Wiki 是人类界面，Graph 是推理引擎，两者通过编译管道同步。
```

### 问题 2：没有 Graph 介入处理能保障推理深度吗？

**回答**：**不能。** 这正是本方案的核心修正。

```
v1 方案（被动 Graph）的问题：
  → Graph 不参与知识处理，只是事后镜像
  → 隐式关系不被发现和记录
  → Tier 2-3 查询的推理深度完全依赖 LLM 的通用能力
  → Graph 形同虚设

v2 方案（主动 Graph 增强）的改进：
  → Graph 增强层在每次 Ingest 时主动介入
  → 通过 LLM 推理发现隐式关系、隐式实体、跨域桥接
  → 这些增强信息存储在 Graph 中，供查询链路使用
  → 高置信度的发现回写到 Wiki，形成知识积累的正循环

推理深度的保障链：
  输入时：Graph 增强发现隐式关系 → 存入 Graph
  查询时：Tier 2-3 查询读取 Graph 隐式关系 → 提供结构化推理线索
  回写时：高置信度关系回写 Wiki → 下次编译时有更丰富的上下文
```

## 十、架构决策总结

| 决策点 | v1 方案（被动 Graph） | v2 方案（主动 Graph 增强） |
|-------|--------------------|--------------------------|
| Graph 角色 | Wiki 的冗余副本 | 编译管道中的主动推理层 |
| 信息流向 | Wiki → Graph 单向 | Wiki ↔ Graph 双向协作 |
| 隐式关系 | 不发现 | LLM 推理发现 + 存储 |
| 隐式实体 | 不识别 | 高频概念识别 + 可选创建页面 |
| 回写 | 无 | 高置信度关系回写 Wiki |
| 推理深度（Tier 2-3） | 等同 Wiki 单独 | 显著超过 Wiki 单独 |
| 额外 LLM 成本 | 0 | 每次 Ingest +1 次 LLM 调用 |
| 复杂度 | 低 | 中（可接受） |

**最终决策**：采用 v2 方案。Graph 作为编译管道中的主动推理层，在每次知识摄入时发现和记录隐式关系，为查询链路的深度推理提供结构化保障。
