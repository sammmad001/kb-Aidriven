# V1.2 查询管道优化方案文档

> **版本**：v1.0
> **日期**：2025-01-25
> **状态**：已实现
> **关联模块**：`kb/app/query/` 全模块（pipeline、understand、resolve、retrieve、gate、generate、trace）

---

## 1. 设计背景

V1.0/V1.1 查询管道存在以下问题：

| 问题 | 影响 | V1.2 解决方案 |
|------|------|--------------|
| 实体名称不匹配导致空检索 | "RAG" 查不到 "检索增强生成" | EntityResolver 三层匹配 |
| 检索结果不足时仍调用 LLM | 浪费 Token + 幻觉答案 | QualityGate 空上下文拦截 |
| 缺乏结构化追踪 | 无法定位性能瓶颈 | PipelineTrace step-level timing |
| 查询分类精度不足 | 短查询被误判为 factual | jieba POS + ambiguity 阈值降低 |
| 对话上下文不连贯 | Follow-up 无法理解指代词 | ConversationContext 集成 |

## 2. 五步管道架构

```
用户问题（可能经 Follow-up 富化）
    │
    ▼
┌─────────────────────────────────────┐
│ Step 1: 查询理解 (QueryUnderstander) │
│  ├─ 规则分类（factual/relational/    │
│  │   reasoning/global）              │
│  ├─ LLM 精确分类（ambiguous 时）     │
│  ├─ jieba POS 实体提取               │
│  └─ Neo4j 实体映射                   │
│  → QueryUnderstanding                │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ Step 1.5: 实体解析 (EntityResolver)  │
│  ├─ Tier 1: 精确匹配（id/name）      │
│  ├─ Tier 2: CONTAINS 模糊匹配        │
│  ├─ Tier 3: difflib 相似度 ≥ 0.6     │
│  └─ 未匹配 → 生成搜索建议            │
│  → ResolutionResult                  │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ Step 2: 图谱检索 (GraphRetriever)    │
│  ├─ Factual: 单节点 (< 10ms)         │
│  ├─ Relational: 1-hop (< 50ms)       │
│  ├─ Reasoning: 多跳 + 隐式关系       │
│  │   + 桥接实体 (< 200ms)            │
│  └─ Global: 簇扫描 (< 500ms)         │
│  → RetrievalResult                   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ Step 2.5: 质量门 (QualityGate)       │
│  ├─ SUFFICIENT: 正常生成             │
│  ├─ PARTIAL: 标记 + 附加建议          │
│  └─ INSUFFICIENT: 直接返回提示        │
│      （跳过 LLM）                    │
│  → GateDecision                      │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ Step 3: 答案生成 (AnswerGenerator)   │
│  ├─ 快速路径: 单节点 factual 无 LLM   │
│  ├─ 标准路径: LLM + 检索上下文       │
│  ├─ 对话历史注入                     │
│  └─ Fallback: LLM 失败 → 原始数据    │
│  → QueryResult                       │
└─────────────────────────────────────┘

全程由 PipelineTrace 追踪：trace_id + 每步 timing + 结构化日志
```

## 3. Step 1: 查询理解（QueryUnderstander）

### 3.1 规则分类（0ms）

基于关键词信号的快速分类：

```python
QUERY_SIGNALS = {
    "global":    ["总结", "全局", "趋势", "所有", "整体", "综合", ...],
    "reasoning": ["为什么", "原因", "导致", "因果", "how", "why", ...],
    "relational": ["关系", "区别", "对比", "比较", "差异", "vs", ...],
    "factual":   ["是什么", "定义", "全称", "what", "who", "when", ...],
}
```

**优先级**：global > reasoning > relational > factual（default）

### 3.2 LLM 精确分类（仅 ambiguous 时）

**Ambiguity 判定**：

```python
def _is_ambiguous(self, rule_result: dict, question: str) -> bool:
    # 阈值从 V1.0 的 30 字符降到 15 字符
    if rule_result.get("depth", 0) == 0 and len(question) > 15:
        return True
    return False
```

**LLM Prompt**：

```
你是一个查询分析器。分析用户查询，输出 JSON：
{"type": "factual|relational|reasoning|global", "entities": [...], "depth": 0-3}

深度定义：
- 0：单节点事实查询
- 1：关联查询（2-3个实体间的关系）
- 2：多跳推理查询
- 3：全局综合分析
```

### 3.3 实体提取（jieba POS）

**V1.2 改进**：用 `jieba.posseg` 替代 V1.0 的 naive regex `r'[\u4e00-\u9fff]+'`。

```python
words = pseg.cut(question)
for w in words:
    if w.flag.startswith(("n", "nr", "ns", "nt", "nz", "v", "vn", "j", "eng")):
        candidate_terms.append(w.word)
```

**保留的词性**：

| jieba flag | 说明 |
|-----------|------|
| `n` | 名词 |
| `nr` | 人名 |
| `ns` | 地名 |
| `nt` | 机构名 |
| `nz` | 其他专名 |
| `v` / `vn` | 动词 / 动名词 |
| `j` | 缩写 |
| `eng` | 英文 |

**过滤**：停用词集合（~50词）+ 长度 < 2 的词。

**Neo4j 映射**：最多取 5 个候选词，在 Neo4j 中 `CONTAINS` 查询匹配已有实体（LIMIT 10）。

## 4. Step 1.5: 实体解析（EntityResolver）

### 4.1 三层匹配架构

```
resolution.resolved（已匹配的实体）
    │
    ├─ Tier 1: 精确匹配
    │   WHERE n.id = $term OR n.name = $term
    │
    ├─ Tier 2: CONTAINS 模糊匹配
    │   WHERE n.id CONTAINS $term OR n.name CONTAINS $term
    │   （与 Tier 1 合并为一条查询）
    │
    └─ Tier 3: difflib 相似度匹配
        SequenceMatcher.ratio() ≥ 0.6
        LIMIT 20 候选节点
```

### 4.2 匹配流程

```python
async def resolve(self, terms: list[str], db: Neo4jDatabase) -> ResolutionResult:
    # Tier 1+2: 精确 + CONTAINS（单次 Cypher 查询）
    resolved = await self._exact_and_contains_match(terms, db)

    # Tier 3: 对未匹配的 terms，difflib 模糊匹配
    unresolved = [t for t in terms if t not in resolved]
    if unresolved:
        fuzzy_matched = await self._fuzzy_match(unresolved, db)
        resolved.extend(fuzzy_matched)

    # 生成搜索建议
    still_unresolved = [t for t in terms if t not in resolved]
    suggestions = self._generate_suggestions(still_unresolved, db)

    return ResolutionResult(resolved=resolved, unresolved=still_unresolved, suggestions=suggestions)
```

### 4.3 difflib 模糊匹配

```python
from difflib import SequenceMatcher

for candidate_node in nodes:  # LIMIT 20
    ratio = SequenceMatcher(None, term.lower(), candidate_name.lower()).ratio()
    if ratio >= 0.6:
        matched.append(candidate_node)
```

**阈值 0.6 的依据**：
- "RAG" vs "检索增强生成" → ratio ≈ 0.0（不匹配，需依赖 aliases 字段）
- "知识图谱" vs "知识图谱技术" → ratio ≈ 0.77（匹配）
- "Transformer" vs "Transformers" → ratio ≈ 0.96（匹配）

### 4.4 搜索建议生成

当实体无法匹配时，生成相似实体的搜索建议，帮助用户调整查询。

## 5. Step 2: 图谱检索（GraphRetriever）

### 5.1 四策略分派

| 查询类型 | 深度 | Cypher 模式 | 典型延迟 |
|---------|------|------------|---------|
| **factual** | 0 | `MATCH (n) WHERE n.id IN $ids` | < 10ms |
| **relational** | 1 | `MATCH (n)-[r]->(m)` 1-hop | < 50ms |
| **reasoning** | 2 | `MATCH path = (a)-[r*1..3]-(b)` 多跳 | < 200ms |
| **global** | 3 | 全簇扫描 + PageRank 排序 | < 500ms |

### 5.2 Factual（Depth 0）

```cypher
MATCH (n) WHERE (n.id IN $ids OR n.name IN $names) AND n.user_id = $_user_id
RETURN n.id, n.name, n.content, n.summary, n.tags
```

### 5.3 Relational（Depth 1）

```cypher
// 节点数据（批量）
MATCH (n) WHERE (n.id IN $ids OR n.name IN $names) AND n.user_id = $_user_id
RETURN n.id, n.name, n.content, n.summary

// 关系路径（批量）
MATCH (n)-[r]->(m) WHERE (n.id IN $ids OR n.name IN $names) AND n.user_id = $_user_id
RETURN n.name, r.type, m.name, m.summary, r.confidence
ORDER BY r.confidence DESC
```

### 5.4 Reasoning（Depth 2）

```cypher
// 多跳路径
MATCH path = (a)-[r*1..3]-(b)
WHERE a.id = $from_id AND b.id = $to_id
  AND a.user_id = $_user_id AND b.user_id = $_user_id
RETURN nodes(path), relationships(path)
LIMIT 5

// 桥接实体
MATCH (a), (b), (bridge)
WHERE a.id = $from_id AND b.id = $to_id
  AND bridge.page_rank > 0.05 AND bridge.user_id = $_user_id
RETURN DISTINCT bridge.name, bridge.summary
LIMIT 5
```

### 5.5 Global（Depth 3）

```cypher
// 全节点（PageRank 排序，LIMIT 50）
MATCH (n) WHERE n:Entity OR n:Concept AND n.user_id = $_user_id
RETURN n.id, n.name, n.summary, n.cluster_id, n.page_rank
ORDER BY n.page_rank DESC LIMIT 50

// 簇信息
MATCH (c:Cluster) WHERE c.user_id = $_user_id
RETURN c.id, c.label, c.summary, c.node_count

// 全局隐式关系
MATCH (n)-[r:IMPLICIT]->(m) WHERE n.user_id = $_user_id
RETURN n.name, r.type, r.confidence, r.evidence, m.name
ORDER BY r.confidence DESC LIMIT 30
```

### 5.6 RetrievalResult 数据结构

```python
RetrievalResult:
    nodes: list[dict]               # 匹配的知识节点
    explicit_paths: list[dict]      # 显式关系路径
    implicit_relations: list[dict]  # 隐式关系（AI 推理）
    bridge_entities: list[dict]     # 桥接实体（多跳枢纽）
    cluster_info: list[dict]        # 知识群落信息
```

## 6. Step 2.5: 质量门（QualityGate）

### 6.1 三级决策

```python
class GateDecision(Enum):
    SUFFICIENT = "sufficient"      # 检索充分，正常生成
    PARTIAL = "partial"            # 部分匹配，标记 + 附加建议
    INSUFFICIENT = "insufficient"  # 检索不足，跳过 LLM
```

### 6.2 评估逻辑

```python
def assess(retrieval, resolution) -> GateDecision:
    has_nodes = len(retrieval.nodes) >= MIN_NODES  # MIN_NODES = 1
    has_paths = len(retrieval.explicit_paths) > 0

    # 无节点且无路径 → 不足以回答
    if not has_nodes and not has_paths:
        return GateDecision.INSUFFICIENT

    # 有未解析的实体 → 部分匹配
    if resolution.unresolved:
        return GateDecision.PARTIAL

    return GateDecision.SUFFICIENT
```

### 6.3 管道中的处理策略

| 决策 | 处理 |
|------|------|
| **SUFFICIENT** | 正常调用 LLM 生成回答 |
| **PARTIAL** | 调用 LLM，但附加 unresolved 实体和搜索建议到 prompt |
| **INSUFFICIENT** | **跳过 LLM**，直接返回 `_build_insufficient_response()` |

### 6.4 INSUFFICIENT 响应

```
抱歉，知识库中未找到与 "{question}" 相关的内容。

建议：
1. 检查实体名称是否正确
2. 尝试更通用的关键词
3. 使用 /search 命令搜索相关内容
```

**设计价值**：避免在无数据时浪费 LLM Token 和产生幻觉答案。

## 7. Step 3: 答案生成（AnswerGenerator）

### 7.1 快速路径（无 LLM）

```python
# 单节点 factual 查询，节点有 content → 直接返回 content
if query_type == FACTUAL and len(retrieval.nodes) == 1:
    node = retrieval.nodes[0]
    if node.get("content"):
        return QueryResult(
            answer=node["content"],      # 直接返回节点内容
            sources=[SourceReference(...)],
            confidence=1.0,
        )
```

**延迟**：0ms（无 LLM 调用）。

### 7.2 标准路径（LLM 生成）

**Prompt 构建**（`_build_prompt`）：

```
【用户问题】
{question}

【对话历史】                    ← 仅当有 context_history
  用户: {previous_question}
  助手: {previous_answer}
  
【检索到的知识节点】
  - {name}: {content/summary}   ← 截断 500 字符
  
【显式关系路径】
  {from} → [{type}] → {to}
  
【隐式关系线索】
  {from} --[{type}, 置信度 85%]--> {to}
  推理依据：{evidence}
  
【桥接实体】
  - {name}: {summary}
  
【知识群落信息】
  - {label}: {summary}
```

**System Prompt 要点**：
- 基于检索信息回答，不编造
- 标注来源 [来源: 节点名]
- 区分确定事实和推理结论
- 隐式关系标注置信度
- 检索不足时坦诚说明
- 结合对话历史理解追问意图

### 7.3 对话历史集成

```python
# context_history 来自 ConversationContext.get_context_history()
# 最多注入最近 3 轮（6 条消息）
if context_history:
    for msg in context_history[-6:]:
        role = msg.get("role")
        content = msg.get("content")[:300]  # 每条截断 300 字符
```

### 7.4 置信度估算

```python
def _estimate_confidence(retrieval) -> float:
    if not retrieval.nodes:
        return 0.1
    score = 0.5  # 基础分
    if retrieval.explicit_paths:
        score += 0.2
    if retrieval.implicit_relations:
        avg_conf = mean(ir.confidence for ir in implicit_relations)
        score += avg_conf * 0.2
    if len(retrieval.nodes) >= 2:
        score += 0.1
    return min(score, 1.0)
```

### 7.5 Fallback（LLM 失败）

LLM 调用异常时，返回原始检索数据的简要列表（confidence=0.3）。

## 8. PipelineTrace — 结构化追踪

### 8.1 数据结构

```python
class PipelineTrace:
    trace_id: str         # uuid.uuid4().hex[:8]
    steps: list[StepRecord]

class StepRecord:
    step: str             # "understand" / "resolve" / "retrieve" / "gate" / "generate"
    input_summary: str    # 输入摘要
    output_summary: str   # 输出摘要
    duration_ms: float    # 耗时
```

### 8.2 方法

```python
trace = PipelineTrace()

trace.step_start("understand", input_summary=question)
# ... 执行查询理解 ...
trace.step_end("understand", output_summary=f"type={understanding.query_type}")

trace.log()
# 输出: [trace:a3f8b2c1] understand in=(什么是RAG？) out=(type=factual, entities=[RAG]) (45ms)
#       [trace:a3f8b2c1] resolve in=([RAG]) out=(resolved=[检索增强生成]) (12ms)
#       ...
```

### 8.3 管道中的使用

```python
# pipeline.py
trace = PipelineTrace()

# Step 1
trace.step_start("understand", ...)
understanding = await self.understander.understand(enriched_question)
trace.step_end("understand", ...)

# Step 1.5
trace.step_start("resolve", ...)
resolution = await self.resolver.resolve(understanding.entities, self.db)
trace.step_end("resolve", ...)

# Step 2
trace.step_start("retrieve", ...)
# 使用 resolution.resolved（回退到 understanding.entities）
entities = resolution.resolved or understanding.entities
retrieval = await self.retriever.retrieve(...)
trace.step_end("retrieve", ...)

# Step 2.5
trace.step_start("gate", ...)
gate = self.gate.assess(retrieval, resolution)
trace.step_end("gate", ...)

# Step 3
trace.step_start("generate", ...)
if gate == INSUFFICIENT:
    result = self._build_insufficient_response(...)
else:
    result = await self.generator.generate(...)
trace.step_end("generate", ...)

trace.log()
```

## 9. 管道完整流程（pipeline.py）

```python
async def query(self, question, context_history=None) -> QueryResult:
    trace = PipelineTrace()

    # Step 1: 查询理解
    understanding = await self.understander.understand(question)

    # Step 1.5: 实体解析
    resolution = await self.resolver.resolve(understanding.entities, self.db)

    # 回退：如果 resolution 未匹配，使用 understanding.entities
    entities = resolution.resolved or understanding.entities
    understanding.entities = entities

    # Step 2: 图谱检索
    retrieval = await self.retriever.retrieve(understanding)

    # Step 2.5: 质量门
    gate = self.gate.assess(retrieval, resolution)

    # Step 3: 答案生成
    if gate == GateDecision.INSUFFICIENT:
        return self._build_insufficient_response(question, resolution)
    
    if gate == GateDecision.PARTIAL:
        # 附加 unresolved 实体到 context
        context_history = context_history or []
        context_history.append({
            "role": "system",
            "content": f"注意：以下实体未在知识库中找到: {resolution.unresolved}"
        })

    result = await self.generator.generate(
        question, retrieval, understanding.query_type, context_history
    )
    
    # 附加 resolution 信息
    if resolution.unresolved:
        result.metadata = result.metadata or {}
        result.metadata["unresolved_entities"] = resolution.unresolved
        result.metadata["suggestions"] = resolution.suggestions

    return result
```

## 10. 性能优化要点

| 优化 | 位置 | 效果 |
|------|------|------|
| 规则分类优先 | understand.py | 80%+ 查询零 LLM 成本 |
| 单节点快速路径 | generate.py | factual 查询 0ms LLM |
| INSUFFICIENT 跳过 LLM | pipeline.py | 空检索不浪费 Token |
| 批量 Cypher 查询 | retrieve.py | 减少数据库往返 |
| jieba POS 替代 regex | understand.py | 实体提取精度提升 |
| difflib 本地匹配 | resolve.py | 无 API 成本 |
