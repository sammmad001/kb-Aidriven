> **版本**：v3.1（Graph-First 统一存储 + DeepSeek 统一架构 + 多用户隔离 + V1.2 查询管道）
> **状态**：生产运行中

## 一、产品概述

### 1.1 产品定位

个人知识库系统是一个基于**知识图谱（Graph）驱动的智能知识管理平台**，核心能力是将碎片化的原始素材自动编译为结构化的知识网络，并提供基于图谱推理的智能问答。

**一句话描述**：你扔进去碎片素材，系统帮你构建可推理的知识网络，你要什么它都能从图谱中推理找到。

### 1.2 核心诉求

| 需求 | 优先级 | 说明 |
|------|-------|------|
| 推理能力 | P0 | 关联分析、逻辑脉络推理、多跳因果推导 |
| 碎片化知识整合 | P0 | 跨文档、跨领域、跨格式的知识自动关联 |
| 成本控制 | P0 | 月成本 < $10，优先使用本地 LLM |
| 知识图谱可视化 | P1 | 完整的知识脉络可视化，支持交互式探索 |
| 可视化检索 | P1 | 自然语言查询 + 图谱导航检索 |

### 1.3 核心设计理念

```
传统知识管理的做法：
  知识 → 存储 → 搜索关键词 → 找到文档 → 人工阅读理解

本系统的做法：
  知识 → Graph 编译（自动提取实体/关系/隐式关联）→ 存入 Neo4j
  查询 → Graph 推理（自动检索路径/隐式关系）→ 直接给出推理结论

  核心差异：知识不是"存完再搜"，而是"存的时候就建立推理网络"
```

### 2.2 核心架构决策

| 决策点 | 结论 | 原因 |
|-------|------|------|
| 数据存储 | **Neo4j 唯一数据库** | 消除双存储同步复杂度，保障数据一致性 |
| Markdown 文件 | **Graph 自动渲染的只读视图** | 保留 Obsidian 兼容性，但不作为权威源 |
| 人类编辑 | **不存在** | 所有知识通过 Ingest 管道输入，避免人为错误 |
| 信息同步 | **单向：Neo4j → Markdown** | 不存在反向同步，无一致性问题 |
| 查询路由 | **不需要路由** | 所有查询统一走 Graph，由 Graph 内部决定检索深度 |
| LLM 提供商 | **DeepSeek 统一架构** | deepseek-v4-pro（编译/推理）+ deepseek-v4-flash（分析/意图），ResilientLLMClient fallback |
| 多用户隔离 | **ContextVar user_id 透明注入** | 复合唯一约束 `(n.id, n.user_id)`，业务代码零修改 |
| 认证体系 | **JWT（access + refresh token）** | python-jose + bcrypt + SQLite UserStore，飞书 open_id 映射 |
| 查询管道 | **V1.2 五步架构** | 理解→实体解析→图谱检索→质量门→答案生成 |
| 外部研究 | **MiroMind 深度研究集成** | /api/ingest/miromind 适配器入口，研究结果自动摄入 |

### 2.3 为什么选择 Graph-First

```
架构演进历程：

  v1: LLM Wiki 为主 + Graph 被动派生
  → 问题：Graph 只是 Wiki 的冗余副本，没有增加推理深度

  v2: Wiki + Graph 双存储协作（Graph 主动推理）
  → 问题：双存储同步复杂，存在一致性风险，回写循环需要五重安全保障

  v3: Graph-First（最终版）
  → Neo4j 存储一切，Markdown 只读渲染
  → 没有同步问题，没有回写循环，没有路由层
  → 简单、一致、推理深度有保障
```

## 三、存储策略

### 3.1 Neo4j 数据模型

#### 节点类型

```
(:Entity {                    实体节点（技术/工具/组织/人物）
  id: "RAG",                  唯一标识（在 user_id 范围内唯一）
  user_id: "user_001",        ★ V2.0 多用户隔离键
  name: "RAG",                显示名称
  type: "technology",         类型标签
  subtype: "framework",       ★ V2.0 子类型
  domain: "AI",               ★ V2.0 知识领域
  content: "完整内容...",      完整的 Markdown 页面内容
  summary: "一句话摘要",       简短描述
  aliases: ["检索增强生成"],   ★ V2.0 别名列表
  definition: "RAG 是...",    ★ V2.0 简明定义
  importance: 0.8,            ★ V2.0 重要性评分
  matched_by: "exact",       ★ V2.0 EntityResolver 匹配方式
  tags: ["检索增强", "LLM"],  标签
  source: "raw/sources/user_001/...",  ★ V2.0 用户分目录
  page_rank: 0.12,            中心性指标
  cluster_id: 3,              所属知识群落
  created_at: datetime,
  updated_at: datetime
})

(:Concept {                   概念节点（理论/方法论/框架）
  id: "知识编译范式",
  name: "知识编译范式",
  type: "methodology",
  content: "完整内容...",
  ...
})

(:Source {                    来源节点（原始素材的元数据）
  id: "2024-01-15-rag-paper",
  name: "RAG Paper",
  raw_path: "raw/sources/2024-01-15-rag-paper.md",
  raw_content: "原始内容...",
  format: "article",
  word_count: 3200,
  ...
})

(:Cluster {                   知识群落节点（Leiden 聚类结果）
  id: 3,
  label: "图谱增强技术",
  summary: "群落概述...",
  node_count: 8
})
```

#### 边类型

```
[:EXPLICIT {                  显式关系（素材分析时直接提取）
  type: "uses",               uses/related_to/contains/part_of/...
  context: "RAG 使用向量数据库进行检索",
  created_at: datetime
}]

[:IMPLICIT {                  隐式关系（Graph 增强 LLM 推理发现）
  type: "depends_on",         depends_on/trade_off/bridges/evolves_to/solves
  confidence: 0.85,           置信度（0.0-1.0）
  evidence: "推理依据",
  discovered_at: datetime
}]

[:BELONGS_TO {                群落归属（节点 → 群落）
  assigned_at: datetime
}]

[:DERIVED_FROM {              知识来源（节点 → 原始素材）
  relevance: 0.9              相关度
}]
```

### 3.2 完整知识 vs 三元组

传统 GraphRAG 只存三元组，丢失大量上下文。本方案在节点中存储完整内容：

```
传统 GraphRAG：
  节点：(RAG, {name: "RAG"})
  边：(RAG)-[:USES]->(向量数据库)
  → 丢失了"为什么用"、"怎么用"的完整解释

本方案：
  节点：(RAG, {
    name: "RAG",
    content: "## Overview\nRAG (Retrieval Augmented Generation) 是一种...(2000字完整内容)",
    summary: "检索增强生成技术",
    tags: ["检索增强", "LLM", "知识管理"]
  })
  显式边：(RAG)-[:EXPLICIT {type: "uses"}]->(向量数据库)
  隐式边：(RAG)-[:IMPLICIT {type: "depends_on", confidence: 0.85}]->(语义理解)
  → 推理时可以读取完整上下文，不丢失信息
```

### 3.3 Markdown 只读渲染

```
目录结构（自动生成，不接受人类编辑）：

wiki/
├── entities/
│   ├── RAG.md                  ← 从 Neo4j Entity 节点 content 渲染
│   ├── 知识图谱.md
│   └── Neo4j.md
├── concepts/
│   ├── 知识编译范式.md
│   └── 图谱增强检索.md
├── comparisons/
│   └── LLM_Wiki_vs_GraphRAG.md
├── sources/
│   └── 2024-01-15-rag-paper.md
├── index.md                    ← 查询所有节点生成
├── log.md                      ← 查询操作日志生成
└── overview.md                 ← 查询群落结构生成

渲染触发：
  每次 Ingest 完成后 → 自动渲染受影响的 Markdown 文件
  每次 lint 完成后   → 完整重新渲染

渲染逻辑：
  Neo4j 节点 content 字段 → .md 文件主体
  Neo4j 关系查询结果      → ## Relationships 区域 + [[wikilinks]]
  Neo4j 群落信息          → overview.md
```

### 4.1 双链路总览

```

║                     双链路整合视图                                   ║
╠══════════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  ┌──── 链路 1：输入（Ingest）────────────────────────────────────┐ ║
║  │                                                              │ ║
║  │  Raw 素材 → 预处理 → 分析分类 → Graph 处理 → Neo4j 写入     │ ║
║  │                                              │               │ ║
║  │                                              └→ 渲染 Markdown│ ║
║  │                                                              │ ║
║  │  LLM 调用：3 次（分类 1 + 编译 1 + 隐式推理 1）              │ ║
║  │  延迟：5-15 秒                                               │ ║
║  └──────────────────────────────────────────────────────────────┘ ║
║                              ↓                                    ║
║                     Neo4j 知识图谱                                 ║
║                              ↓                                    ║
║  ┌──── 链路 2：输出（Query）────────────────────────────────────┐ ║
║  │                                                              │ ║
║  │  用户查询 → 查询理解 → Graph 检索 → LLM 生成回答 → 返回用户 │ ║
║  │                                                              │ ║
║  │  LLM 调用：1-2 次（查询理解 0-1 + 回答生成 1）               │ ║
║  │  延迟：0.5-5 秒                                              │ ║
║  └──────────────────────────────────────────────────────────────┘ ║
║                                                                    ║
╚══════════════════════════════════════════════════════════════════════╝
```

### 4.2 输入链路概要

> 详细设计见 `输入流方案文档.md`

```
四步处理流程：

  Step 1: 预处理
    格式统一 → Markdown → 存入 raw/sources/（不可变）

  Step 2: 分析分类（LLM 调用 1）
    素材类型（5类分型）+ 实体提取 + 矛盾检测

  Step 3: Graph 处理（LLM 调用 1-2）
    创建/更新 Neo4j 节点 + 显式关系 + 隐式关系推理 + 图结构更新

  Step 4: 渲染 Markdown（无 LLM）
    Neo4j 节点 → .md 文件（自动，只读）
```

### 4.3 输出链路概要

> 详细设计见 `输出流方案文档.md`

```
五步处理流程（V1.2 查询管道）：

  Step 1: 查询理解（jieba POS + 规则/LLM）
    jieba POS 标注实体提取 + 混合分类（factual/relational/reasoning/global）
    + 对话上下文 follow-up 富化

  Step 1.5: 实体解析 EntityResolver（★ V1.2 新增）
    三层渐进式匹配（精确 → CONTAINS → difflib 模糊，阈值 0.6）

  Step 2: Graph 检索（4 策略 dispatch）
    factual 快速路径 / relational 一跳 / reasoning 多跳+隐式 / global 聚类

  Step 2.5: 质量门 QualityGate（★ V1.2 新增）
    SUFFICIENT / PARTIAL / INSUFFICIENT 三级判定，空上下文拦截

  Step 3: LLM 生成回答（deepseek-v4-pro）
    基于 Graph 检索结果 + QualityGate 判定 → 生成回答
    + PipelineTrace 结构化追踪
```

### 7.2 技术栈
