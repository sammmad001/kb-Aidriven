# 验证&发布自动化方案

> 基于个人知识库系统 CI/CD 自动化体系的全面复盘与标准化总结。
> 
> 本文档形成一套完整、可复制、可复用的标准化开发发布流程体系，可作为其他项目的实施参考。

## 一、流程梳理：标准化开发-验证-发布全链路

### 1.1 总览

系统实现了 **5 层自动化**验证-发布链路，开发者只需执行 `git commit` + `git push`，其余验证、测试、部署全部自动完成。

```
开发者修改代码
    │
    ▼
┌─────────────────────────────────────────────────┐
│ Layer 1: git commit → pre-commit hook           │
│   ├─ Python 编译检查（仅暂存的 .py 文件）         │
│   └─ pytest（仅 kb/ 目录有变更时）                │
└─────────────────────────────────────────────────┘
    │ 通过 → 允许 commit
    ▼
┌─────────────────────────────────────────────────┐
│ Layer 2: git push → pre-push hook               │
│   └─ pre-deploy-check.sh 完整 6 步验证           │
│       ├─ Python 编译检查（全量 39 个文件）         │
│       ├─ 应用导入检查（from app.main import app） │
│       ├─ pytest 全量测试（49 个用例）              │
│       ├─ 前端 TypeScript 类型检查                 │
│       ├─ 前端 Vite 构建验证                       │
│       └─ Git 工作区状态检查                       │
└─────────────────────────────────────────────────┘
    │ 通过 → 允许 push
    ▼
┌─────────────────────────────────────────────────┐
│ Layer 3: GitHub Actions CI（云端并行）            │
│   ├─ backend-test: Python 编译 + pytest          │
│   └─ frontend-build: TypeScript 检查 + Vite 构建 │
└─────────────────────────────────────────────────┘
    │ 两个 job 全部通过
    ▼
┌─────────────────────────────────────────────────┐
│ Layer 4: GitHub Actions CD（自动部署）            │
│   ├─ 上传运维脚本（backup.sh 等）                 │
│   ├─ ECS 备份（Neo4j dump + 素材 + 配置）        │
│   ├─ rsync 上传后端代码                           │
│   ├─ pip install 依赖                            │
│   ├─ systemctl restart 服务                      │
│   └─ /health 健康检查（5 次重试，间隔 3 秒）       │
└─────────────────────────────────────────────────┘
    │ 健康检查通过
    ▼
┌─────────────────────────────────────────────────┐
│ Layer 5: 生产环境                                │
│   ├─ 服务正常运行（Neo4j/LLM/Feishu 组件正常）   │
│   └─ GitHub Step Summary 输出部署报告             │
└─────────────────────────────────────────────────┘
```

### 1.2 各层详细说明

| 层级 | 触发时机 | 执行环境 | 耗时 | 阻断能力 |
|------|----------|----------|------|----------|
| **Layer 1** | `git commit` | 本地 | <1s | 阻止 commit |
| **Layer 2** | `git push` | 本地 | ~30s | 阻止 push |
| **Layer 3** | push 到 GitHub | GitHub Actions (Ubuntu) | ~25s | CI 标红，CD 不触发 |
| **Layer 4** | CI 全部通过 | GitHub Actions → SSH to ECS | ~60s | 部署失败告警 |
| **Layer 5** | CD 完成 | ECS 生产环境 | - | 健康检查失败需手动介入 |

### 1.3 开发者视角操作链路

**日常开发（最常见场景）**：
```bash
# 1. 修改代码
vim kb/app/xxx.py

# 2. 提交（自动触发 Layer 1 检查）
git add -A
git commit -m "feat: 新增 xxx 功能"
# → pre-commit: Python 编译检查 ✓, pytest ✓

# 3. 推送（自动触发 Layer 2~5）
git push origin main
# → pre-push: 6 步验证 ✓
# → GitHub CI: backend-test ✓, frontend-build ✓
# → GitHub CD: 备份→上传→重启→健康检查 ✓
# → 生产环境已更新
```

**全流程耗时**：从 `git push` 到生产环境更新完成，约 **2 分钟**（本地验证 30s + CI 25s + CD 60s）。

|----------|----------|----------|----------|----------|
| **git commit** | 开发者执行 `git commit` | pre-commit hook: ① 暂存 .py 文件编译检查 ② pytest（kb/ 有变更时） | `✅ pre-commit 检查全部通过` → 允许 commit | `❌ Python 编译检查失败` 或 `❌ pytest 失败` → 阻止 commit，输出错误详情 |

## 四、问题复盘：实施过程中的技术挑战

### 4.1 CI 环境 Python 模块导入失败

| 项目 | 内容 |
|------|------|
| **现象** | GitHub Actions CI 的 `Run pytest` 步骤失败，exit code 1 |
| **根因** | CI 中使用 `cd kb && pytest`，Python 无法找到 `app` 模块。本地不受影响是因为本地有 `.env` 和虚拟环境 |
| **解决方案** | 使用 `working-directory: kb` + 显式设置 `PYTHONPATH: ${{ github.workspace }}/kb` |
| **关键发现** | GitHub Actions 的 `run: cd xxx && command` 与 `working-directory: xxx` + `run: command` 行为不同，后者更可靠 |

### 4.2 ECS 上 backup.sh 不存在

| 项目 | 内容 |
|------|------|
| **现象** | CD deploy job 在 "Backup on ECS" 步骤失败，exit code 127 (command not found) |
| **根因** | CD 流程先执行 `./backup.sh`，但此时脚本尚未上传到 ECS |
| **解决方案** | 增加 "Upload scripts first" 步骤，在任何其他操作前先将 `backup.sh`、`deploy.sh`、`rollback.sh`、`pre-deploy-check.sh` 上传到 ECS 并赋予执行权限 |
| **关键发现** | CD 流水线的步骤顺序至关重要——运维脚本必须先于业务代码上传 |

### 4.3 ECS 部署路径不匹配

| 项目 | 内容 |
|------|------|
| **现象** | CD 上传后端代码后，ECS 服务启动失败 |
| **根因** | CI 将 `kb/` 上传到 `/opt/knowledge-base/kb/`，但 ECS 上的应用代码直接在 `/opt/knowledge-base/app/`。手动 `deploy.sh` 的逻辑是将 `kb/` 的内容直接同步到 `/opt/knowledge-base/` |
| **解决方案** | 修正 rsync 目标路径为 `/opt/knowledge-base/`（与 deploy.sh 保持一致） |
| **关键发现** | 自动化 CD 的路径逻辑必须与手动部署脚本完全一致，建议统一抽取为配置变量 |

### 4.4 pip install 未使用虚拟环境

| 项目 | 内容 |
|------|------|
| **现象** | CD 的 "Install dependencies" 步骤可能安装到系统 Python 而非项目虚拟环境 |
| **根因** | CI 中直接执行 `pip install`，未激活 ECS 上的 `venv/` 虚拟环境 |
| **解决方案** | 修改为 `source venv/bin/activate && pip install -r requirements.txt -q` |
| **关键发现** | ECS 上的 Python 依赖安装在 venv 中，CI 部署必须与 ECS 运行环境保持一致 |

### 4.5 前端构建产物上传目录不存在

| 项目 | 内容 |
|------|------|
| **现象** | CD 的 "Upload frontend build" 步骤失败，rsync exit code 11/23 |
| **根因** | ECS 上 `/opt/knowledge-base/kb-web/` 目录不存在（ECS 仅运行后端 API） |
| **解决方案** | 由于 ECS 当前仅运行后端 API，前端由其他途径部署，因此移除 CD 中的前端上传步骤 |
| **关键发现** | CD 流水线应只部署实际运行的组件，避免上传不需要的产物 |

## 五、标准化工具包：可复用工具模板

### 5.1 hooks/pre-commit — 提交前快速检查

**文件路径**: `hooks/pre-commit`

```bash
#!/bin/bash
# pre-commit hook: 快速检查（每次 git commit 自动执行）
# 仅检查暂存的 Python 文件编译 + pytest
set -e

echo "🔍 [pre-commit] 自动验证中..."

# 1. 检查暂存的 .py 文件是否有语法错误
STAGED_PY=$(git diff --cached --name-only --diff-filter=ACM | grep '\.py$' || true)
if [ -n "$STAGED_PY" ]; then
    echo "  检查 Python 编译..."
    ERRORS=0
    for f in $STAGED_PY; do
        if [ -f "$f" ] && ! python3 -m py_compile "$f" 2>/dev/null; then
            echo "  ✗ 编译失败: $f"
            ERRORS=$((ERRORS + 1))
        fi
    done
    if [ $ERRORS -gt 0 ]; then
        echo "❌ Python 编译检查失败，commit 已阻止"
        exit 1
    fi
    echo "  ✓ Python 编译通过"
fi

# 2. 运行 pytest（仅后端有变更时）
STAGED_KB=$(git diff --cached --name-only | grep '^kb/' || true)
if [ -n "$STAGED_KB" ]; then
    echo "  运行 pytest..."
    cd kb && python3 -m pytest tests/ -x -q --tb=short 2>&1
    EXIT_CODE=$?
    cd ..
    if [ $EXIT_CODE -ne 0 ]; then
        echo "❌ pytest 失败，commit 已阻止"
        exit 1
    fi
    echo "  ✓ pytest 通过"
fi

echo "✅ pre-commit 检查全部通过"
```

**适配指引**：
- **Go 项目**: 将 `python3 -m py_compile` 替换为 `go vet`
- **TypeScript 项目**: 将 `python3 -m py_compile` 替换为 `npx tsc --noEmit`
- **Java 项目**: 将 `python3 -m py_compile` 替换为 `javac`
- **触发条件**: 修改 `grep '\.py$'` 和 `grep '^kb/'` 匹配目标目录

### 5.2 hooks/pre-push — 推送前完整验证

**文件路径**: `hooks/pre-push`

```bash
#!/bin/bash
# pre-push hook: 完整验证（每次 git push 自动执行）
# 调用 pre-deploy-check.sh 进行全量验证
set -e

echo "🔍 [pre-push] 完整部署前验证..."

REPO_ROOT="$(git rev-parse --show-toplevel)"

if [ -x "$REPO_ROOT/pre-deploy-check.sh" ]; then
    "$REPO_ROOT/pre-deploy-check.sh"
    EXIT_CODE=$?
    if [ $EXIT_CODE -ne 0 ]; then
        echo "❌ 部署前验证失败，push 已阻止"
        echo "   修复后重新 commit 再 push"
        exit 1
    fi
else
    echo "⚠️  pre-deploy-check.sh 不存在或不可执行，跳过完整验证"
fi

echo "✅ pre-push 验证通过，允许推送"
```

**适配指引**：
- 修改验证脚本路径（`pre-deploy-check.sh`）为项目自定义验证脚本
- 可在此处增加额外检查（如 lint、安全扫描等）

### 5.3 setup-hooks.sh — Hook 安装器

**文件路径**: `setup-hooks.sh`

```bash
#!/bin/bash
# 安装 Git Hooks（开发环境初始化时运行一次）
# 用法: ./setup-hooks.sh

REPO_ROOT="$(git rev-parse --show-toplevel)"
HOOKS_DIR="$REPO_ROOT/hooks"
GIT_HOOKS_DIR="$REPO_ROOT/.git/hooks"

echo "安装 Git Hooks..."

for hook in pre-commit pre-push; do
    if [ -f "$HOOKS_DIR/$hook" ]; then
        ln -sf "$HOOKS_DIR/$hook" "$GIT_HOOKS_DIR/$hook"
        echo "  ✓ $hook → .git/hooks/$hook"
    fi
done

echo "✅ Git Hooks 安装完成"
```

**设计要点**：
- hooks 脚本存放在仓库内的 `hooks/` 目录（可被 git 追踪和版本控制）
- 通过符号链接安装到 `.git/hooks/`（该目录不被 git 追踪）
- 新项目 clone 后需运行一次 `./setup-hooks.sh`

### 5.4 .github/workflows/ci.yml — CI/CD 流水线

**文件路径**: `.github/workflows/ci.yml`

```yaml
name: CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  backend-test:
    name: Backend Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: |
          cd kb
          pip install -r requirements.txt
      - name: Compile check
        working-directory: kb
        run: find app -name "*.py" -exec python3 -m py_compile {} \;
      - name: Run pytest
        working-directory: kb
        env:
          PYTHONPATH: ${{ github.workspace }}/kb
        run: python3 -m pytest tests/ -v --tb=long

  frontend-build:
    name: Frontend Build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      - name: Install dependencies
        run: cd kb-web && npm ci
      - name: TypeScript check
        run: cd kb-web && npx tsc -b --noEmit
      - name: Build
        run: cd kb-web && npm run build

  deploy:
    name: Deploy to ECS
    needs: [backend-test, frontend-build]
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Setup SSH
        run: |
          mkdir -p ~/.ssh
          echo "${{ secrets.ECS_SSH_KEY }}" > ~/.ssh/id_ed25519
          chmod 600 ~/.ssh/id_ed25519
          ssh-keyscan -H ${{ secrets.ECS_HOST }} >> ~/.ssh/known_hosts 2>/dev/null || true

      - name: Upload scripts first
        run: |
          for script in backup.sh deploy.sh rollback.sh pre-deploy-check.sh; do
            if [ -f "$script" ]; then
              scp -o ConnectTimeout=10 "$script" ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }}:/opt/knowledge-base/
            fi
          done
          ssh -o ConnectTimeout=10 ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }} \
            "chmod +x /opt/knowledge-base/*.sh 2>/dev/null || true"

      - name: Backup on ECS
        run: |
          ssh -o ConnectTimeout=10 ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }} \
            "cd /opt/knowledge-base && ./backup.sh"

      - name: Upload backend code
        run: |
          rsync -avz --delete \
            --exclude='__pycache__' \
            --exclude='*.pyc' \
            --exclude='.env' \
            --exclude='venv/' \
            --exclude='raw/sources/*' \
            --exclude='wiki/*' \
            --exclude='tests/' \
            -e "ssh -o ConnectTimeout=10" \
            kb/ ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }}:/opt/knowledge-base/

      - name: Install dependencies
        run: |
          ssh -o ConnectTimeout=10 ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }} \
            "cd /opt/knowledge-base && source venv/bin/activate && pip install -r requirements.txt -q"

      - name: Restart service
        run: |
          ssh -o ConnectTimeout=10 ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }} \
            "systemctl restart knowledge-base"

      - name: Health check
        run: |
          echo "等待服务启动..."
          sleep 5
          for i in {1..5}; do
            if curl -sf http://${{ secrets.ECS_HOST }}:8080/health | grep -q ok; then
              echo "✅ 健康检查通过"
              exit 0
            fi
            echo "  尝试 $i/5..."
            sleep 3
          done
          echo "❌ 健康检查失败"
          ssh -o ConnectTimeout=10 ${{ secrets.ECS_USER }}@${{ secrets.ECS_HOST }} \
            "journalctl -u knowledge-base --no-pager -n 20"
          exit 1

      - name: Deployment summary
        if: success()
        run: |
          echo "## 🚀 部署成功" >> $GITHUB_STEP_SUMMARY
          echo "- **Commit**: ${{ github.sha }}" >> $GITHUB_STEP_SUMMARY
          echo "- **Branch**: main" >> $GITHUB_STEP_SUMMARY
          echo "- **ECS**: ${{ secrets.ECS_HOST }}" >> $GITHUB_STEP_SUMMARY
          echo "- **Time**: $(date '+%Y-%m-%d %H:%M:%S UTC')" >> $GITHUB_STEP_SUMMARY
```

**适配指引**：
- **不同服务器**: 修改 GitHub Secrets（`ECS_SSH_KEY` / `ECS_HOST` / `ECS_USER`）
- **不同部署目录**: 修改 rsync 目标路径和 `systemctl` 服务名
- **不同健康检查**: 修改 `curl` 的 URL 和端口
- **不同语言**: 修改 `setup-python` / `setup-node` 为对应的 setup action

## 六、可复用性设计

### 6.1 适配不同技术栈

| 适配维度 | 当前实现 | 适配方式 |
|----------|----------|----------|
| **Go 项目** | Python py_compile | pre-commit: `go vet ./...`; CI: `actions/setup-go` + `go test ./...` |
| **Java 项目** | Python py_compile | pre-commit: `mvn compile`; CI: `actions/setup-java` + `mvn test` |
| **纯前端项目** | Python + TypeScript | pre-commit: `npx tsc --noEmit`; CI: 仅保留 frontend-build job |
| **Monorepo** | 单一 kb/ 目录 | pre-commit: 按 package 分别检查; CI: 增加 matrix 策略 |

### 6.2 适配不同部署环境

| 部署环境 | 修改内容 |
|----------|----------|
| **ECS（当前）** | 使用 rsync + SSH + systemctl（已实现） |
| **Docker** | 将 CD 中的 rsync 替换为 `docker build` + `docker push`，systemctl 替换为 `docker-compose pull` + `docker-compose up -d` |
| **Kubernetes** | 将 CD 中的 rsync 替换为 `kubectl set image` 或 Helm upgrade |
| **Vercel/Netlify** | 前端项目可直接使用对应的 deploy action（如 `vercel-deploy` skill） |
| **AWS Lambda** | 将 CD 替换为 `aws lambda update-function-code` |

### 6.3 适配不同测试框架

| 测试框架 | pre-commit 命令 | CI 命令 |
|----------|-----------------|---------|
| **pytest** (当前) | `python3 -m pytest tests/ -x -q --tb=short` | `python3 -m pytest tests/ -v --tb=long` |
| **Jest** | `npx jest --passWithNoTests` | `npx jest --coverage` |
| **Go test** | `go test ./... -count=1` | `go test ./... -v -cover` |
| **Maven** | `mvn test -q` | `mvn test` |

### 6.4 多环境部署策略

```yaml
# 在 CI 中增加环境判断
deploy:
  needs: [backend-test, frontend-build]
  runs-on: ubuntu-latest
  steps:
    - name: Determine environment
      run: |
        if [ "${{ github.ref }}" = "refs/heads/main" ]; then
          echo "ENV=production" >> $GITHUB_ENV
        elif [ "${{ github.ref }}" = "refs/heads/staging" ]; then
          echo "ENV=staging" >> $GITHUB_ENV
        fi
```

### 6.5 快速迁移检查清单

将本方案迁移到新项目时，按以下清单逐步适配：

- [ ] 修改 `hooks/pre-commit` 中的语言检查命令和目录匹配规则
- [ ] 修改 `hooks/pre-push` 中的验证脚本路径
- [ ] 修改 `pre-deploy-check.sh` 中的 `KB_DIR` / `WEB_DIR` 和验证步骤
- [ ] 修改 `.github/workflows/ci.yml` 中的 setup action、测试命令、部署配置
- [ ] 配置 GitHub Secrets（SSH 密钥、服务器地址、用户名）
- [ ] 修改 `deploy.sh` / `rollback.sh` 中的 ECS 配置
- [ ] 运行 `./setup-hooks.sh` 安装 hooks
- [ ] 执行一次 `git commit` + `git push` 验证全链路
