# 查询路由层深度优化方案

> **[已废弃]** 本文档描述的四级路由方案已被 V1.0 Graph-First 单链路查询替代。
> 当前实现请参考 `输出流方案文档.md` 和 `个人知识库系统-V1.0-产品存档文档.md`。

## Context

当前 `个人知识库方案计划.md` 第 136-139 行的路由层设计存在过度简化问题：

```
│  知识量 < 10万 token → Wiki 直接查询     │
│  知识量 > 10万 token → 图谱检索          │
│  复杂多跳查询 → GraphRAG                 │
```

**问题**：以知识总量（token 数）作为唯一路由标准，忽略了查询本身的语义特征。对于个人知识库场景，单次输入很少达到 10 万 token，这意味着所有查询都会走 Wiki 通道，GraphRAG 层形同虚设。路由决策应该基于**查询本身的复杂度**而非**知识库的总体量**。

---

## 一、当前路由策略的问题分析

### 问题 1：Token 数阈值不适合个人场景

| 场景 | 知识库总 token | 单次查询 token | 当前路由结果 |
|------|---------------|---------------|-------------|
| 刚起步 | 2 万 | 50 | Wiki（正确） |
| 积累中 | 8 万 | 200 | Wiki（可能不够——跨领域关联查询需要图谱） |
| 成熟期 | 15 万 | 100 | GraphRAG（过度——简单事实查询不需要图谱） |

**核心矛盾**：个人知识库的知识量通常在 1-20 万 token 范围内，且单次查询的复杂度差异巨大。一个简单的"X 的定义是什么"和一个复杂的"X 和 Y 的因果链如何推导到 Z"应该走不同路径，但当前路由策略无法区分。

### 问题 2：忽略了查询语义复杂度

学术研究（arxiv 2604.14222v1, Adaptive Query Routing）已经证实：**没有任何单一检索范式在所有查询类型上占优**。查询的语义复杂度才是路由决策的核心因素，而非知识库规模。

### 问题 3：三元路由过于僵化

当前的三条路由规则是互斥的硬编码逻辑，无法适应查询的多样化特征。

---

## 二、改进方案：基于查询复杂度的智能路由

### 核心设计原则

> **路由决策不看"知识库有多大"，而看"这个问题有多复杂"。**

### 路由架构重新设计

```
                    ┌──────────────────────────────────────────┐
                    │           用户查询输入                     │
                    └──────────────────┬───────────────────────┘
                                       │
                    ┌──────────────────▼───────────────────────┐
                    │        查询复杂度分析器（Query Analyzer）   │
                    │                                           │
                    │  维度 1: 实体数量 (1-2 vs 3+)              │
                    │  维度 2: 关系类型 (事实 vs 因果/对比)       │
                    │  维度 3: 推理深度 (单跳 vs 多跳)            │
                    │  维度 4: 跨域需求 (单域 vs 跨域)            │
                    │  维度 5: 实时性要求 (秒级 vs 可等待)        │
                    │                                           │
                    │  输出: 复杂度评分 + 推荐路由                │
                    └──────────────────┬───────────────────────┘
                                       │
                    ┌──────────────────▼───────────────────────┐
                    │           智能路由决策层                    │
                    │                                           │
                    │  Tier 0 (简单事实): Wiki 直接读取           │
                    │  Tier 1 (单跳关联): Wiki + wikilinks 遍历  │
                    │  Tier 2 (多跳推理): Wiki 编译 + 图谱辅助    │
                    │  Tier 3 (全局综合): GraphRAG + Wiki 全文   │
                    └──────┬──────────┬──────────┬──────────────┘
                           │          │          │
              ┌────────────▼──┐  ┌───▼───────┐  │
              │  Wiki 直接查询 │  │ Wiki+图谱  │  │
              │  (Tier 0-1)   │  │ (Tier 2)   │  │
              └───────────────┘  └───────────┘  │
                                               │
                                  ┌────────────▼────┐
                                  │  GraphRAG 检索   │
                                  │  (Tier 3)        │
                                  └─────────────────┘
```

## 三、查询复杂度分析器设计

### 四级复杂度分类体系

基于学术研究（arxiv 2604.14222v1 四层基准 + Query-Adaptive RAG 的复杂度信号），设计如下分类：

#### Tier 0：简单事实查询

**特征信号**：
- 实体数量：1-2 个
- 关系类型：直接属性（"是什么"、"在哪里"、"什么时候"）
- 无因果/对比/推理关键词
- 单域查询

**示例**：
- "RAG 的全称是什么？"
- "Neo4j 是什么类型的数据库？"

**路由**：Wiki 页面直接读取，无需遍历

#### Tier 1：单跳关联查询

**特征信号**：
- 实体数量：2-3 个
- 关系类型：直接关联（"X 和 Y 的关系"、"X 如何影响 Y"）
- 含简单关联词（"关联"、"关系"、"影响"）
- 单域或浅跨域

**示例**：
- "RAG 和知识图谱有什么关系？"
- "LLM Wiki 的 Schema 层如何控制编译行为？"

**路由**：Wiki wikilinks 遍历（Agent 读取相关页面后跳转）

#### Tier 2：多跳推理查询

**特征信号**：
- 实体数量：3+ 个
- 关系类型：因果/对比/推理链
- 含复杂连接词（"因为"、"导致"、"相比"、"推导"、"为什么会"）
- 需要跨文档证据整合
- 跨域查询

**示例**：
- "LLM Wiki 的知识编译范式与 GraphRAG 的图谱检索在推理精度上的差异原因是什么？"
- "为什么个人场景下 LLM Wiki 比 GraphRAG 更经济但推理更准确？"

**路由**：Wiki 编译 + 图谱辅助（先查 Wiki 页面获取上下文，再用图谱补充关系路径）

#### Tier 3：全局综合查询

**特征信号**：
- 实体数量：5+ 个或无明确实体
- 关系类型：全局模式/趋势/综合分析
- 含全局性关键词（"总结"、"全局"、"趋势"、"所有"、"整体架构"）
- 需要大规模跨文档、跨领域综合
- 高推理深度

**示例**：
- "综合分析我所有知识库中关于 AI 知识管理的内容，总结技术演进趋势"
- "对比我记录的所有知识库方案，给出最佳选型建议"

**路由**：GraphRAG 社区摘要 + Wiki 全文扫描

## 四、路由决策判断标准与权重

### 复杂度评分模型

每个查询在五个维度上评分（0-2 分），加权求和得出总分：

```python
def compute_complexity_score(query: str) -> float:
    """
    查询复杂度评分模型
    总分范围: 0.0 - 2.0
    """
    
    scores = {
        "entity_count":    score_entity_count(query),      # 权重 0.20
        "relation_type":   score_relation_type(query),      # 权重 0.25
        "reasoning_depth": score_reasoning_depth(query),    # 权重 0.30
        "cross_domain":    score_cross_domain(query),       # 权重 0.15
        "realtime_need":   score_realtime_need(query),      # 权重 0.10
    }
    
    weights = {
        "entity_count":    0.20,
        "relation_type":   0.25,
        "reasoning_depth": 0.30,
        "cross_domain":    0.15,
        "realtime_need":   0.10,
    }
    
    total = sum(scores[k] * weights[k] for k in scores)
    return total  # 0.0 ~ 2.0
```

### 各维度评分标准

#### 维度 1：实体数量（权重 0.20）

| 评分 | 条件 | 示例 |
|------|------|------|
| 0 分 | 1 个实体，直接查询 | "什么是 RAG？" |
| 1 分 | 2-3 个实体，需关联 | "RAG 和知识图谱的区别" |
| 2 分 | 4+ 个实体，需多跳串联 | "A 到 B 到 C 到 D 的关系链" |

#### 维度 2：关系类型（权重 0.25）

| 评分 | 条件 | 关键词信号 |
|------|------|-----------|
| 0 分 | 事实性查询（是什么/在哪/何时） | "是什么"、"定义"、"全称" |
| 1 分 | 关联性查询（比较/影响/关系） | "区别"、"关系"、"对比"、"影响" |
| 2 分 | 推理性查询（因果/推导/为什么） | "为什么"、"导致"、"因为"、"推导"、"因果" |

#### 维度 3：推理深度（权重 0.30）—— 最重要维度

| 评分 | 条件 | 判断逻辑 |
|------|------|---------|
| 0 分 | 单跳可答，答案在一个页面 | 直接匹配 Wiki 页面标题 |
| 1 分 | 需 2-3 步关联 | 需要 wikilinks 跳转 2-3 次 |
| 2 分 | 需 4+ 步推理或全局综合 | 需要跨多个知识群落推理 |

#### 维度 4：跨域需求（权重 0.15）

| 评分 | 条件 | 判断逻辑 |
|------|------|---------|
| 0 分 | 单一主题域 | 所有实体属于同一 Wiki 分类 |
| 1 分 | 2 个域交叉 | 实体分属不同 Wiki 分类 |
| 2 分 | 3+ 域综合 | 需要跨多个分类综合分析 |

#### 维度 5：实时性要求（权重 0.10）

| 评分 | 条件 | 判断逻辑 |
|------|------|---------|
| 0 分 | 需要秒级响应 | 简单事实，优先速度 |
| 1 分 | 秒级可接受 | 一般查询，平衡速度和质量 |
| 2 分 | 可等待，追求深度 | 复杂分析，优先质量 |

### 路由阈值

```python
def route_query(complexity_score: float) -> str:
    if complexity_score < 0.6:
        return "Tier 0: Wiki 直接读取"
    elif complexity_score < 1.0:
        return "Tier 1: Wiki wikilinks 遍历"
    elif complexity_score < 1.5:
        return "Tier 2: Wiki + 图谱辅助"
    else:
        return "Tier 3: GraphRAG + Wiki 全文"
```

## 五、复杂度分析器的实现方式

### 方案 A：关键词 + 规则匹配（推荐起步）

```python
COMPLEXITY_SIGNALS = {
    "factual": ["是什么", "定义", "全称", "什么叫", "什么是", "who", "what is"],
    "relational": ["关系", "区别", "对比", "比较", "差异", "影响", "联系", "vs"],
    "causal": ["为什么", "原因", "导致", "因果", "因为", "所以", "推导", "为什么"],
    "comparative": ["相比", "对比", "优劣", "优缺点", "更好", "更差"],
    "global": ["总结", "全局", "趋势", "所有", "整体", "综合", "全景"],
}

def classify_query(query: str) -> str:
    """基于关键词和规则的快速分类"""
    signal_counts = {}
    for category, keywords in COMPLEXITY_SIGNALS.items():
        signal_counts[category] = sum(1 for kw in keywords if kw in query.lower())
    
    if signal_counts["global"] > 0:
        return "Tier 3"
    if signal_counts["causal"] > 0 or signal_counts["comparative"] > 0:
        return "Tier 2"
    if signal_counts["relational"] > 0:
        return "Tier 1"
    return "Tier 0"
```

**优点**：零成本、零延迟、确定性高
**缺点**：关键词覆盖有限，无法理解深层语义

### 方案 B：LLM 作为分类器（推荐优化）

```
System Prompt:
你是一个查询复杂度分类器。根据用户查询，判断其复杂度等级：

Tier 0：简单事实查询（1-2个实体，直接属性查询）
Tier 1：单跳关联查询（2-3个实体，需要关联两个概念）
Tier 2：多跳推理查询（3+实体，因果/对比/推理链）
Tier 3：全局综合查询（多域综合分析，模式发现）

输出格式：{"tier": 0-3, "entities": [...], "reasoning_type": "factual|relational|causal|global"}

用户查询：{query}
```

**优点**：语义理解准确率高（>90%），可处理模糊查询
**缺点**：每次查询额外消耗一次 LLM 调用（可用小模型降低成本）

### 方案 C：混合方案（最终推荐）

```
1. 先用方案 A（关键词规则）快速判断
2. 如果置信度高（信号明确）→ 直接路由
3. 如果置信度低（模糊查询）→ 调用方案 B（LLM 分类器）
```

**实测效果**（来自行业数据）：
- 80% 的查询可被规则直接路由（< 5ms 延迟）
- 20% 的模糊查询走 LLM 分类（额外 ~200ms）
- 整体分类准确率 > 92%
- 平均路由延迟 < 50ms

## 七、验证方案

1. **分类准确率测试**：准备 20 条不同复杂度的测试查询，验证路由分类是否正确
2. **性能测试**：对比规则路由 vs LLM 路由 vs 混合路由的延迟和准确率
3. **边界测试**：测试模糊查询（如"帮我看看这个"）的路由行为
4. **回退测试**：验证 LLM 分类器不可用时，规则路由的兜底能力
