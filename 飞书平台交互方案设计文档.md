## 一、方案概述

### 1.1 设计目标

本方案设计飞书平台作为个人知识库系统的**主要交互入口**，实现：

```
输入：用户通过飞书消息（文本/图片/文件/语音/链接）将知识送入 Ingest 管道
查询：用户通过自然语言在飞书中查询知识库内容
展示：查询结果通过飞书卡片结构化呈现（含关系路径可视化）
反馈：输入处理过程的三级渐进式反馈
体验：拟人化交互，简洁直观
```

### 1.2 设计原则

```
原则 1：飞书只做"采集"和"展示"，不做"编译"
  消息接收 → 构造标准请求 → 调用 Knowledge API
  查询结果 ← 卡片渲染 ← Knowledge API 返回
  所有 LLM 调用发生在后端 Ingest/Query 管道中

原则 2：输入 vs 查询通过命令前缀区分
  默认行为 = 输入（保存知识）
  /q 前缀 = 查询（检索知识）
  其他命令 = 系统操作（统计/帮助等）

原则 3：异步反馈，绝不沉默
  收到消息 → 1秒内回复确认
  处理完成 → 主动发送结果通知
  处理失败 → 发送错误说明 + 建议

原则 4：拟人化交互
  不是冷冰冰的命令行工具
  用友好、简洁的语言引导用户
  错误时给建议，而非只报错
```

### 3.2 六种消息类型处理

#### (1) 文本消息

```
接收格式（飞书事件）：
  msg_type: "text"
  content: {"text": "RAG 是 Retrieval Augmented Generation 的缩写，它结合了检索和生成..."}

处理流程：
  1. 提取 text 字段
  2. 检测是否包含 URL（http:// 或 https://）
     ├── 含 URL → format="url", source=URL
     └── 不含 → format="text", source=原文
  3. 构造 Ingest 请求

伪代码：
  async def handle_text(content: dict, message_id: str):
      text = content["text"]
      
      # 检测 URL
      if re.match(r'https?://\S+', text.strip()):
          payload = {
              "source": text.strip(),
              "options": {"format": "url", "channel": "feishu"}
          }
      else:
          payload = {
              "source": text,
              "options": {"format": "text", "channel": "feishu"}
          }
      
      # 调用 Ingest API
      result = await ingest_api(payload)
      
      # 发送即时确认卡片
      await send_card(message_id, build_ack_card(result["task_id"]))
```

#### (2) 富文本消息

```
接收格式：
  msg_type: "post"
  content: {
    "title": "文章标题",
    "content": [
      [{"tag": "text", "text": "正文内容"}],
      [{"tag": "a", "href": "https://...", "text": "链接文字"}],
      [{"tag": "img", "image_key": "img_xxx"}],
      [{"tag": "code_block", "language": "PYTHON", "text": "code..."}]
    ]
  }

处理流程：
  1. 遍历 content 数组，按 tag 类型拼接
     ├── text → 直接拼接文本
     ├── a → 拼接为 Markdown 链接 [text](href)
     ├── img → 标记图片位置 [图片]
     ├── code_block → 拼接为 Markdown 代码块
     ├── at → 忽略
     └── hr → 插入分隔线 ---
  2. 如果提取到链接，额外标注 options.url
  3. format="markdown", source=拼接后的内容

伪代码：
  async def handle_rich_text(content: dict, message_id: str):
      parts = []
      urls = []
      for line in content.get("content", []):
          for element in line:
              tag = element.get("tag")
              if tag == "text":
                  parts.append(element["text"])
              elif tag == "a":
                  parts.append(f"[{element['text']}]({element['href']})")
                  urls.append(element["href"])
              elif tag == "code_block":
                  lang = element.get("language", "").lower()
                  parts.append(f"```{lang}\n{element['text']}\n```")
              elif tag == "img":
                  parts.append("[图片]")
              elif tag == "hr":
                  parts.append("---")
      
      markdown_text = "\n".join(parts)
      payload = {
          "source": markdown_text,
          "options": {
              "format": "markdown",
              "channel": "feishu",
              "title": content.get("title", ""),
              "urls": urls
          }
      }
      result = await ingest_api(payload)
      await send_card(message_id, build_ack_card(result["task_id"]))
```

#### (3) 图片消息

```
接收格式：
  msg_type: "image"
  content: {"image_key": "img_4adb3cc3-902b-4187-b0f1-842f67fd017g"}

处理流程：
  1. 通过飞书 API 下载图片
     GET /open-apis/im/v1/messages/{message_id}/resources/{image_key}
     Headers: Authorization: Bearer <tenant_access_token>
  2. 将图片字节 Base64 编码
  3. format="image", source=Base64字符串
  4. 后端 Ingest 管道 Step 1 使用多模态 LLM 或 OCR 提取文字

伪代码：
  async def handle_image(content: dict, message_id: str):
      image_key = content["image_key"]
      
      # 下载图片
      image_bytes = await feishu_client.download_resource(message_id, image_key)
      
      payload = {
          "source": base64.b64encode(image_bytes).decode(),
          "options": {
              "format": "image",
              "channel": "feishu",
              "file_name": f"{image_key}.jpg",
              "file_mime": "image/jpeg"
          }
      }
      result = await ingest_api(payload)
      await send_card(message_id, build_ack_card(result["task_id"]))
```

#### (4) 文件消息

```
接收格式：
  msg_type: "file"
  content: {"file_key": "75235e0c-xxx", "file_name": "论文.pdf"}

处理流程：
  1. 通过飞书 API 下载文件
     GET /open-apis/im/v1/messages/{message_id}/resources/{file_key}
  2. Base64 编码
  3. format="file", source=Base64字符串
  4. 后端 Ingest 管道 Step 1 根据 MIME 类型选择处理方式：
     - PDF → pymupdf 提取
     - Word → python-docx 提取
     - 其他文本 → 直接读取
  5. 文件大小限制：50MB

伪代码：
  async def handle_file(content: dict, message_id: str):
      file_key = content["file_key"]
      file_name = content.get("file_name", "unknown")
      
      # 文件大小检查（飞书文件限制 100MB，我们限制 50MB）
      file_bytes = await feishu_client.download_resource(message_id, file_key)
      if len(file_bytes) > 50 * 1024 * 1024:
          await reply_text(message_id, 
              "📎 文件太大了（超过 50MB），处理起来比较吃力。\n"
              "试试拆分成小一点的文件？或者直接把关键内容复制粘贴给我。")
          return
      
      # 推断 MIME 类型
      mime_type = guess_mime_type(file_name)
      
      payload = {
          "source": base64.b64encode(file_bytes).decode(),
          "options": {
              "format": "file",
              "channel": "feishu",
              "file_name": file_name,
              "file_mime": mime_type
          }
      }
      result = await ingest_api(payload)
      await send_card(message_id, build_ack_card(result["task_id"]))
```

#### (5) 音频消息

```
接收格式：
  msg_type: "audio"
  content: {"file_key": "75235e0c-xxx", "duration": 5000}

处理流程：
  1. 通过飞书 API 下载音频文件
  2. 后端 Ingest 管道 Step 1 使用 Whisper 转文字
  3. 转文字后走标准文本处理流程
  4. 附加音频时长信息

伪代码：
  async def handle_audio(content: dict, message_id: str):
      file_key = content["file_key"]
      duration = content.get("duration", 0)
      duration_sec = duration / 1000
      
      # 下载音频
      audio_bytes = await feishu_client.download_resource(message_id, file_key)
      
      payload = {
          "source": base64.b64encode(audio_bytes).decode(),
          "options": {
              "format": "audio",
              "channel": "feishu",
              "file_name": f"voice_{message_id}.ogg",
              "file_mime": "audio/ogg",
              "duration_seconds": duration_sec
          }
      }
      result = await ingest_api(payload)
      
      # 确认卡片中标注音频时长
      card = build_ack_card(result["task_id"], 
                           extra=f"🎵 语音消息 ({duration_sec:.0f}秒)")
      await send_card(message_id, card)
```

#### (6) URL 链接（文本中检测）

```
处理流程：
  1. 在文本消息处理中检测 URL 模式
  2. 如果用户只发了一个 URL（无其他文字）→ format="url"
  3. 如果 URL 嵌在文字中 → format="text"，URL 作为附加信息
  4. 后端 Ingest 管道 Step 1 使用 readability 提取网页正文

判断逻辑：
  text = "https://example.com/article"
  → 仅 URL → format="url"
  
  text = "这篇文章讲的 RAG 技术很好 https://example.com/article"
  → 文本 + URL → format="text", options.urls=["https://..."]
```

### 3.3 与后端 Ingest 管道对接

```
飞书消息 → 适配层构造标准请求 → Knowledge API

统一请求格式：
  POST /api/ingest
  Headers: Authorization: Bearer <API_TOKEN>
  Body: {
    "source": "内容（文本/Base64/URL）",
    "options": {
      "format": "text|markdown|url|file|image|audio",
      "channel": "feishu",
      "title": "可选标题",
      "tags": ["可选标签"],
      "file_name": "可选文件名",
      "file_mime": "可选MIME类型",
      "urls": ["可选URL列表"],
      "duration_seconds": "可选音频时长"
    }
  }

统一响应格式（异步）：
  202 Accepted
  {
    "task_id": "uuid",
    "status": "queued"
  }

飞书不阻塞等待——收到 202 后立即发送确认卡片，
后续通过回调或轮询获取处理结果。
```

## 四、知识查询功能

### 4.1 查询触发机制

```
命令前缀区分输入 vs 查询：

  /q RAG 和知识图谱有什么关系？
  /q 为什么个人场景下推荐 Graph-First 架构？
  /q 总结一下所有知识库方案的演进趋势

  ↑ /q 前缀 → 触发查询流程

  RAG 是 Retrieval Augmented Generation 的缩写...
  
  ↑ 无前缀 → 触发输入流程

  /stats → 知识库统计
  /search 知识图谱 → 关键词搜索
  /recent → 最近输入记录
  /help → 使用帮助
```

### 4.2 查询处理流程

```
用户发送 /q <问题>
        │
        ▼
  飞书消息适配层
  ├── 提取问题文本（去掉 /q 前缀）
  ├── 发送"正在思考..."即时反馈
  │
  ▼
  POST /api/query
  Body: { "question": "RAG 和知识图谱有什么关系？" }
        │
        ▼
  后端 Query 管道（3步处理）
  │
  ├── Step 1: 查询理解
  │     识别实体：[RAG, 知识图谱]
  │     查询类型：关联查询 (relational)
  │     推理深度：depth = 1
  │
  ├── Step 2: Graph 检索
  │     Cypher: MATCH (n)-[r]->(m) WHERE n.id IN ['RAG', '知识图谱']
  │     返回：节点内容 + 显式关系 + 隐式关系
  │
  └── Step 3: LLM 生成回答
        基于检索结果生成结构化回答
        │
        ▼
  构造查询结果卡片 → 发送到飞书
```

### 4.3 四种查询深度的展示策略

| 查询类型 | 触发信号 | 检索策略 | 飞书展示 |
|---------|---------|---------|---------|
| 事实查询 | "是什么/定义/全称" | 单节点 content | 摘要卡片 |
| 关联查询 | "关系/区别/对比" | 节点 + 一跳关系 | 双节点 + 关系路径 |
| 推理查询 | "为什么/导致/因果" | 多跳路径 + 隐式关系 | 推理链 + 置信度 |
| 全局查询 | "总结/趋势/整体" | 群落扫描 + 桥接 | 群落概览 + 桥接节点 |

### 5.2 Level 1：即时确认卡片

### 8.2 认证与安全

```
双重认证：

  1. 飞书 → 服务器（入站安全）
     ──────────────────────────
     飞书事件推送携带签名，服务端验证：
     - Verification Token 校验
     - Encrypt Key 解密（可选但推荐）
     - 确保请求来自飞书，防止伪造
     
     实现：
       @router.post("/webhook/feishu")
       async def feishu_webhook(request: Request):
           # 验证签名
           timestamp = request.headers.get("X-Lark-Signature-Timestamp")
           nonce = request.headers.get("X-Lark-Signature-Nonce")
           signature = request.headers.get("X-Lark-Signature")
           
           if not verify_feishu_signature(timestamp, nonce, signature):
               raise HTTPException(403, "签名验证失败")
           
           body = await request.json()
           # ... 处理消息

  2. 服务器 → Knowledge API（内部安全）
     ──────────────────────────────────
     内部调用使用静态 API Token
     Authorization: Bearer <KNOWLEDGE_API_TOKEN>
     
     单用户模式，Token 通过环境变量管理
```

    def __init__(self):

        self.app_secret = os.environ["FEISHU_APP_SECRET"]
        self._token = None
        self._token_expires = 0
    
    async def get_tenant_token(self) -> str:
        """获取飞书 tenant_access_token（自动缓存和刷新）"""
        if self._token and time.time() < self._token_expires:
            return self._token
        resp = await httpx_client.post(
            "https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal",
            json={"app_id": self.app_id, "app_secret": self.app_secret}
        )
        data = resp.json()
        self._token = data["tenant_access_token"]
        self._token_expires = time.time() + data.get("expire", 7200) - 300
        return self._token

        """下载消息中的资源文件（图片/文件/音频）"""
        token = await self.get_tenant_token()
        resp = await httpx_client.get(
            f"https://open.feishu.cn/open-apis/im/v1/messages/{message_id}"

            params={"type": "file"}

### 8.4 异步任务管理

```
流程：
  1. 收到飞书事件 → 立即返回 {"code": 0}
  2. 将消息处理放入 BackgroundTasks（FastAPI 后台任务）
  3. 后台任务：
     a. 发送即时确认卡片
     b. 调用 Ingest/Query API
     c. 轮询任务状态（GET /api/tasks/{task_id}）
     d. 任务完成后发送结果卡片

轮询策略：
  间隔 1 秒，最多等待 60 秒
  超时后发送"处理超时"通知

备选方案（更高效）：
  Ingest 管道完成后主动回调飞书适配层
  通过 Redis Pub/Sub 或 asyncio.Event 通知
  避免轮询开销
```
