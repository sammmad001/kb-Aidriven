# 输入链路完整设计文档：知识摄入编译管道

> **⚠️ 废弃声明**：本文档为 V1.0 之前的设计草案，部分设计已被实际实现取代。请以代码为准。

## 文档定位

本文档是个人知识库系统**输入链路（链路 1）**的完整设计方案，整合以下已有方案的核心决策：

- **`个人知识库方案计划.md`**：整体架构和三阶段渐进策略
- **`输入链路设计方案.md`**：输入链路的基础四层架构和流程设计
- **`存储架构设计方案.md`**（v2）：Wiki-Graph 双写入协作架构，Graph 作为主动推理层介入编译过程
- **`查询路由层优化方案.md`**：查询链路的四级路由设计（与本链路对称）

系统的两条核心链路：
- **链路 1（输入）**：输入 → 分析推理 → 存储（**本文档**）
- **链路 2（查询）**：查询 → 分析推理 → 输出（见查询路由层优化方案）

## 二、第 1 层：输入层详细设计

### 2.1 支持的输入格式

| 输入类型 | 预处理方式 | 产出格式 | 说明 |
|---------|----------|---------|------|
| 文档 / PDF | PDF 解析工具（如 pymupdf）提取文本 | Markdown | 保留标题层级结构 |
| 网页 / 文章 | HTML→Markdown 清洗（去广告/导航/侧栏） | Markdown | 保留正文和超链接 |
| 笔记 / 想法 | 无需预处理 | Markdown | 直接存入 |
| 代码片段 | 包裹在代码块中 | Markdown | 标注语言类型 |
| 对话记录 | 结构化提取（区分角色） | Markdown | 保留对话上下文 |

### 2.2 Raw 素材存储规范

```
raw/sources/
├── 2024-01-15-rag-retrieval-augmented-generation.md
├── 2024-01-20-llm-wiki-karpathy-method.md
└── 2024-02-03-personal-knowledge-graph-notes.md
```

**命名规范**：`<YYYY-MM-DD>-<简短标题>.md`

**不可变原则**：
- Raw 文件一旦存入，永远不被修改或删除
- 所有后续处理只读 raw 文件
- 如果素材需要修正，重新存入一个新版本，旧版本保留

### 2.3 元数据提取

每个 raw 文件在存入时自动提取以下元数据：

```yaml
---
title: "RAG: Retrieval Augmented Generation"
source: "arxiv paper + blog post"
date: 2024-01-15
tags: [RAG, LLM, 检索, 知识管理]
format: article
word_count: 3200
---
```

## 三、第 2 层：分类分析层详细设计

### 3.1 素材五类分型

素材经过 LLM 分析后，被归入以下五种类型之一。每种类型对应不同的编译策略和页面模板：

| 类型 | 识别信号 | 典型内容 | 编译策略 | 页面模板 |
|------|---------|---------|---------|---------|
| **事实型** | 数据、定义、规格、日期、数字、技术参数 | "RAG 全称是 Retrieval Augmented Generation" | 创建/更新实体页属性 | entity-template |
| **概念型** | 理论、方法论、框架、范式、设计模式 | "知识编译范式是一种将原始信息转化为结构化知识的方法" | 创建概念页 + 关联实体 | concept-template |
| **经验型** | "我发现"、"实践证明"、"个人感悟"、"踩坑" | "我发现用 Ollama 本地跑 LLM Wiki，编译质量比 GPT-3.5 差但成本为零" | 创建经验页 + 关联概念 | experience-template |
| **对比型** | "vs"、"比较"、"优缺点"、"区别"、"对比" | "LLM Wiki vs GraphRAG：前者成本低但多跳弱，后者多跳强但成本高" | 创建对比页 + 关联双方 | comparison-template |
| **关系型** | "导致"、"因为"、"推导"、"因果"、"依赖" | "RAG 的多跳推理瓶颈导致了 GraphRAG 的出现" | 更新双方页面 + 添加语义标签 | 内嵌更新 |

```

{raw_content}

{index_summary}

请输出：
{
  "type": "factual|conceptual|experiential|comparative|relational",
  "entities": [
    {"name": "RAG", "exists": true, "wiki_path": "wiki/entities/RAG.md"},

  ],
  "relations": [
    {"from": "RAG", "to": "GraphRAG", "type": "evolves_to", "evidence": "GraphRAG 是 RAG 的图谱增强版本"}
  ],
  "conflicts": [
    {"page": "wiki/entities/RAG.md", "field": "适用场景", "existing": "大规模", "new": "小规模也可"}
  ],
  "gaps": ["缺少 GraphRAG 的成本数据"],
  "compile_suggestion": "创建对比页面，更新 RAG 和 GraphRAG 的实体页面"
}
```

### 3.3 实体存在性判断流程

```
新素材进入
    │
    ▼
提取实体列表（LLM 识别关键实体：人/组织/技术/概念）
    │
    ├─── 实体 A ───┐
    ├─── 实体 B ───┤
    └─── 实体 C ───┴──→ 逐个判断存在性
                              │
                    ┌─────────┴──────────┐
                    ▼                    ▼
              精确匹配              模糊匹配
              查 index.md          别名/缩写/中英文
              页面名完全一致       (如 "RAG" ↔ "检索增强生成")
                    │                    │
              ┌─────┴─────┐        ┌─────┴─────┐
              │ 已存在     │        │ 不存在     │
              │           │        │           │
              │ 对比新内容 │        │ 标记为     │
              │ 与已有内容 │        │ "新实体"   │
              │           │        │           │
              │ ┌───────┐ │        │ 编译时创建 │
              │ │ 无冲突 │ │        │ 新页面    │
              │ │ → 补充 │ │        │ + 加入    │
              │ └───────┘ │        │  index.md  │
              │ ┌───────┐ │        └───────────┘
              │ │ 有冲突 │ │
              │ │ → 标注 │ │
              │ │   矛盾 │ │
              │ └───────┘ │
              └───────────┘
```

### 3.4 矛盾检测逻辑

```
检测到实体已存在时，对比以下维度：

1. 事实冲突
   ────────────
   已有："RAG 由 Meta 在 2020 年提出"
   新增："RAG 由 Facebook AI Research 在 2020 年提出"
   → 标注为 "factual_conflict"（两者可能等价，需人工确认）

2. 观点冲突
   ────────────
   已有："个人场景下 GraphRAG 是最佳选择"
   新增："个人场景下 LLM Wiki 更经济实用"
   → 标注为 "opinion_conflict"（两种观点共存）

3. 时序冲突
   ────────────
   已有："系统仅支持 Markdown 输入"（记录于 2024-01）
   新增："系统支持 PDF/Word/HTML 输入"（记录于 2024-03）
   → 标注为 "temporal_update"（以新信息替换旧信息）
```

## 四、第 3 层：编译决策层详细设计

### 4.1 编译动作决策表

根据分类分析的结果（素材类型 + 实体状态 + 关系类型），确定具体的编译操作：

| 场景 | 素材类型 | 实体状态 | 编译动作 | 影响范围 |
|------|---------|---------|---------|---------|
| 场景 1 | 事实型 | 新实体 | 创建实体页 + 更新 index.md | 1 个新页面 + 1 个 index 更新 |
| 场景 2 | 事实型 | 已有实体（无冲突） | 更新实体页属性 | 1 个页面更新 |
| 场景 3 | 事实型 | 已有实体（有冲突） | 标注矛盾 + 创建冲突记录 | 2-3 个页面 |
| 场景 4 | 概念型 | 新概念 + 已有实体 | 创建概念页 + 关联 3-5 个实体页 | 1 个新页面 + 3-5 个更新 |
| 场景 5 | 经验型 | 关联已有概念 | 创建经验页 + 关联概念页 | 1 个新页面 + 1-3 个更新 |
| 场景 6 | 对比型 | 双方已存在 | 创建对比页 + 更新双方页面 | 1 个新页面 + 2 个更新 |
| 场景 7 | 关系型 | 双方已存在 | 更新双方页面 + 添加语义标签 | 2-5 个页面 + wikilinks |
| 场景 8 | 综合型 | 多个实体 | 创建综合分析页 + 关联 5-10 个实体 | 1 个新页面 + 5-10 个关联 |
| 场景 9 | 来源摘要 | — | 创建来源摘要页 + 更新 log.md | 1 个新页面 |

### 4.2 级联更新范围判断

```
新素材编译完成后，需要扫描以下范围确定级联影响：

扫描范围 1：同主题域（同目录下的其他页面）
──────────────────────────────────────
  新素材创建了 wiki/entities/RAG.md
  → 扫描 wiki/entities/ 目录下所有页面
  → 检查是否需要添加对 RAG 的 wikilink
  → 预期影响：0-3 个页面

扫描范围 2：跨主题域（index.md 中相关概念）
──────────────────────────────────────
  新素材涉及 "RAG" 和 "知识图谱"
  → 查找 index.md 中同时提及这两个概念的其他页面
  → 检查是否需要添加新的交叉引用
  → 预期影响：0-5 个页面

扫描范围 3：综合分析页面
──────────────────────────────────────
  新素材涉及的内容可能影响已有的综合分析结论
  → 扫描 wiki/synthesis/ 目录
  → 检查是否需要纳入新信息
  → 预期影响：0-2 个页面

级联更新规则：
  ✅ 每个受影响的页面更新 "Updated" 日期
  ✅ 添加新的 [[wikilinks]]（如果缺失）
  ❌ 不修改 archive 页面（归档页面是时间快照，不级联更新）
  ❌ 不修改页面的主体结论（除非是事实性修正）
```

## 五、第 4 层：编译执行层详细设计

### 5.1 执行顺序

编译执行必须严格按照以下优先级顺序进行：

```
优先级 1（最高）：创建/更新主要页面
  → 素材直接对应的内容页面
  → 使用对应的页面模板
  → 这是编译的核心产出

优先级 2：级联更新关联页面
  → 受影响的实体页、概念页
  → 主要是添加新的 [[wikilinks]] 或更新 "Related" 区域

优先级 3：更新 [[wikilinks]] 双向链接
  → 确保新创建的页面被正确链接
  → 确保已有页面指向新页面的链接存在

优先级 4：更新 index.md
  → 添加新页面条目
  → 更新已有页面的摘要（如内容有变化）

优先级 5：追加 log.md
  → 记录本次 Ingest 的操作日志
  → 格式：日期 | 操作类型 | 影响页面 | 原始素材

优先级 6：标注矛盾（如有）
  → 在冲突页面添加 "⚠️ Contradiction" 标记
  → 链接到冲突记录页面

优先级 7（最低）：更新 overview.md
  → 只有当新素材影响全局认知时才更新
  → 例如引入了全新的知识领域
```

### 5.2 页面模板

每个页面类型有对应的模板，保证结构一致性：

**实体页模板**（entity-template）：
```markdown
# {实体名}

> {一句话定义}

## Overview
{概述，2-5 段}

## Key Facts
- {事实 1}
- {事实 2}

## See Also
- [[关联实体 A]]
- [[关联概念 B]]

## Sources
- [{来源标题}](../raw/{来源文件名})

---
*Created: {日期} | Updated: {日期}*
```

**概念页模板**（concept-template）：
```markdown
# {概念名}

> {一句话定义}

## Core Idea
{核心思想，2-5 段}

## Related Entities
- [[实体 A]] — {关联说明}
- [[实体 B]] — {关联说明}

## Applications
- {应用场景 1}
- {应用场景 2}

## See Also
- [[关联概念 C]]
- [[对比分析 D]]

## Sources
- [{来源标题}](../raw/{来源文件名})

---
*Created: {日期} | Updated: {日期}*

## 六、第 5 层：Graph 增强层详细设计

### 6.1 设计背景

Graph 增强层的核心价值：**发现 Wiki 页面中没有被显式记录的隐式关系**。

Wiki 页面通过 `[[wikilinks]]` 记录的是**显式关联**——即"编写者认为 A 和 B 有关联"。但知识的语义网络中存在大量**隐式关联**——即"A 和 B 确实存在逻辑关系，但编写时没有创建链接"。这些隐式关联是推理深度的关键保障。

```
显式关系 vs 隐式关系示例：

显式（Wiki 中有 wikilinks）：
  [[RAG]] ──uses──→ [[向量数据库]]         ← 页面明确写了这个链接

隐式（需要推理才能发现）：
  [[RAG]] ──depends_on──→ [[语义理解]]      ← RAG 核心依赖语义理解，但页面没直接链接
  [[LLM Wiki]] ──superior_in──→ [[个人知识管理]]  ← 个人场景更优，但需推理得出
  [[知识图谱]] ──solves──→ [[RAG 多跳瓶颈]]  ← 知识图谱解决了 RAG 的痛点
```

### 6.2 Graph 增强的四步处理流程

#### 步骤 1：显式关系提取（确定性，无 LLM，< 100ms）

```
输入：新编译的 Wiki 页面
处理：
  ├── 解析所有 [[wikilinks]] → 创建显式边（edge.type = "wikilink"）
  ├── 解析页面所在目录 → 设置节点类型（entity / concept / source / ...）
  ├── 解析 YAML frontmatter → 设置节点标签（tags）
  └── 解析页面一句话摘要 → 设置 node.summary

输出：更新后的 nodes.json + edges-explicit.json
```

#### 步骤 2：隐式关系推理（LLM 介入，2-5s）

```
输入：
  ├── 新编译的 Wiki 页面（完整内容）
  ├── 邻居页面摘要（1-2 跳范围内的 Wiki 页面，每页一行摘要）
  └── 现有 Graph 子图（与新增节点相关的部分）

LLM Prompt：

  你正在分析一个个人知识库系统。以下是新编译的知识页面及其上下文：

  【新页面】{页面完整内容}

  【关联页面摘要】
  - {页面A}: {一行摘要}
  - {页面B}: {一行摘要}

  【现有图谱关系】
  ({实体A}) --[{关系类型}]--> ({实体B})

  请发现以下类型的隐式关系：
  1. 依赖关系（depends_on）：A 的核心能力依赖 B
  2. 对立关系（trade_off）：A 和 B 存在取舍或矛盾
  3. 桥接关系（bridges）：A 连接不同知识领域
  4. 演化关系（evolves_to）：A 是 B 的前身/替代/演进
  5. 解决关系（solves）：A 解决了 B 的某个问题

  对每个关系，输出 JSON：
  {
    "source": "实体名",
    "target": "实体名",
    "type": "关系类型",
    "confidence": 0.0-1.0,
    "evidence": "推理依据（一句话）"
  }

输出：隐式关系列表（含置信度）
存储：edges-implicit.json
```

#### 步骤 3：隐式实体识别（< 200ms）

```
处理：
  ├── 扫描新页面 + 邻居页面内容
  ├── 提取高频概念（出现 3+ 次）
  ├── 对比现有 Graph 节点和 Wiki 页面
  └── 标记无对应页面的高频概念为"隐式实体"

输出：entities-implicit.json
  {
    "entities": [
      {
        "name": "语义理解",
        "occurrence_count": 15,
        "occurrence_pages": ["RAG.md", "知识图谱.md", "LLM_Wiki.md", ...],
        "has_wiki_page": false,
        "suggested_type": "concept",
        "auto_create_threshold": 10  // 出现超过 10 次自动创建页面
      }
    ]
  }
```

#### 步骤 4：图结构计算（500ms - 2s）

```
将新节点和边加入 Graph 后，增量重新计算：

  ├── Leiden 聚类
  │     → 发现知识群落（如"检索技术群落"、"图谱技术群落"、"知识编译群落"）
  │     → 更新 clusters.json
  │
  ├── PageRank
  │     → 识别核心实体（被最多页面引用的概念）
  │     → 更新 nodes.json 中的 page_rank 字段
  │
  ├── 桥接系数
  │     → 识别连接不同知识群落的关键节点
  │     → 如 "成本控制" 连接了 "技术选型" 和 "项目管理" 两个群落
  │     → 更新 metrics.json
  │
  └── 结构洞
        → 识别信息流通的关键瓶颈节点
        → 如果某个节点被移除会导致图谱断裂，标记为结构洞
        → 更新 metrics.json
```

### 6.3 Graph 增强层产出总结

| 产出物 | 存储位置 | 内容 | 来源 |
|-------|---------|------|------|
| 显式边 | edges-explicit.json | wikilinks 对应的关系 | 确定性解析 |
| 隐式关系 | edges-implicit.json | LLM 推理发现的关系（含置信度） | LLM 推理 |
| 隐式实体 | entities-implicit.json | 高频但无页面的概念 | 统计分析 |
| 聚类结果 | clusters.json | 知识群落划分 | Leiden 算法 |
| 结构指标 | metrics.json | PageRank / 桥接系数 / 结构洞 | 图算法 |

## 七、第 6 层：回写层详细设计

### 7.1 回写触发条件

回写不是每次 Ingest 都触发的，只有在 Graph 增强层发现**高质量**信息时才执行：

```
触发条件 A：高置信度隐式关系
  ──────────────────────────────
  置信度 > 0.8 的隐式关系
  且 该关系在 Wiki 页面中未被记录（无 wikilink 或文字描述）

触发条件 B：高频隐式实体
  ──────────────────────────────
  隐式实体出现次数 > 阈值（默认 10 次）
  且 仍无独立 Wiki 页面

触发条件 C：重要桥接发现
  ──────────────────────────────
  发现新的跨群落桥接实体
  该实体连接了两个之前无关联的知识群落
```

### 7.2 回写动作

```
触发回写后，执行以下动作：

动作 1：更新 Wiki 页面的 "Related Concepts" 区域
──────────────────────────────────────────────────
  在相关 Wiki 页面的 "## Related Concepts" 区域添加：

  - **{关系类型}** → [[{目标实体}]]（置信度: {confidence}）
    推理依据：{evidence}

动作 2：创建隐式实体的新 Wiki 页面
──────────────────────────────────────
  如果隐式实体出现频率 > 阈值：
  → 使用 concept-template 创建新页面
  → 页面内容自动生成（基于出现该概念的页面摘要）
  → 页面标记为 "Auto-generated, pending review"
  → 添加到 index.md

动作 3：更新 index.md
─────────────────────────
  添加回写发现的关联
  标注来源为 "[Graph Enhanced]"

动作 4：记录到 log.md
─────────────────────────
  格式：
  {时间} | GRAPH_WRITEBACK | {关系/实体} | 置信度 {value} | 触发条件 {A/B/C}
```

### 7.3 回写的安全保障（防止无限循环）

```
安全机制 1：单次循环限制
──────────────────────────
  每个 Raw 素材最多触发一轮 "编译→增强→回写" 循环
  回写后的 Wiki 更新不再触发新一轮 Graph 增强

安全机制 2：置信度门控
──────────────────────────
  只有置信度 > 0.8 的关系才回写
  低置信度关系只存在于 Graph 中

安全机制 3：去重检测
──────────────────────────
  回写前检查 Wiki 页面是否已包含该关联
  如果已有 → 跳过（避免重复）

安全机制 4：内容隔离
──────────────────────────
  回写内容只添加到 "## Related Concepts" 区域
  不修改页面主体内容

安全机制 5：人类审计标记
──────────────────────────
  所有回写内容标记为 "[Auto]" 前缀
  人类可以通过搜索 "[Auto]" 快速审查所有自动写入的内容
```

## 九、与查询链路的对称关系

输入链路的分类决策和查询链路的路由决策是对称设计，保证"怎么存的就能怎么查"：

| 维度 | 输入链路（本方案） | 查询链路 |
|------|------------------|---------|
| **Tier 0** | 事实型素材 → 创建/更新实体页属性 | 简单事实查询 → Wiki 直接读取单页面 |
| **Tier 1** | 关系型素材 → 两个页面间添加 wikilink | 单跳关联查询 → Wiki wikilinks 遍历 |
| **Tier 2** | 概念型/对比型素材 → 跨页面综合分析页 + Graph 隐式关系 | 多跳推理查询 → Wiki + Graph 隐式关系辅助 |
| **Tier 3** | 综合型素材 → overview.md 全局摘要更新 + Graph 社区结构 | 全局综合查询 → Graph 社区摘要 + Wiki 全文 |

**对称性保证了查询时可以利用输入时积累的所有结构化信息。**

## 十、LLM 调用成本分析

每次 Ingest 的 LLM 调用明细：

| 步骤 | LLM 调用次数 | 预估 Token | 可否用小模型 | 说明 |
|------|------------|-----------|------------|------|
| 第 2 层 分类分析 | 1 次 | ~2000 input + ~500 output | ⚠️ 需中等模型 | 分类+提取需要理解力 |
| 第 4 层 编译执行 | 1-2 次 | ~3000 input + ~2000 output | ❌ 需强模型 | 页面编写需要质量 |
| 第 5 层 隐式关系推理 | 1 次 | ~2000 input + ~500 output | ✅ 可用小模型 | 结构化输出，格式简单 |
| **合计** | **3-4 次** | **~7000-10000 token** | — | — |

**成本控制策略**：
- 隐式关系推理（第 5 层）可用 Ollama 本地小模型（如 Qwen2.5-7B），零 API 成本
- 分类分析（第 2 层）可用中等模型（如 Claude Haiku / GPT-4o-mini），极低成本
- 只有页面编写（第 4 层）需要强模型（如 Claude Sonnet / GPT-4o）
- **预估单次 Ingest 成本：~$0.01-0.03**（使用 API）或 **$0**（全部本地模型）

## 十二、质量检查（Lint）机制

Lint 是输入链路的配套维护操作，定期执行，保持 Wiki 和 Graph 的健康。

### 12.1 确定性检查（自动修复）

| 检查项 | 检测方法 | 修复动作 |
|-------|---------|---------|
| index.md 与实际文件一致性 | 对比 index.md 条目与 wiki/ 目录实际文件 | 缺失条目自动添加；指向不存在文件标记 `[MISSING]` |
| 内部 wikilinks 有效性 | 检查所有 `[[link]]` 目标文件是否存在 | 断链尝试模糊匹配修复；无法修复则标记 `[BROKEN]` |
| raw/ 引用有效性 | 检查所有 `../raw/` 路径指向的文件 | 缺失的 raw 文件标记 `[SOURCE_MISSING]` |
| Graph 节点与 Wiki 页面对应 | 检查 nodes.json 中的 wiki_path | 页面不存在则删除节点（孤儿清理） |
| Graph 边与 wikilinks 一致 | 检查 edges-explicit.json 与实际 wikilinks | 补充缺失的显式边，删除多余的显式边 |

### 12.2 启发式检查（报告，不自动修复）

| 检查项 | 说明 | 严重程度 |
|-------|------|---------|
| 跨页面事实矛盾 | 不同页面对同一事实的描述冲突 | 高 |
| 过时内容 | 已被新来源取代但未更新的内容 | 中 |
| 孤儿页面 | 无入站 wikilinks 的页面 | 低 |
| 缺失概念页 | 频繁提及但缺少独立页面的概念 | 中 |
| 缺失交叉引用 | 语义相关但无 wikilinks 的页面对 | 低 |
| 隐式实体积累 | Graph 中标记为隐式但未创建页面的高频实体 | 中 |
| 低置信度关系积累 | 置信度 < 0.5 的隐式关系（可能是噪音） | 低 |

## 十三、端到端示例：一次完整的 Ingest 过程

**输入**：用户保存了一篇关于 "PathRAG" 的笔记到 raw/sources/

```
Step 0: 预处理
────────────────
  输入：一段关于 PathRAG 的笔记

  提取元数据：title="PathRAG笔记", tags=[PathRAG, RAG, 知识图谱, 路径检索]
