# 个人知识库系统：输出流方案文档

> **版本**：v3.1（V1.2 五步查询管道 + DeepSeek 统一架构）
> **依赖**：产品整体方案文档 v3.1、V1.2 查询管道优化方案文档
> **最后更新**：2026-06-15
>
> **V1.2 变更摘要**：
> - 输出流从 3 步架构升级为 5 步管道：查询理解 → 实体解析 → 图谱检索 → 质量门 → 答案生成
> - 新增 EntityResolver 三层匹配（精确 → CONTAINS → difflib 模糊，阈值 0.6）
> - 新增 QualityGate 质量门判定（SUFFICIENT / PARTIAL / INSUFFICIENT）
> - 新增 PipelineTrace 结构化追踪（trace_id + step-level timing）
> - 查询理解新增 jieba POS 标注实体提取
> - 单节点 factual 快速路径（无 LLM 调用）
> - 对话上下文 follow-up 富化（指代词替换 + 上下文实体继承）
> - LLM 从 Ollama/DashScope 迁移到 DeepSeek API

## 一、输出流概述

### 1.1 设计原则

```
原则 1：统一走 Graph，不需要路由层
  所有查询都经过 Neo4j，由 Graph 内部决定检索深度
  不存在"走 Wiki 还是走 Graph"的路由决策

原则 2：隐式关系是推理的关键增量
  Graph 增强层在输入时发现的隐式关系，在输出时提供结构化推理线索
  LLM 基于这些线索生成回答，推理深度远超纯文本检索

原则 3：检索深度自适应
  简单查询只读单节点（< 10ms），复杂查询遍历图结构（< 500ms）
  Graph 内部自然区分查询复杂度，无需外部路由
```

### 1.2 与旧方案的对比

```
旧方案（双存储 + 四级路由）：
  查询 → 复杂度评分 → Tier 0/1/2/3 路由 → Wiki/Graph/混合 → 生成回答
  → 需要五维评分模型
  → 需要路由决策层
  → Tier 0-1 不经过 Graph，无法利用隐式关系
  → 路由判断可能错误，导致推理深度不足

新方案（Graph-First 统一检索）：
  查询 → 查询理解 → Graph 检索（自适应深度）→ LLM 生成回答
  → 不需要路由层
  → 所有查询都能利用隐式关系
  → Graph 内部自然区分检索深度（简单查询更快，复杂查询更深）
  → 不存在路由错误的风险
```

## 二、Step 1：查询理解

### 2.1 查询类型分类

| 类型 | 特征信号 | 示例 | 检索策略 |
|------|---------|------|---------|
| **事实查询** | 1-2 实体，"是什么/定义/全称" | "RAG 的全称是什么？" | 单节点直接返回 |
| **关联查询** | 2-3 实体，"关系/区别/对比" | "RAG 和知识图谱有什么关系？" | 一跳关系遍历 |
| **推理查询** | 3+ 实体，"为什么/导致/因果" | "为什么个人场景下 LLM Wiki 比 GraphRAG 更适合？" | 多跳路径 + 隐式关系 |
| **全局查询** | 无明确实体，"总结/趋势/整体" | "综合分析所有知识库方案的技术演进趋势" | 群落扫描 + 隐式关系 |

### 2.2 实现方式

#### 方式 A：jieba POS + 关键词规则（快速，零成本）

```python
QUERY_SIGNALS = {
    "factual": ["是什么", "定义", "全称", "什么叫", "什么是", "who", "what"],
    "relational": ["关系", "区别", "对比", "比较", "差异", "影响", "联系", "vs"],
    "causal": ["为什么", "原因", "导致", "因果", "因为", "所以", "推导"],
    "global": ["总结", "全局", "趋势", "所有", "整体", "综合", "全景"],
}

def classify_query(query: str) -> dict:
    """基于关键词的快速查询分类"""
    
    # 检测全局信号
    for kw in QUERY_SIGNALS["global"]:
        if kw in query:
            return {"type": "global", "depth": 3}
    
    # 检测因果信号
    for kw in QUERY_SIGNALS["causal"]:
        if kw in query:
            return {"type": "reasoning", "depth": 2}
    
    # 检测关联信号
    for kw in QUERY_SIGNALS["relational"]:
        if kw in query:
            return {"type": "relational", "depth": 1}
    
    # 默认事实查询
    return {"type": "factual", "depth": 0}
```

**V1.2 升级**：
- 新增 jieba 分词 + POS 标注，提取名词、人名、技术术语等实体候选
- ambiguity 评分阈值从 30 降低到 15（更积极地回退到 LLM 分类）
- 零延迟（< 5ms），零成本，覆盖 80% 的查询

#### 方式 B：LLM 分类器（deepseek-v4-flash，精确）

```
System Prompt:
你是一个查询分析器。分析用户查询，提取以下信息：
1. 查询类型：factual / relational / reasoning / global
2. 涉及的实体列表
3. 推理深度估算（0=单节点, 1=关联, 2=多跳推理, 3=全局综合）

输出 JSON：
{"type": "reasoning", "entities": ["LLM_Wiki", "GraphRAG"], "depth": 2}

用户查询：{query}
```

**V1.2 模型**：deepseek-v4-flash，2s 超时
**特点**：~200ms 延迟，准确率 > 95%，适用于模糊查询

#### 方式 C：混合（推荐）

```
1. jieba POS 实体提取 + 规则分类（方式 A）
2. 如果 ambiguity_score < 15 → 直接使用规则结果（80% 的查询）
3. 如果 ambiguity_score >= 15 → 调用 LLM 分类器（方式 B）（20% 的查询）
4. LLM 分类器调用 deepseek-v4-flash，2s 超时

平均延迟：< 50ms
平均准确率：> 92%
```

### 2.3 实体识别与映射

```
查询中提取实体名称
        │
        ▼
  Neo4j 模糊匹配查询：
  MATCH (n) WHERE n.id CONTAINS $name OR n.name CONTAINS $name 
  OR ANY(tag IN n.tags WHERE tag CONTAINS $name) RETURN n

  支持的匹配方式：
  ├── 精确匹配：id = "RAG"
  ├── 名称包含：name CONTAINS "检索增强"
  ├── 标签匹配：tags 包含查询词
  └── 模糊匹配：similarity(name, $name) > 0.8
```

### 2.4 Step 1.5：实体解析（EntityResolver）★ V1.2 新增

V1.2 将实体匹配从单层模糊查询升级为三层渐进式匹配，提升匹中率并降低误匹配：

```
EntityResolver 三层匹配策略

  Layer 1：精确匹配（exact）
    Cypher: MATCH (n) WHERE n.id = $name RETURN n
    场景：用户输入的实体名与节点 ID 完全一致
    示例："RAG" → 精确匹配 node.id="RAG"

  Layer 2：CONTAINS 匹配（contains）
    Cypher: MATCH (n) WHERE n.id CONTAINS $name OR n.name CONTAINS $name
            OR $name CONTAINS n.id OR ANY(a IN n.aliases WHERE a = $name)
    场景：用户输入部分名称或别名
    示例："检索增强" → CONTAINS 匹配 node.id="RAG"
          alias["检索增强生成"] → 匹配成功

  Layer 3：difflib 模糊匹配（fuzzy）
    Python: difflib.SequenceMatcher(None, name, candidate).ratio() >= 0.6
    场景：拼写错误、缩写差异
    示例："GraphRAG" vs "GraphRAG技术" → ratio=0.87 ≥ 0.6 → 匹配

匹配结果处理：
  ├── 精确匹配 → 直接使用
  ├── 多个 CONTAINS 匹配 → 返回候选列表供用户选择
  ├── 模糊匹配 → 标注 match_type=fuzzy + score
  └── 无任何匹配 → 生成搜索建议（相似实体 Top-5）

产出格式：
  resolved_entities: [
    {name: "RAG", node_id: "RAG", match_type: "exact", score: 1.0},
    {name: "检索增强", node_id: "RAG", match_type: "contains", score: 0.8}
  ]

影响文件：query/resolve.py
```

## 三、Step 2：Graph 检索

### 3.1 简单检索（事实查询）

```cypher
-- 查询单个节点的完整内容
MATCH (n) WHERE n.id = $entity_id
RETURN n.id, n.name, n.content, n.summary, n.tags, n.updated_at

延迟：< 10ms（Neo4j 索引查找）
```

**说明**：由于 Neo4j 节点存储了完整 content，简单查询直接返回即可。不需要额外的检索步骤。

### 3.2 关联检索（关联查询）

```cypher
-- 查询实体间的直接关系
MATCH (n)-[r]->(m) WHERE n.id = $entity_id
RETURN n.name, type(r) AS rel_type, 
       CASE WHEN r.type IS NOT NULL THEN r.type ELSE type(r) END AS semantic_type,
       m.name, m.summary, m.id
ORDER BY 
  CASE WHEN r.confidence IS NOT NULL THEN r.confidence ELSE 1.0 END DESC

延迟：< 50ms
```

**说明**：同时返回显式关系（EXPLICIT）和隐式关系（IMPLICIT），隐式关系按置信度排序。

### 3.3 推理检索（多跳推理）★ 核心

```cypher
-- 第一部分：路径查询（找到实体间的多跳连接）
MATCH path = (a)-[r*1..3]->(b) 
WHERE a.id = $entity_from AND b.id = $entity_to
RETURN path
LIMIT 5

-- 第二部分：隐式关系查询（提供推理线索）
MATCH (n)-[r:IMPLICIT]->(m) 
WHERE n.id IN $all_related_entities AND m.id IN $all_related_entities
RETURN n.name, r.type AS implicit_type, r.confidence, r.evidence, m.name
ORDER BY r.confidence DESC
LIMIT 10

-- 第三部分：桥接实体查询（发现连接不同知识群落的枢纽）
MATCH (a)-[:BELONGS_TO]->(c1:Cluster), (b)-[:BELONGS_TO]->(c2:Cluster)
WHERE a.id = $entity_from AND b.id = $entity_to AND c1.id <> c2.id
MATCH (bridge)-[:BELONGS_TO]->(c1), (bridge)-[:BELONGS_TO]->(c2)
WHERE bridge.page_rank > 0.05
RETURN DISTINCT bridge.name, bridge.summary
LIMIT 5

延迟：< 200ms
```

**说明**：推理检索同时提供三方面信息：
- **路径**：实体间的多跳显式连接
- **隐式关系**：需要推理才能发现的关联（核心增量价值）
- **桥接实体**：连接不同知识领域的关键节点

### 3.4 全局检索（综合分析）

```cypher
-- 第一部分：获取相关群落的所有节点
MATCH (n)-[:BELONGS_TO]->(c:Cluster)
WHERE c.label CONTAINS $topic_keyword
RETURN n.name, n.summary, n.cluster_id, c.label

-- 第二部分：获取群落内的隐式关系网络
MATCH (n)-[r:IMPLICIT]->(m)-[:BELONGS_TO]->(c:Cluster)
WHERE c.label CONTAINS $topic_keyword
RETURN n.name, r.type, r.confidence, r.evidence, m.name
ORDER BY r.confidence DESC

-- 第三部分：获取群落间的桥接关系
MATCH (c1:Cluster)<-[:BELONGS_TO]-(n)-[r:IMPLICIT]->(m)-[:BELONGS_TO]->(c2:Cluster)
WHERE c1.id <> c2.id
RETURN c1.label, n.name, r.type, m.name, c2.label, r.confidence
ORDER BY r.confidence DESC
LIMIT 10

延迟：< 500ms
```

**说明**：全局检索从知识群落层面分析，提供：
- 群落内的知识节点全景
- 群落内的隐式关系网络
- 群落间的桥接关系（展示不同知识领域的交叉点）

### 3.5 Step 2.5：质量门判定（QualityGate）★ V1.2 新增

QualityGate 在 Graph 检索后、LLM 生成前评估检索结果的充分性，避免基于不充分上下文生成幻觉回答：

```
QualityGate 判定逻辑

  输入：Step 2 检索结果（节点列表 + 关系列表）

  判定规则：
  ┌── 空上下文（0 节点）
  │     → INSUFFICIENT：拦截，不调用 LLM
  │     → 返回：“未找到相关知识，请尝试更具体的查询”
  │
  ├── 单节点 + factual 查询
  │     → SUFFICIENT：直接进入 Step 3（factual 快速路径）
  │
  ├── 节点数 < 2 且 relational/reasoning 查询
  │     → PARTIAL：降级回答 + 生成搜索建议
  │     → 回答后附“以下内容可能也相关：[建议列表]”
  │
  └── 节点数 >= 2 或 global 查询
        → SUFFICIENT：正常进入 Step 3

  GateDecision 枚举：
    SUFFICIENT   = "sufficient"
    PARTIAL      = "partial"
    INSUFFICIENT = "insufficient"

  搜索建议生成：
    基于 resolved_entities 的 fuzzy 匹配结果
    返回 Top-5 相似实体（score >= 0.5）

影响文件：query/gate.py
```

## 四、Step 3：答案生成（V1.2 升级）

### 4.0 V1.2 快速路径与对话富化

#### Factual 快速路径

```
当 QualityGate 判定为 SUFFICIENT 且 query_type=factual 且仅 1 个节点时：
  → 直接返回节点 content（不调用 LLM）
  → 延迟：< 10ms（Graph 检索时间）
  → 成本：$0（无 LLM 调用）

示例：
  查询：“RAG 的全称是什么？”
  EntityResolver → node.id="RAG"
  QualityGate → SUFFICIENT
  快速路径 → 直接返回 RAG.content
```

#### Follow-up 对话富化

```
ConversationContext（per-user，max_turns=5，TTL=600s）

当查询被识别为 follow-up 时：
  ① 指代词替换：
     “它的优势是什么？” + 上文实体“RAG” → “RAG 的优势是什么？”

  ② 上下文实体继承：
     上文查询的 resolved_entities 传递到当前查询
     “跟它有什么关系？” → 继承上文实体 “RAG”

富化后查询重新进入 Step 1 分析

影响文件：feishu/context.py
```

#### PipelineTrace 结构化追踪

```
每次查询生成唯一 trace_id，记录每步耗时和结果摘要：

PipelineTrace {
  trace_id: "tr_20260615_a1b2c3",
  steps: [
    {name: "understand", duration_ms: 5, entities: ["RAG", "知识图谱"]},
    {name: "resolve",    duration_ms: 12, resolved: [{"RAG", "exact"}]},
    {name: "retrieve",   duration_ms: 45, nodes_found: 3},
    {name: "gate",       duration_ms: 1, decision: "sufficient"},
    {name: "generate",   duration_ms: 2300, answer_length: 450}
  ],
  total_duration_ms: 2363
}

用途：性能监控、查询质量分析、调试
影响文件：query/trace.py
```

### 4.1 回答生成的 Prompt

```
你是一个个人知识库的智能助手。请基于以下从知识图谱中检索到的信息回答用户的问题。

【用户问题】
{question}

【检索到的知识节点】
- {node_name}: {node_summary}
  {node_content}

【显式关系路径】
{entity_A} → [{rel_type}] → {entity_B} → [{rel_type}] → {entity_C}

【隐式关系线索】
- {entity_X} --[{implicit_type}, 置信度 {confidence}]--> {entity_Y}
  推理依据：{evidence}

请生成回答，要求：
1. 基于检索到的信息回答，不编造知识库中没有的内容
2. 引用来源：在关键论断后标注 [来源: {node_name}]
3. 区分确定事实和推理结论
4. 对于隐式关系线索，说明这是系统推理发现的关系，标注置信度
```

高质量回答可以归档到 Neo4j，成为知识库的一部分：

```cypher
CREATE (q:Query {

  question: $question,
```

## 五、查询复杂度自适应

### 5.1 为什么不需要外部路由层

```
旧方案（双存储）需要路由的原因：
  Wiki 和 Graph 是两个独立存储，必须决定查询走哪条路
  → 错误的路由决策会导致推理深度不足（如 Tier 0 不经过 Graph）

新方案（Graph-First）不需要路由的原因：
  Neo4j 是唯一数据库，所有查询天然走 Graph
  → 简单查询：MATCH 单节点，< 10ms（比读 Wiki 文件还快）
  → 复杂查询：MATCH 多跳路径 + 隐式关系，< 500ms
  → Graph 内部自动根据查询条件选择检索深度
  → 不存在"路由错误"的问题
```

### 5.2 检索深度自适应机制

```
Step 1 分析查询 → 输出 entities + query_type + depth
        │
        ├── depth = 0（事实查询）
        │     → 只查询单节点 content
        │     → 延迟 < 10ms
        │     → 不需要 LLM 生成回答（直接返回 content 即可）
        │
        ├── depth = 1（关联查询）
        │     → 查询节点 + 一跳关系
        │     → 延迟 < 50ms
        │     → 需要 LLM 生成回答
        │
        ├── depth = 2（推理查询）★
        │     → 多跳路径 + 隐式关系 + 桥接实体
        │     → 延迟 < 200ms
        │     → 需要 LLM 基于结构化线索生成回答
        │
        └── depth = 3（全局查询）
              → 群落扫描 + 群落间桥接 + 所有隐式关系
              → 延迟 < 500ms
              → 需要 LLM 综合分析
```

### 5.3 与旧路由方案的整合

旧方案的五维评分模型（实体数量 0.20 / 关系类型 0.25 / 推理深度 0.30 / 跨域需求 0.15 / 实时性 0.10）不再需要作为路由决策使用。但其中的分析维度仍然有价值：

| 旧方案维度 | 在新方案中的作用 |
|-----------|----------------|
| 实体数量 | 用于确定 Cypher 查询的实体列表 |
| 关系类型 | 用于确定检索深度（事实/关联/推理） |
| 推理深度 | 直接映射到 Graph 检索的跳数范围 |
| 跨域需求 | 用于确定是否需要群落间桥接查询 |
| 实时性要求 | 用于决定是否跳过 LLM 生成（depth=0 时直接返回） |

**总结**：旧方案的复杂度分析从"路由决策"降级为"检索参数"，不再决定走哪条路，而是决定 Graph 检索的深度和范围。

## 六、特殊查询场景

### 6.1 图谱浏览查询

```
graph() 接口：获取完整图谱数据（vis-network 格式）

Cypher：
  MATCH (n) 
  RETURN n.id, n.name, n.type, n.summary, n.cluster_id, n.page_rank
  
  MATCH (n)-[r]->(m)
  RETURN n.id, type(r) AS edge_type, r.type AS semantic_type, 
         r.confidence, m.id

渲染为 vis-network 格式：
  {
    "nodes": [{"id": "RAG", "label": "RAG", "group": 3, "value": 0.12}],
    "edges": [{"from": "RAG", "to": "向量数据库", "label": "uses", "dashes": false},
              {"from": "RAG", "to": "语义理解", "label": "depends_on (85%)", "dashes": true}]
  }

  显式关系：实线（dashes: false）
  隐式关系：虚线（dashes: true），标注置信度
```

### 6.2 路径查询

```
graph_path(from, to) 接口：两实体间最短路径

Cypher：
  MATCH path = shortestPath((a)-[*..6]-(b))
  WHERE a.id = $from AND b.id = $to
  RETURN path

返回：路径上的所有节点和边，用于可视化推理路径
```

### 6.3 群落查询

```
graph_cluster(cluster_id) 接口：获取知识群落子图

Cypher：
  MATCH (n)-[:BELONGS_TO]->(c:Cluster) WHERE c.id = $cluster_id
  RETURN n
  
  MATCH (n)-[r]->(m)-[:BELONGS_TO]->(c:Cluster) WHERE c.id = $cluster_id
  RETURN n, r, m

返回：群落内所有节点和关系
```

## 七、LLM 调用成本分析（V1.2 DeepSeek 统一架构）

### 7.1 单次查询的 LLM 调用明细

| 查询类型 | 查询理解 | 实体解析 | Graph 检索 | 质量门 | 答案生成 | 总 LLM 调用 |
|---------|---------|---------|-----------|--------|---------|------------|
| factual 快速路径 | jieba 规则（0次） | 精确匹配（0次） | < 10ms | SUFFICIENT | **快速路径（0次）** | **0 次** |
| factual（含模糊） | jieba 规则（0次） | 模糊匹配（0次） | < 10ms | SUFFICIENT | 1 次（v4-pro） | **1 次** |
| relational | 规则（0次） | 精确/CONTAINS（0次） | < 50ms | SUFFICIENT | 1 次（v4-pro） | **1 次** |
| reasoning | 规则或 LLM（0-1次） | 模糊匹配（0次） | < 200ms | SUFFICIENT | 1 次（v4-pro） | **1-2 次** |
| global | LLM（1次，v4-flash） | N/A | < 500ms | SUFFICIENT | 1 次（v4-pro） | **2 次** |

> **V1.2 优化**：factual 快速路径跳过 LLM 生成，大幅降低高频简单查询的成本。
> QualityGate INSUFFICIENT 时也跳过 LLM 生成，节省无效调用。

### 7.2 成本估算（DeepSeek API）

| 查询类型 | 模型 | Token 估算 | 单次成本 |
|---------|------|-----------|---------|
| factual 快速路径 | — | 0 | **$0** |
| factual（含 LLM） | v4-pro | ~2000 in + 500 out | ~$0.003 |
| relational | v4-pro | ~3000 in + 800 out | ~$0.005 |
| reasoning（含 LLM 理解） | v4-flash + v4-pro | ~5000 in + 1300 out | ~$0.008 |
| global | v4-flash + v4-pro | ~8000 in + 2000 out | ~$0.015 |
| QualityGate 拦截 | — | 0 | **$0** |

### 7.3 月度成本估算

```
V1.2 DeepSeek 架构下假设每月查询分布（500 次）：
  factual 快速路径：200 次（40%）→ $0
  factual（含 LLM）：  50 次（10%）→ ~$0.15
  relational：       100 次（20%）→ ~$0.50
  reasoning：        100 次（20%）→ ~$0.80
  global：            50 次（10%）→ ~$0.75

月查询成本：~$2.2/月
（对比 V1.0 Ollama 混合方案 ~$1.3/月，成本略增但质量大幅提升）
```

> **成本优势**：factual 快速路径（40% 查询零成本）+ QualityGate 拦截（~10% 查询节省无效 LLM 调用）
> 综合有效节省约 50% 的 LLM 调用。

## 八、性能基准

### 8.1 延迟预期

| 查询类型 | Graph 检索 | LLM 生成 | 总延迟 |
|---------|-----------|---------|-------|
| 事实查询 | < 10ms | 0ms（直接返回） | **< 10ms** |
| 关联查询 | < 50ms | 1-3s | **1-3s** |
| 推理查询 | < 200ms | 2-5s | **2-5s** |
| 全局查询 | < 500ms | 3-8s | **3-8s** |

### 8.2 推理质量保障

```
推理质量的三个保障层：

保障层 1：完整内容
  → Neo4j 节点存储完整 Markdown 内容
  → LLM 回答时有精确上下文，不存在三元组信息丢失

保障层 2：隐式关系
  → Graph 增强层在输入时发现的隐式关系
  → 推理查询时直接获取结构化推理线索
  → LLM 不需要"凭空推理"，有图谱引导

保障层 3：图结构指标
  → PageRank 帮助识别核心实体
  → 聚类帮助发现知识群落
  → 桥接系数帮助发现跨界关联
```
