# 个人知识库系统完整架构设计：Graph-First 统一存储

> **版本**：v2.0（V1.2 查询管道 + DeepSeek 统一架构 + 多用户隔离）
> **更新日期**：2025-01-20

## 核心架构决策

| 决策点 | 结论 |
|-------|------|
| 数据存储 | **Neo4j 唯一数据库**（Single Source of Truth） |
| Markdown 文件 | Graph 自动渲染的**只读视图**（供 Obsidian 浏览） |
| 人类编辑 | **不存在**。所有知识通过 Ingest 管道输入 |
| 输入流程 | Raw → 分析 → Graph 处理 → 写入 Neo4j → 渲染 Markdown |
| 输出流程 | 查询 → Graph 推理 → 返回结果 |
| 同步方向 | **单向**：Neo4j → Markdown（不存在反向） |
| 推理保障 | Graph 存储完整内容 + 显式关系 + 隐式关系，所有查询都能利用 |

### 2.1.1 多用户数据隔离机制

> **详见独立方案**：`多用户认证与数据隔离方案文档.md`

V2.0 引入多用户数据隔离，确保不同用户的知识图谱完全独立：

```
隔离架构：

  1. ContextVar user_id 注入
     ─────────────────────────
     FastAPI 请求中间件 → 提取 JWT → 设置 ContextVar user_id
     → database.py 的 execute_read_for_user / execute_write_for_user
     → 所有 Cypher 自动追加 WHERE n.user_id = $user_id
     → 对业务代码透明

  2. 复合唯一约束
     ─────────────────────────
     CREATE CONSTRAINT entity_unique
     FOR (n:Entity) REQUIRE (n.id, n.user_id) IS UNIQUE
     → 同名实体在不同用户下可独立存在
     → 避免跨用户数据污染

  3. 文件系统隔离
     ─────────────────────────
     raw/sources/{user_id}/2025-01-20-topic.md
     → 每个用户的原始素材分目录存储
     → Preprocessor._user_raw_dir 自动路由

  4. 飞书多用户绑定
     ─────────────────────────
     UserStore (SQLite) 维护 feishu_open_id → user_id 映射
     → /register 创建账户 + /bind 绑定飞书
     → 未绑定用户拦截 ingest/query 请求
```

### 2.2 Neo4j 存储的完整知识

Graph 不是只存三元组——每个节点都存储了**完整的页面内容**：

```
传统 GraphRAG 的做法：
  (RAG) -[:USES]-> (向量数据库)
  → 只有关系，丢失了"RAG 为什么用向量数据库"的完整解释

本方案的做法：
  (RAG) -[:USES]-> (向量数据库)
  节点 RAG 的 content 字段 = "RAG (Retrieval Augmented Generation) 是一种结合检索和生成的技术。
  它通过向量数据库进行相似性检索，找到与问题相关的文档片段，然后将其作为上下文传递给 LLM..."
  → 完整内容保留，推理时可以读取完整上下文
```

│   └── 2024-01-15-rag-paper.md

  每次 Ingest 完成后自动渲染受影响的 Markdown 文件
  每次 lint 后完整重新渲染
```

Markdown 文件的内容完全由 Neo4j 中的数据决定：

```python
def render_wiki_page(neo4j_node):
    """将 Neo4j 节点渲染为 Markdown 文件"""
    md = f"# {node['name']}\n\n"
    md += f"> {node['summary']}\n\n"
    md += node['content']  # 完整内容直接从 Neo4j 读取
    
    # 从 Neo4j 查询该节点的所有关系
    relationships = neo4j.query(
        "MATCH (n)-[r]->(m) WHERE n.id = $id RETURN type(r), m.name, r.type ORDER BY r.confidence DESC",
        id=node['id']
    )
    
    md += "\n## Relationships\n"
    for rel in relationships:
        md += f"- **{rel['type']}** → [[{rel['m.name']}]]"
        if rel.get('confidence'):
            md += f" (置信度: {rel['confidence']})"
        md += "\n"
    
    # 写入文件
    write_file(f"wiki/{node_type}/{node['name']}.md", md)
```

### 3.1 输入流程

```
用户输入素材
     │
     ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 预处理                                               │
│                                                              │
│  格式统一 → Markdown → 存入 raw/sources/（不可变）           │
│  提取元数据（标题、来源、日期、标签）                         │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: 分析分类（LLM 调用 1）                               │
│                                                              │
│  输入：raw 素材 + 现有 Neo4j 节点摘要                        │
│                                                              │
│  分析内容：                                                   │
│  ├── 素材类型（事实/概念/经验/对比/关系）                    │
│  ├── 实体提取 + 存在性判断（查 Neo4j）                       │
│  ├── 关系推理（与已有知识的关系）                             │
│  ├── 矛盾检测（对比 Neo4j 中已有内容）                       │
│  └── 知识缺口识别                                            │
│                                                              │
│  产出：分析报告                                              │
│  LLM 调用：1 次                                              │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: Graph 处理（写入 Neo4j + LLM 推理）                  │
│                                                              │
│  3a. 创建/更新节点（LLM 调用 2）                              │
│     ──────────────────────────                                │
│     根据分析报告，创建或更新 Neo4j 节点：                    │
│     ├── 新实体 → CREATE (:Entity {..., content: "..."})      │
│     ├── 已有实体 → SET content = "更新后的内容"              │
│     ├── 新概念 → CREATE (:Concept {...})                     │
│     └── 来源 → CREATE (:Source {raw_content: "..."})         │
│                                                              │
│     节点 content 字段 = 完整页面内容（由 LLM 编写）          │
│                                                              │
│  3b. 创建显式关系                                            │
│     ──────────────────                                       │
│     根据分析报告中的关系，创建显式边：                       │
│     MATCH (a), (b) WHERE a.id = $a AND b.id = $b             │
│     CREATE (a)-[:EXPLICIT {type: "uses", ...}]->(b)          │
│                                                              │
│  3c. 隐式关系推理（LLM 调用 3）★                             │
│     ──────────────────────────                                │
│     组装上下文（新节点内容 + 邻居节点 + 现有子图）           │
│     → LLM 推理发现隐式关系                                  │
│     → 为每条关系分配置信度                                   │
│     → CREATE (a)-[:IMPLICIT {type: "depends_on",             │
│                              confidence: 0.85,               │
│                              evidence: "..."}]->(b)          │
│                                                              │
│  3d. 隐式实体识别                                            │
│     ──────────────────                                       │
│     扫描新内容中的高频概念                                   │
│     对比 Neo4j 已有节点                                     │
│     → 识别无对应节点的隐式概念                              │
│                                                              │
│  3e. 图结构更新                                              │
│     ──────────────────                                       │
│     重新计算 Leiden 聚类 / PageRank / 桥接系数               │
│     → 更新节点的 cluster_id / page_rank 字段                 │
│                                                              │
│  LLM 调用：1-2 次（页面编写 1 + 隐式推理 1）                 │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 4: 渲染 Markdown（自动，无 LLM）                       │
│                                                              │
│  从 Neo4j 读取受影响的节点 → 渲染为 Markdown 文件           │
│  更新 index.md / log.md / overview.md                        │
│                                                              │
│  Obsidian 自动检测文件变化 → 用户可以在 Obsidian 中浏览     │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 输入的 LLM 调用汇总

> **V2.0 变更**：LLM 提供商从 Ollama + DashScope 统一为 DeepSeek API。详见 `DeepSeek迁移方案文档.md`。

| 步骤 | 调用 | 模型要求 | V2.0 模型 |
|------|------|---------|----------|
| Step 2 分析 | 1 次 | 中等（分类+提取） | deepseek-v4-flash |
| Step 3a 页面编写 | 1 次 | 强（内容质量） | deepseek-v4-pro |
| Step 3c 隐式推理 | 1 次 | 中等（结构化输出） | deepseek-v4-flash |
| **合计** | **3 次** | — | ResilientLLMClient 封装 fallback |

> **ResilientLLMClient**：封装 deepseek-v4-pro → deepseek-v4-flash fallback 链 + 指数退避重试，确保可用性。

### 3.3 输入成本估算

```
V2.0 DeepSeek 统一架构：
  Step 2 分析 (deepseek-v4-flash)  ~3K token  ≈ $0.001
  Step 3a 页面编写 (deepseek-v4-pro) ~5K token  ≈ $0.014
  Step 3c 隐式推理 (deepseek-v4-flash) ~3K token  ≈ $0.001
  合计：~$0.016/次

假设月均 200 次摄入：
  ~$0.016 × 200 = ~$3.2/月

辅助 API 成本（OCR + ASR）：
  PaddleOCR 免费本地运行（置信度 ≥ 0.6）
  qwen-vl-max OCR fallback ≈ $0.008/张（少量使用）
  DashScope ASR (Paraformer) ≈ $0.004/分钟（少量使用）
```

### 4.1.1 PipelineTrace 结构化追踪 ★ V1.2

每次查询生成唯一 trace_id，记录每个步骤的耗时、输入/输出摘要：

```python
trace = PipelineTrace(
    trace_id="q_20250120_a1b2c3",
    steps=[
        TraceStep(name="understand", duration_ms=45, ...),
        TraceStep(name="resolve", duration_ms=12, ...),
        TraceStep(name="retrieve", duration_ms=150, ...),
        TraceStep(name="gate", duration_ms=3, decision="SUFFICIENT"),
        TraceStep(name="generate", duration_ms=800, ...),
    ],
    total_ms=1010
)
```

### 4.2 为什么不需要 Tier 路由

方案 B 需要 Tier 路由是因为 Wiki 和 Graph 是两个独立存储，需要决定查询走哪条路。

本方案只有 Neo4j 一个数据库，所有查询天然走 Graph：

```
方案 B：
  Tier 0 → Wiki（一个路径）
  Tier 1 → Wiki + links（另一个路径）
  Tier 2 → Wiki + Graph（混合路径）
  Tier 3 → Graph（又一个路径）
  → 需要路由决策

本方案：
  所有 Tier → Neo4j（统一路径）
  → Graph 内部根据查询复杂度自动选择检索深度
  → 简单查询 Neo4j 直接返回（< 10ms，比读 Wiki 文件还快）
  → 复杂查询 Neo4j 路径遍历 + 隐式关系（< 500ms）
  → 不需要外部路由层
```

  → 用于调试和优化图结构

### 5.2 完整知识脉络

```
Neo4j 中的完整知识脉络：

  节点 = 完整知识单元（包含所有内容）
  显式边 = 直接关联（素材中明确提到的关系）

  桥接 = 跨域枢纽（连接不同群落的关键节点）

  → 任何维度的知识脉络都可以通过 Cypher 查询获取
  → 不需要额外的索引或目录（Neo4j 本身就是最完整的索引）
```
