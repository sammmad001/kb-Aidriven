# 个人知识库系统完整架构设计：Graph-First 统一存储

## 核心架构决策

| 决策点 | 结论 |
|-------|------|
| 数据存储 | **Neo4j 唯一数据库**（Single Source of Truth） |
| Markdown 文件 | Graph 自动渲染的**只读视图**（供 Obsidian 浏览） |
| 人类编辑 | **不存在**。所有知识通过 Ingest 管道输入 |
| 输入流程 | Raw → 分析 → Graph 处理 → 写入 Neo4j → 渲染 Markdown |
| 输出流程 | 查询 → Graph 推理 → 返回结果 |
| 同步方向 | **单向**：Neo4j → Markdown（不存在反向） |
| 推理保障 | Graph 存储完整内容 + 显式关系 + 隐式关系，所有查询都能利用 |

## 二、存储层设计

### 2.1 Neo4j 数据模型

```
节点类型：

(:Entity {                    实体节点
  id: "RAG",
  name: "RAG",
  type: "technology",         // technology / person / organization / concept
  content: "完整页面内容...",   // 完整的 Markdown 内容
  summary: "一句话摘要",
  tags: ["检索增强", "LLM"],
  source: "raw/sources/2024-01-15-rag.md",
  page_rank: 0.12,
  cluster_id: 3,
  created_at: "2024-01-15",
  updated_at: "2024-01-20"
})

(:Concept {                   概念节点
  id: "知识编译范式",
  name: "知识编译范式",
  content: "完整页面内容...",
  summary: "一种将原始信息转化为结构化知识的方法论",
  ...
})

(:Source {                    来源节点
  id: "2024-01-15-rag-paper",
  name: "RAG Paper",
  raw_content: "原始素材内容...",
  ...
})


边类型：

(:EXPLICIT {                  显式关系（来自素材分析）
  type: "uses",               // uses / related_to / contains / ...
  context: "RAG 使用向量数据库进行检索",
  created_at: "2024-01-15"
})

(:IMPLICIT {                  隐式关系（Graph 增强 LLM 推理发现）
  type: "depends_on",         // depends_on / trade_off / bridges / evolves_to / solves
  confidence: 0.85,
  evidence: "RAG 的检索质量直接取决于语义理解模型的准确性",
  discovered_at: "2024-01-20T15:30:00"
})
```

### 2.2 Neo4j 存储的完整知识

Graph 不是只存三元组——每个节点都存储了**完整的页面内容**：

```
传统 GraphRAG 的做法：
  (RAG) -[:USES]-> (向量数据库)
  → 只有关系，丢失了"RAG 为什么用向量数据库"的完整解释

本方案的做法：
  (RAG) -[:USES]-> (向量数据库)
  节点 RAG 的 content 字段 = "RAG (Retrieval Augmented Generation) 是一种结合检索和生成的技术。
  它通过向量数据库进行相似性检索，找到与问题相关的文档片段，然后将其作为上下文传递给 LLM..."
  → 完整内容保留，推理时可以读取完整上下文
```

### 2.3 Markdown 文件（只读渲染）

```
wiki/                              ← 自动生成，不接受人类编辑
├── entities/
│   ├── RAG.md                     ← 从 Neo4j 节点 content 字段渲染
│   ├── 知识图谱.md
│   └── Neo4j.md
├── concepts/
│   ├── 知识编译范式.md
│   └── 图谱增强检索.md
├── comparisons/
│   └── LLM_Wiki_vs_GraphRAG.md
├── sources/
│   └── 2024-01-15-rag-paper.md
├── index.md                       ← 从 Neo4j 查询所有节点生成
├── log.md                         ← 从 Neo4j 操作日志生成
└── overview.md                    ← 从 Neo4j 社区结构生成

渲染时机：
  每次 Ingest 完成后自动渲染受影响的 Markdown 文件
  每次 lint 后完整重新渲染
```

Markdown 文件的内容完全由 Neo4j 中的数据决定：

```python
def render_wiki_page(neo4j_node):
    """将 Neo4j 节点渲染为 Markdown 文件"""
    md = f"# {node['name']}\n\n"
    md += f"> {node['summary']}\n\n"
    md += node['content']  # 完整内容直接从 Neo4j 读取
    
    # 从 Neo4j 查询该节点的所有关系
    relationships = neo4j.query(
        "MATCH (n)-[r]->(m) WHERE n.id = $id RETURN type(r), m.name, r.type ORDER BY r.confidence DESC",
        id=node['id']
    )
    
    md += "\n## Relationships\n"
    for rel in relationships:
        md += f"- **{rel['type']}** → [[{rel['m.name']}]]"
        if rel.get('confidence'):
            md += f" (置信度: {rel['confidence']})"
        md += "\n"
    
    # 写入文件
    write_file(f"wiki/{node_type}/{node['name']}.md", md)
```

## 三、输入链路设计

### 3.1 输入流程

```
用户输入素材
     │
     ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 预处理                                               │
│                                                              │
│  格式统一 → Markdown → 存入 raw/sources/（不可变）           │
│  提取元数据（标题、来源、日期、标签）                         │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: 分析分类（LLM 调用 1）                               │
│                                                              │
│  输入：raw 素材 + 现有 Neo4j 节点摘要                        │
│                                                              │
│  分析内容：                                                   │
│  ├── 素材类型（事实/概念/经验/对比/关系）                    │
│  ├── 实体提取 + 存在性判断（查 Neo4j）                       │
│  ├── 关系推理（与已有知识的关系）                             │
│  ├── 矛盾检测（对比 Neo4j 中已有内容）                       │
│  └── 知识缺口识别                                            │
│                                                              │
│  产出：分析报告                                              │
│  LLM 调用：1 次                                              │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: Graph 处理（写入 Neo4j + LLM 推理）                  │
│                                                              │
│  3a. 创建/更新节点（LLM 调用 2）                              │
│     ──────────────────────────                                │
│     根据分析报告，创建或更新 Neo4j 节点：                    │
│     ├── 新实体 → CREATE (:Entity {..., content: "..."})      │
│     ├── 已有实体 → SET content = "更新后的内容"              │
│     ├── 新概念 → CREATE (:Concept {...})                     │
│     └── 来源 → CREATE (:Source {raw_content: "..."})         │
│                                                              │
│     节点 content 字段 = 完整页面内容（由 LLM 编写）          │
│                                                              │
│  3b. 创建显式关系                                            │
│     ──────────────────                                       │
│     根据分析报告中的关系，创建显式边：                       │
│     MATCH (a), (b) WHERE a.id = $a AND b.id = $b             │
│     CREATE (a)-[:EXPLICIT {type: "uses", ...}]->(b)          │
│                                                              │
│  3c. 隐式关系推理（LLM 调用 3）★                             │
│     ──────────────────────────                                │
│     组装上下文（新节点内容 + 邻居节点 + 现有子图）           │
│     → LLM 推理发现隐式关系                                  │
│     → 为每条关系分配置信度                                   │
│     → CREATE (a)-[:IMPLICIT {type: "depends_on",             │
│                              confidence: 0.85,               │
│                              evidence: "..."}]->(b)          │
│                                                              │
│  3d. 隐式实体识别                                            │
│     ──────────────────                                       │
│     扫描新内容中的高频概念                                   │
│     对比 Neo4j 已有节点                                     │
│     → 识别无对应节点的隐式概念                              │
│                                                              │
│  3e. 图结构更新                                              │
│     ──────────────────                                       │
│     重新计算 Leiden 聚类 / PageRank / 桥接系数               │
│     → 更新节点的 cluster_id / page_rank 字段                 │
│                                                              │
│  LLM 调用：1-2 次（页面编写 1 + 隐式推理 1）                 │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 4: 渲染 Markdown（自动，无 LLM）                       │
│                                                              │
│  从 Neo4j 读取受影响的节点 → 渲染为 Markdown 文件           │
│  更新 index.md / log.md / overview.md                        │
│                                                              │
│  Obsidian 自动检测文件变化 → 用户可以在 Obsidian 中浏览     │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 输入的 LLM 调用汇总

| 步骤 | 调用 | 模型要求 | 可用 Ollama |
|------|------|---------|------------|
| Step 2 分析 | 1 次 | 中等（分类+提取） | ✅ 可用 Qwen2.5-7B |
| Step 3a 页面编写 | 1 次 | 强（内容质量） | ⚠️ 建议 API 或 14B+ |
| Step 3c 隐式推理 | 1 次 | 中等（结构化输出） | ✅ 可用 Qwen2.5-7B |
| **合计** | **3 次** | — | — |

### 3.3 输入成本估算

```
全部使用 API：
  ~10000 token/次 Ingest × $0.003/1K token = ~$0.03/次

混合方案（Ollama 本地 + API）：
  Step 2 + 3c 用 Ollama = $0
  Step 3a 用 API = ~$0.01/次
  合计：~$0.01/次

全部使用 Ollama：
  $0（但编译质量可能下降）
```

## 四、输出链路设计

### 4.1 统一查询流程

```
用户查询
     │
     ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 查询理解（LLM 或规则）                              │
│                                                              │
│  分析查询复杂度：                                             │
│  ├── 识别实体                                                │
│  ├── 识别关系类型                                            │
│  └── 估算需要的推理深度                                      │
│                                                              │
│  输出：实体列表 + 关系类型 + 推理深度                        │
│  延迟：< 50ms（规则匹配）或 ~200ms（LLM）                   │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: Graph 检索（统一走 Graph，不需要路由）              │
│                                                              │
│  简单查询：                                                   │
│    MATCH (n:Entity {id: "RAG"}) RETURN n.content             │
│    → 直接返回节点完整内容                                    │
│    → 延迟：< 10ms                                            │
│                                                              │
│  关联查询：                                                   │
│    MATCH (n)-[r]->(m) WHERE n.id = "RAG"                     │
│    RETURN n, r, m                                            │
│    → 返回节点 + 一跳关系                                     │
│    → 延迟：< 50ms                                            │
│                                                              │
│  推理查询：                                                   │
│    MATCH path = (a)-[r*1..3]->(b)                            │
│    WHERE a.id = "LLM_Wiki" AND b.id = "GraphRAG"             │
│    RETURN path                                               │
│    + 查询隐式关系边                                          │
│    → 返回路径 + 隐式关系线索                                 │
│    → 延迟：< 200ms                                           │
│                                                              │
│  全局查询：                                                   │
│    MATCH (n) WHERE n.cluster_id = $cluster                   │
│    RETURN n.content, n.summary                               │
│    + MATCH (n)-[:IMPLICIT]->(m) WHERE n.cluster_id = $cluster│
│    → 返回群落内所有节点 + 隐式关系                           │
│    → 延迟：< 500ms                                           │
└──────────────────────┬───────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: LLM 生成回答                                        │
│                                                              │
│  输入：用户查询 + Graph 检索结果                             │
│  ├── 节点完整内容（精确上下文）                              │
│  ├── 显式关系路径（关联线索）                                │
│  └── 隐式关系（推理线索，含置信度）                          │
│                                                              │
│  LLM 基于这些信息生成回答                                    │
│  → 附带引用来源（Neo4j 节点 ID）                             │
│  → 可选：归档回答到 Neo4j（CREATE (:Query {...})）           │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 为什么不需要 Tier 路由

方案 B 需要 Tier 路由是因为 Wiki 和 Graph 是两个独立存储，需要决定查询走哪条路。

本方案只有 Neo4j 一个数据库，所有查询天然走 Graph：

```
方案 B：
  Tier 0 → Wiki（一个路径）
  Tier 1 → Wiki + links（另一个路径）
  Tier 2 → Wiki + Graph（混合路径）
  Tier 3 → Graph（又一个路径）
  → 需要路由决策

本方案：
  所有 Tier → Neo4j（统一路径）
  → Graph 内部根据查询复杂度自动选择检索深度
  → 简单查询 Neo4j 直接返回（< 10ms，比读 Wiki 文件还快）
  → 复杂查询 Neo4j 路径遍历 + 隐式关系（< 500ms）
  → 不需要外部路由层
```

## 五、知识脉络与可视化

### 5.1 三层可视化

```
层 1：Obsidian 图谱视图（日常浏览）
─────────────────────────────────────
  基于自动渲染的 Markdown 文件中的 [[wikilinks]]
  → Obsidian 内置图谱直接可用
  → 零配置，打开 wiki/ 目录即可

层 2：Web UI 图谱可视化（深度分析）
─────────────────────────────────────
  从 Neo4j 直接读取图结构 → vis-network 渲染
  → 显式关系（实线）+ 隐式关系（虚线，标注置信度）
  → 聚类高亮（不同颜色区分知识群落）
  → 桥接节点标注（跨界枢纽）
  → 路径查询可视化（A 到 B 的推理路径）

层 3：Neo4j Browser（技术调试）
─────────────────────────────────────
  Cypher 查询任意图模式
  → 用于验证 Graph 推理结果
  → 用于调试和优化图结构
```

### 5.2 完整知识脉络

```
Neo4j 中的完整知识脉络：

  节点 = 完整知识单元（包含所有内容）
  显式边 = 直接关联（素材中明确提到的关系）
  隐式边 = 推理关联（Graph 增强 LLM 发现的关系）
  聚类 = 知识群落（相关知识的自然分组）
  PageRank = 核心实体（被最多关系指向的节点）
  桥接 = 跨域枢纽（连接不同群落的关键节点）

  → 任何维度的知识脉络都可以通过 Cypher 查询获取
  → 不需要额外的索引或目录（Neo4j 本身就是最完整的索引）
```
